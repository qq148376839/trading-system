## 核心判定

修订方向正确，实施发现3个必修bug（已修复）。

## 一、修订方向验证（数据是否支撑改造？）

| 假设 | 数据验证 | 判定 |
|------|---------|------|
| 2连亏后应禁入 | 2连亏后9笔avg -9.3%(-$883) | **强支撑** |
| 不同退出原因应区分冷却 | TP后7笔avg -6.5%, TS后9笔avg +3.3% | **支撑** |

## 二、修订是否对症？

诊断报告核心矛盾：胜率56%但盈亏比0.71，EV=-$9.84/笔。

| 诊断核心问题 | 本次修订是否解决 |
|-------------|---------------|
| 低分信号入场(EV=-$28.74) | 否 — 需要另起分析项目 |
| 止损金额过大($225.60) | 否 — 需要调整止损幅度 |
| 流动性差导致滑点 | 否 — 需要流动性过滤器 |
| 2连亏后冲动重入(-$883) | **是** — A1 直接解决 |
| TP后重入表现差(-6.5%) | **是** — A2 延长TP后冷却 |

结论：修订解决"二阶问题"（冷却/重入），"一阶问题"（入场门槛、止损幅度、流动性）需另起项目。

## 三、代码审查发现

### 已修复的 bug

| # | 问题 | 修复 |
|---|------|------|
| BUG1 | `POSITION_EXIT_CLEANUP` 包含 `consecutiveLossPctSum: null`，IRON_DOME/CROSS_STRATEGY 路径使用时清除该字段，导致与 `consecutiveLosses` 不一致，A1 门控失效 | 从 POSITION_EXIT_CLEANUP 移除，知PnL路径已有显式写入 |
| BUG2 | ATR 缓存 `size === 0` 判断导致部分加载后永不重试失败标的 | 改为逐个补充缺失标的 |
| BUG3 | A4 注释"全组禁入"与实际"全策略禁入"行为不符 | 修正注释 |

### 审查确认正确的部分

- 6因子权重 sum=1.0 (0.25+0.20+0.10+0.15+0.05+0.25)
- A1 门控在两个入场点（processSymbol + evaluateIdleSymbol）均有
- 所有3个 day-reset 块都有 `consecutiveLossPctSum: 0`
- 所有10个 `recordSymbolExit` 调用点都传入了 `exitReason` + `exitPrice`
- MIT回调路径 PnL% 用 allocationAmount 做分母 — 正确
- f6 数据缺失时 fail-open (=0.5) — 正确
- TAKE_PROFIT 2.0x 乘数有数据支撑（TP后7笔avg -6.5%，趋势走完后重入表现最差）

### 审查标注（非bug，已确认为有意设计）

1. **A4 是全策略禁入**而非仅同组 — PRD明确要求"不限同组"，已修正注释
2. **TAKE_PROFIT 冷却 > STOP_LOSS 冷却** — 反直觉但数据支撑：TP意味着趋势走完，重入是逆趋势
3. **getExitReasonMultiplier 未覆盖 OPTION_EXPIRED / BROKER_NO_POSITION** — 这些路径走 default:1.0，但它们已在各自路径中有独立的冷却计算，不经过统一乘数

### 后续可改进（非阻塞）

- `parseInt`/`parseFloat` 用于 JSONB 字段违反规则 #3（应用 `Number()` + `isNaN`），功能上不影响，规范上应统一
- computeCooldownUntil 在部分路径被重复调用两次（一次写DB，一次传 recordSymbolExit），存在毫秒级时间不一致
- f6 fallback 阈值 0.5% 在低波动标的上偏低，可能导致冷却效果偏弱
