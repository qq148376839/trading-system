# 同标的重入冷却机制 — 第一性原理审查

> 审查时间: 2026-03-14
> 触发事件: 3/13 IWM C246 止盈退出 22 秒后秒级重入 C248，冷却机制未生效
> 文件: `api/src/services/strategy-scheduler.service.ts`

---

## 一、核心需求（第一性原理）

**需求只有一句话：标的退出后，等待 N 分钟再允许重入。**

N 的值可以根据场景不同：
- 0DTE 首笔：0 分钟
- 0DTE 非首笔：1-3 分钟
- 0DTE 连亏：3-15 分钟
- 非 0DTE：10-20 分钟

这本质上是一个 **"退出时写入 deadline，入场前检查 deadline"** 的两步操作。

---

## 二、当前实现的复杂度分析

### 2.1 冷却信息分散在 6 个字段

冷却检查（L2424-2474）依赖以下字段：

| 字段 | 用途 | 写入位置 |
|------|------|---------|
| `lastExitTime` | 退出时间戳 | 各退出路径 |
| `dailyTradeCount` | 当日交易次数 | 各退出路径 |
| `consecutiveLosses` | 连续亏损次数 | 各退出路径 |
| `lastTradePnL` | 上一笔盈亏 | 各退出路径 |
| `optionMeta.expirationMode` | 是否 0DTE | 入场时写入 |
| `is0DTE` | 是否 0DTE（备用） | 持仓监控写入 |

**问题**: 冷却分钟数在**检查时动态计算**，而非**退出时一次性决定**。这意味着检查时必须能读到完整的退出上下文，增加了对数据完整性的依赖。

### 2.2 冷却规则有 8 个分支

```
if 0DTE:
  if consecLosses >= 3 → 15min
  if consecLosses == 2 → 5min
  if consecLosses == 1 → 3min
  if dailyTradeCount == 0 → 0min
  if dailyTradeCount <= 3 && lastPnL > 0 → 3min
  if dailyTradeCount <= 3 && lastPnL <= 0 → 1min
  if dailyTradeCount >= 4 → 3min
else:
  if consecLosses >= 2 → 当日禁止
  if lastPnL > 0 → 10min
  else → 20min
```

这些分支全部可以在**退出时**计算好，只存一个 `cooldownUntil` 时间戳。

### 2.3 冷却规则重复实现 2 次

完全相同的 0DTE 冷却逻辑出现在：
- `evaluateIdleSymbol()` L2424-2474（正常调度路径）
- `handleHoldingState()` L1945-1985（LATE 时段路径）

违反 DRY 原则，任何修改都需要同步两处。

### 2.4 IDLE 转换路径有 8 条，冷却信息写入不一致

| # | 退出路径 | lastExitTime | dailyTradeCount | 问题 |
|---|---------|:---:|:---:|------|
| 1 | trade-push 卖出回调 (L1260) | ✓ | ✓ | **主路径，但可能被 #2 抢先** |
| 2 | processClosingState (L4413) | ✗ | ✗ | **竞态：不写任何冷却字段** |
| 3 | 动态止盈 broker 无持仓 (L4192) | ✓ | ✓ | OK |
| 4 | 期权过期无价格 (L3484) | ✓ | ✓ | OK |
| 5 | 期权过期有价格 broker 无持仓 (L4059) | ✓ | ✓ | OK |
| 6 | 定期核对 broker 无持仓 (L4330) | ✓ | ✓ | OK |
| 7 | MIT 保护单触发 (L3818) | ✓ | ✓ | 但 consecutiveLosses 不更新 |
| 8 | Iron Dome (L5182) | ✓ | ✓ | OK |

**路径 #2 是 bug 的直接原因。** processClosingState 是一个轮询兜底机制，在状态为 CLOSING 时检查券商持仓是否已清零。它在 trade-push 回调之前检测到持仓为 0，直接调用 `updateState(symbol, 'IDLE')` 不传 context。

## 三、根因：状态机职责模糊

### 3.1 CLOSING 状态有两个 owner

```
动态止盈止损 → updateState(CLOSING) → 提交卖出订单
                                         ↓
                    ┌────────────────────────────────────┐
                    │                                    │
           processClosingState                   trade-push 回调
           (轮询: 无持仓 → IDLE)              (事件: 成交 → IDLE)
           ❌ 不写冷却字段                     ✓ 写冷却字段
           15:47:27 先到                      15:47:47 后到
                    │                                    │
                    └──────── 竞态条件 ─────────────────┘
```

**第一性原理问题**: 一个状态转换不应该有两个不协调的触发者。processClosingState 的设计初衷是"兜底"（万一 trade-push 丢失），但它不具备计算冷却信息的能力（没有成交价、手续费等），却有能力完成状态转换。

### 3.2 冷却信息写在错误的时间点

当前设计：冷却信息在 **IDLE 转换时** 才写入（L1260-1280）。

问题：IDLE 转换可能由没有完整信息的路径（processClosingState）先完成。

**正确做法**: 冷却信息应在 **退出决策时** 就确定（此时拥有全部信息：entryPrice、exitPrice、PnL、连续亏损等），不依赖 IDLE 转换的时机。

## 四、设计简化方案

### 核心思想：退出时决定，入场时检查

```
退出时（HOLDING → CLOSING）:
  1. 计算 cooldownUntil = now + cooldownMinutes
  2. 写入 context: { cooldownUntil, dailyTradeCount+1, ... }

入场时（evaluateIdleSymbol）:
  1. if (now < cooldownUntil) → 跳过
```

### 具体改动

#### 1. 新增 `computeCooldownMinutes()` 纯函数

```typescript
function computeCooldownMinutes(params: {
  is0DTE: boolean;
  dailyTradeCount: number;
  consecutiveLosses: number;
  lastPnL: number;
}): number {
  const { is0DTE, dailyTradeCount, consecutiveLosses, lastPnL } = params;

  if (!is0DTE) {
    if (consecutiveLosses >= 2) return Infinity; // 当日禁止
    return lastPnL > 0 ? 10 : 20;
  }

  // 0DTE
  if (consecutiveLosses >= 3) return 15;
  if (consecutiveLosses === 2) return 5;
  if (consecutiveLosses === 1) return 3;
  if (dailyTradeCount === 0) return 0;
  if (dailyTradeCount <= 3) return lastPnL > 0 ? 3 : 1;
  return 3;
}
```

#### 2. 在 CLOSING 转换时写入 `cooldownUntil`

在动态止盈止损触发处（L4238 `updateState(symbol, 'CLOSING', ...)`），计算并写入：

```typescript
const cooldownMinutes = computeCooldownMinutes({ is0DTE, dailyTradeCount, consecutiveLosses, lastPnL: pnl.netPnL });
await strategyInstance.updateState(symbol, 'CLOSING', {
  ...context,
  exitReason: action,
  // 冷却信息在退出决策时就确定
  cooldownUntil: new Date(Date.now() + cooldownMinutes * 60000).toISOString(),
  lastExitTime: new Date().toISOString(),
  dailyTradeCount: (context.dailyTradeCount ?? 0) + 1,
  consecutiveLosses: pnl.netPnL < 0 ? (context.consecutiveLosses ?? 0) + 1 : 0,
  lastTradePnL: pnl.netPnL,
  dailyRealizedPnL: (parseFloat(String(context.dailyRealizedPnL ?? 0)) + pnl.netPnL),
});
```

#### 3. 简化 processClosingState

processClosingState 不再需要特殊处理，因为冷却信息已在 CLOSING 时写入：

```typescript
// L4413: 原本不传 context 的 IDLE 转换现在安全了
// 因为 cooldownUntil 等字段已在 CLOSING 时写入，JSONB || 保留旧值
await strategyInstance.updateState(symbol, 'IDLE');
```

#### 4. 简化冷却检查

```typescript
// evaluateIdleSymbol 中的冷却检查
const cooldownUntil = cancelCtx?.cooldownUntil;
if (cooldownUntil && new Date(cooldownUntil).getTime() > Date.now()) {
  const remainMin = Math.ceil((new Date(cooldownUntil).getTime() - Date.now()) / 60000);
  summary.idle.push(`${symbol}(COOLDOWN_${remainMin}m)`);
  return null;
}
```

从 ~50 行分支逻辑 → **3 行检查**。冷却规则集中在一个纯函数中，易于测试和修改。

#### 5. 消除 LATE 时段重复

handleHoldingState 中的冷却逻辑（L1945-1985）也用同一个 `cooldownUntil` 检查替代，不再重复计算。

### 改动总量

| 变更 | 说明 |
|------|------|
| 新增 `computeCooldownMinutes()` | ~20 行纯函数 |
| 修改 CLOSING 转换（~3 处） | 加入冷却信息写入 |
| 简化 evaluateIdleSymbol 冷却检查 | 从 ~50 行 → ~5 行 |
| 简化 handleHoldingState 冷却检查 | 从 ~40 行 → ~5 行 |
| processClosingState | 无需改动（已安全） |

### 不改动的部分

- trade-push 回调（L1260）仍然写入精确的 PnL/手续费信息覆盖 CLOSING 时的估算
- 其他退出路径（#3-#8）继续正常工作
- 竞态条件自动消除：无论 processClosingState 还是 trade-push 先到，cooldownUntil 都已存在

## 五、3/13 IWM 重入事件还原

### 时间线（秒级精度）

```
15:47:22.067  动态止盈触发 TAKE_PROFIT (+31.9%)
              → updateState('IWM.US', 'CLOSING', {...context, exitReason, exitPrice...})
              ❌ 未写入 lastExitTime/dailyTradeCount/cooldownUntil

15:47:22.302  提交卖出市价单

15:47:22.971  收到订单变更（新订单）

15:47:23.468  持仓变 0（订单成交）

15:47:26.956  processClosingState 轮询到 CLOSING 状态
              → 检查券商：无持仓 + 无待成交卖单
              → updateState('IWM.US', 'IDLE')  ← context=null，保留 CLOSING 的 context
              → CLOSING context 中无 lastExitTime/cooldownUntil → 冷却检查被跳过

15:47:40.986  下一调度周期：IWM.US 为 IDLE，进入 evaluateIdleSymbol
              → cancelCtx.lastExitTime = undefined（CLOSING 时没写入）
              → cooldownMinutes > 0 但 lastExitTime 为空 → if 条件不满足 → 冷却被跳过

15:47:43.435  Phase B: IWM.US 通过所有检查，进入候选
15:47:43.436  Phase C: IWM.US 竞价胜出
15:47:44.668  IWM C248 买入成功 — 距卖出仅 22 秒

15:47:46.850  trade-push 回调终于处理 C246 卖出成交
              → 但 IWM.US 已变为 HOLDING（C248 已入场）
              → instanceKeySymbol 映射可能指向 IWM.US → 覆盖新的 HOLDING context
```

### 根因链

```
CLOSING 转换不写冷却字段
  → processClosingState 抢先完成 IDLE 转换（不带 context）
    → evaluateIdleSymbol 读到的 context 无冷却信息
      → 冷却检查被跳过
        → 22 秒重入
```

## 七、结论

| 维度 | 当前设计 | 简化方案 |
|------|---------|---------|
| 冷却字段数 | 6 个 | 1 个（`cooldownUntil`） |
| 冷却规则位置 | 2 处重复（evaluateIdle + handleHolding） | 1 个纯函数 |
| 冷却检查代码量 | ~50 行/处 × 2 = ~100 行 | ~5 行/处 × 2 = ~10 行 |
| IDLE 写入路径 | 8 条，1 条有 bug | 不依赖 IDLE 路径 |
| 竞态安全 | ✗（processClosingState 竞态） | ✓（CLOSING 时已写入） |
| 可测试性 | 需要模拟完整调度周期 | 纯函数可单元测试 |

**核心问题不是某个路径漏写了字段，而是"在错误的时间点写入冷却信息"的设计缺陷。** 冷却决策应在退出时做出（此时信息完整），而非在 IDLE 转换时才拼凑。
