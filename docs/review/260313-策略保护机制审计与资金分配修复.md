> 审计时间: 2026-03-13
> 审计范围: 策略 10（7000刀期权），strategy-scheduler / capital-manager 全链路
> 触发原因: 资金申请被拒绝 + 跨组 floor 连锁误过滤，导致修复后仍无法入场

## 一、修复的问题

### 问题 1: BY_GROUP 资金分配在期权竞价路径不生效

**现象**: 前端已切换 `capitalSplitMode: BY_GROUP`，但策略 10 资金申请持续被拒绝，日志显示 `1 个标的, 集中度上限33%`（BY_SYMBOL 行为）。

**根因**: `resolveDenominator()` 的优先级设计缺陷。期权策略走 Phase D 竞价路径，总会传 `survivorCount >= 1`。survivorCount 分支（优先级 1）直接返回 `maxConcentration ?? 0.33`，完全跳过 BY_GROUP 检查（优先级 2）。

**计算过程**:
```
allocatedAmount = 2000, survivorCount = 1, concentration = 0.33
maxPositionPerSymbol = min(2000/1, 2000*0.33) = min(2000, 660) = 660
申请金额 761~999 > 660 → 全部被拒绝
```

**修复**: survivorCount 分支中增加 BY_GROUP 检查，BY_GROUP 模式下 concentration 使用 `1/groupCount`。

**修复后**:
```
allocatedAmount = 2000, survivorCount = 1, groupCount = 2, concentration = 0.5
maxPositionPerSymbol = min(2000/1, 2000*0.5) = min(2000, 1000) = 1000
申请金额 761~999 < 1000 → 全部通过
```

**修改文件**: `api/src/services/capital-manager.service.ts` (resolveDenominator)
**提交**: `f82e94a`

### 问题 2: 跨组 floor 连锁过滤误阻所有标的

**现象**: 资金修复部署后，37 笔 PENDING 信号全部被 `R5v2_PHASE2_FILTERED` 的"跨组 floor 连锁过滤"拦截，持续 17+ 分钟。ORCL (GROUP_0) 止损退出后，不仅阻止了同组的 IWM/QQQ/SPY，还阻止了不同组的 NVDA/TSLA/MU。

**根因 (P0 Bug)**: 记录 floor exit 时调用 `getCorrelationGroup(symbol)` 未传入 `correlationMap`，导致 ORCL 的组名被解析为 `'ORCL.US'`（默认 map 无此标的，返回 symbol 自身）。检查时传了策略配置的 correlationMap，候选标的组名正确（如 `'GROUP_0'`、`'NVDA组'`）。`'ORCL.US' !== 'GROUP_0'` 和 `'ORCL.US' !== 'NVDA组'` 均为 true，所有标的都被误判为"不同组"，全部触发跨组 floor 连锁。

**根因 (P1 设计缺陷)**: 即使修复 P0 bug，floor 连锁的设计意图——"A 组 floor exit → 阻止 B 组 30 分钟"——与相关性分组的第一性原理矛盾。分组的目的是区分不相关标的，如果 ORCL 和 NVDA 不相关，ORCL 的止损不应影响 NVDA 入场。

**冗余分析**: floor 连锁保护的风险已被三重机制覆盖：
- **冷却期**: 管单标的退出后的时间间隔
- **同组互斥**: 管同组并行持仓
- **日内熔断**: 管策略级累计亏损阈值

**处理**: 移除 floor 连锁机制（实盘版 + 回测版同步清理）。

**修改文件**:
- `api/src/services/strategy-scheduler.service.ts` (crossSymbolState 数据结构、scoringAuction Phase 2、退出记录点)
- `api/src/services/option-backtest.service.ts` (回测版 floor 连锁)

**提交**: `45f60ad`

## 三、策略保护机制全面审计

### 审计结论: 移除 floor 连锁后无可安全移除的冗余

14 个实际存在的保护机制（`consecutiveConfirmCycles` 和 `rsiFilter` 代码中不存在）逐一审计：

### 不可移除的机制

| # | 机制 | 保护维度 | 不可替代原因 |
|---|------|---------|-------------|
| 1 | Phase 1 同组竞价 | 同 cycle 同组去重 | 消除对 Phase D 顺序执行的隐式依赖 |
| 2 | Phase 2 跨组并发(3min) | 跨 cycle 跨组并发 | 条件3 只管同 cycle |
| 4 | Phase 2 条件3 跨组竞争 | 同 cycle 跨组去重 | 条件1 只管跨 cycle |
| 5 | 冷却期 | 同标的退出后时间间隔 | 唯一单标的级别时间保护 |
| 7 | 日内熔断 | 策略级累计亏损 | 最后一道防线 |
| 8 | tradeWindow | 收盘前不开仓 | 唯一绝对时间窗口入场限制 |
| 9 | 开盘冲量守卫 | 开盘市场状态 | 信号质量维度，调度层无替代 |
| 12 | 资金检查 | 资金充足性 | 唯一资金维度 |
| 13 | 同标的期权持仓检查 | broker 端底层标的持仓 | 粒度不同于策略状态检查 |
| 14 | pending 订单检查 | 底层标的级别未成交 | 粒度不同于精确 symbol 检查 |
| 15 | zdteCooldownMinutes | 0DTE 开盘绝对窗口 | 与非首笔冷却维度不同 |
| 17 | 0DTE 非首笔冷却 | 0DTE 退出后相对窗口 | 与 zdteCooldownMinutes 维度不同 |

### 安全类冗余（保留，属于防御纵深）

| # | 机制 | 冗余于 | 保留理由 |
|---|------|--------|---------|
| 3 | Phase 2 同组互斥 | evaluateIdleSymbol 同组互斥 | evaluateIdleSymbol 是性能前置过滤，scoringAuction 是安全兜底 |
| 10 | validateStrategyExecution 部分子检查 | 前置路径 pending/状态检查 | 执行层最后防线，含高买低卖/60s去重等独有检查 |

### 分层防御架构

```
信号层 (generateSignal)
  ├─ 开盘冲量守卫
  └─ zdteCooldownMinutes

评估层前置过滤 (evaluateIdleSymbol)
  ├─ tradeWindow 限制
  ├─ 日内熔断检查
  ├─ 冷却期（0DTE / 非0DTE）
  ├─ 同标的期权持仓检查
  ├─ pending 订单检查
  └─ 同组互斥（前置）

竞价层去重 (scoringAuction)
  ├─ Phase 1: 同组竞价
  └─ Phase 2: 跨组并发(3min) + 同组互斥(兜底) + 同cycle跨组竞争

执行层兜底 (executeSymbolEntry + validateStrategyExecution)
  ├─ 高买低卖检测
  ├─ 重复买入检测
  ├─ pending 订单检测
  ├─ 持仓量验证
  └─ 60s 重复提交去重

资金层确认 (requestAllocation)
  ├─ 可用资金检查
  └─ 单标的集中度上限
```

### 发现的其他问题（非阻塞）

1. **processSymbol 非期权路径的冷却逻辑**: 使用简单配置值 `latePeriod.cooldownMinutes ?? 3`，未实现升级机制（盈利/亏损/连亏区分）。如果未来非期权策略需要类似机制，需同步。

2. **`non0DTECooldownMinutes` 命名混淆**: 值来源于 `zdteCooldownMinutes` 配置，但语义完全不同（一个是"0DTE 开盘禁入窗口"，另一个是"非0DTE 入场后冷静期"）。不影响运行但增加理解成本。
