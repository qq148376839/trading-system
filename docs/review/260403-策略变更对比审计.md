## 二、逐模块详细对比

### 2.1 退出机制重构 — maxHoldMinutes → Theta Bleed 检测

**文件**: `api/src/services/option-dynamic-exit.service.ts`

#### 旧版 (f54ebf22)
```
固定时间止损: 持仓超过 exitRules.maxHoldMinutes 且 PnL < 5% → TIME_STOP
判据: 单一时间阈值，由配置决定
```

#### 新版 (HEAD)
```
PnL 轨迹斜率检测: 基于线性回归判断 theta 侵蚀
触发条件:
  1. 持仓 ≥ 5min (BLEED_WARMUP)
  2. 历史峰值 PnL < 4% (BLEED_PEAK_THRESHOLD)
  3. 当前 PnL 在 [-10%, 4%) 区间
  4. 3min 回看窗口内 ≥ 8 条快照
  5. 斜率在 (-0.5, 0] %/min (非方向错误的缓慢衰减)
  6. 窗口内所有点 ≤ 首点 + 0.5% (连续非改善)
退出标签: theta_bleed (替代 max_hold_minutes)
```

#### 审计发现

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| **E1** | **exitRules 接口删除 maxHoldMinutes 字段** | INFO | `ExitRulesOverride` 接口已移除 `maxHoldMinutes?` 属性，数据库迁移也清理了。确认向后兼容 |
| **E2** | **BLEED_STEEP = -0.5 的边界** | **WARN** | 文档称"最缓方向错 -0.60"，阈值设为 -0.5。若实际分布有 -0.55 的方向错误交易，会误判为 theta bleed 提前退出 |
| **E3** | **BLEED_PNL_LOWER = -10 与 STOP_LOSS 间隙** | **WARN** | theta bleed 仅在 PnL ≥ -10% 时生效，STOP_LOSS 在 -20%。-10% 到 -20% 之间的持仓如果缓慢下跌，既不触发 bleed 也不触发 stop loss，可能形成**无人管辖区** |
| **E4** | **线性回归对非线性衰减的适用性** | INFO | Theta 衰减理论上是非线性的（到期日加速），线性回归可能低估尾盘衰减斜率 |
| **E5** | **BLEED_LOOKBACK_MS = 3min 但 BLEED_MIN_SNAPSHOTS = 8** | INFO | 以 15s 采样间隔，3min 最多 12 条，需至少 8 条才计算。即有效窗口 = [2min, 3min]。若采样不稳定（延迟/跳过），可能在需要检测时无法触发 |

### 2.2 方向系统重构 — 权重/趋势/去噪

**文件**: `api/src/services/option-recommendation.service.ts`

#### 综合得分公式变更

| 组件 | 旧版权重 | 新版权重 | 变化 |
|------|---------|---------|------|
| 大盘环境评分 (marketScore) | 20% | 20% | 不变 |
| 分时动量评分 (intradayScore) | 60% | **80%** | +20% |
| 时间窗口调整 (timeWindowAdjustment) | 20% | **0%** | **移除** |

**关键变化**: timeWindowAdjustment 不再参与方向得分计算，改为 `timeThresholdFactor` 修正入场阈值。

#### 分时动量评分内部权重变更

| 子组件 | 旧版权重 | 新版权重 | 变化 |
|--------|---------|---------|------|
| 底层1m动量 | 30% | **45%** | +15% |
| VWAP位置 | 15% | **20%** | +5% |
| SPX日内 | 25% | **35%** | +10% |
| BTC时K | 15% | **0%** | **归零** |
| USD时K | 15% | **0%** | **归零** |
| 合计 | 100% | **100%** | ✓ |

#### 大盘环境评分变更

| 子组件 | 旧版 | 新版 | 变化 |
|--------|------|------|------|
| SPX趋势 | 单一 `analyzeMarketTrend` (20日均线) | **`analyzeMarketTrendMultiWindow`** (3d/10d/20d 加权) | 重写 |
| Gap信号 | 无 | **新增** `calculateGapSignal` (±0.3%阈值, 最大±15分) | 新增 |
| BTC共振 | 共振0.2/不共振0.1 | **共振0.1/不共振0.05** | 减半 |
| USD/VIX | 不变 | 不变 | — |

#### 审计发现

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| **R1** | **BTC/USD 分时权重归零但计算保留** | INFO | 代码仍执行 `calculateMomentum`，仅权重设为 0，浪费算力但不影响结果。如确认永久去除应删除代码 |
| **R2** | **timeWindowAdjustment 被移除出得分但仍保留计算** | **WARN** | `calculateTimeWindowAdjustment()` 仍被调用"用于日志"，但其返回值之前包含方向偏置(+20=CALL)。如果未来有人误将其重新加入公式，会重新引入偏置 |
| **R3** | **Gap 信号线性映射上限** | INFO | `gapPct * 8` 封顶±15，意味着 gap ≥ 1.875% 后得分饱和。对于极端跳空（如财报后 5%+），信号强度不会进一步增加 |
| **R4** | **多窗口趋势分析硬编码 avg 计算** | INFO | `data.slice(-3).reduce(...) / Math.min(3, data.length)` — 当 data.length ≥ 20（前面已检查），`Math.min(3, data.length)` 永远等于 3，`Math.min` 冗余但无害 |
| **R5** | **多窗口一致性判断缺少 sign=0 过滤** | **WARN** | 当 trend3d ≈ 0（盘整）时，`sign3 = 0`，不会命中第一个 if（所有同向）也不会命中第二个 if（短长反向），落入混合状态。这可能是预期行为，但在极窄震荡市中权重分配依赖默认值 |
| **R6** | **开盘第1小时 timeThresholdFactor=0.85** | **WARN** | 降低入场阈值 15%。如果原始阈值刚好卡在边界线上（如 directionalScoreMin=30），降低后变为 ~25.5，可能放入低质量交易。需确认此偏移量在回测中验证过 |
| **R7** | **方向系统权重总和验证** | **WARN** | 大盘环境评分内部: SPX 0.4 + Gap 最多±15(加法) + USD 0.2 + BTC 0.05~0.1 + VIX 0.1。SPX/USD/BTC/VIX 权重和 = 0.75~0.8（不含 Gap），加上 Gap 的加法信号，总分可超过 ±100 的理论范围。虽然最终有 `Math.max(-100, Math.min(100, ...))` 裁剪，但 Gap 的加法性质与其他乘法性质不一致 |

### 2.3 入场策略 — 时间阈值修正系数

**文件**: `api/src/services/strategies/option-intraday-strategy.ts`

#### 变更内容
- 新增 `currentTimeThresholdFactor` 字段，从 `optionRec.timeThresholdFactor` 接收
- `getThresholds()` 中入场阈值公式变为: `baseThreshold × vixFactor × timeFactor`
- 影响 `directionalScoreMin` 和 `spreadScoreMin` 两个阈值

#### 审计发现

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| **S1** | **timeFactor 与 vixFactor 乘法叠加** | **WARN** | 高VIX时 vixFactor > 1（提高阈值），开盘首小时 timeFactor=0.85（降低阈值）。两者方向相反可能互相抵消，导致高波动开盘时阈值接近正常值。是否符合预期？ |
| **S2** | **timeFactor 尾盘最高 3.0** | INFO | 15:30+ 阈值 ×3，directionalScoreMin 30 → 90，几乎不可能入场。这是预期的（尾盘拒绝入场） |

### 2.4 调度核心 — PnL 快照机制

**文件**: `api/src/services/strategy-scheduler.service.ts`

#### 变更内容
1. `POSITION_CONTEXT_RESET` 新增 `pnlSnapshots: []`
2. `POSITION_EXIT_CLEANUP` 新增 `pnlSnapshots: null`
3. `PositionContext` 传递 `pnlSnapshots` 到退出服务
4. 每周期追加快照并持久化（保留最近 20 条）
5. `maxHoldMinutes` 从 exitRulesOverride 中移除

#### 审计发现

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| **C1** | :rotating_light: **不可达代码 — `return { actionTaken: false }` (第 4756 行)** | **CRITICAL** | 第 4754 行 `return { actionTaken: true }` 之后，第 4756 行 `return { actionTaken: false }` **永远不会执行**。这是旧版条件分支重构后的遗留。旧版逻辑: `if (currentPnL > peak) { updateState; return true; }` + `return false;`（只有 peak 更新时才持久化）。新版每周期都 updateState 后直接 return true，但忘记删除 else 分支的 return false。**后果**: 此函数现在无论 PnL 是否更新 peak 都返回 `actionTaken: true`，语义变化可能影响上游调用者 |
| **C2** | **每周期都 updateState 的性能影响** | **WARN** | 旧版仅在 peak 更新时写数据库。新版每个 holding 周期都执行一次 `updateState`（写入 pnlSnapshots）。以 15s 间隔，一个持仓每分钟 4 次 DB 写入。多持仓时可能显著增加数据库负载 |
| **C3** | **pnlSnapshots 持久化 bug 已修复但需验证** | INFO | commit `0b9c20f` 修复了"满20条后 updateState 条件恒 false"的 bug，现在每周期无条件 updateState。但这个修复方式（每周期都写）本身引入了 C2 问题 |

### HIGH（建议尽快评估）

| ID | 问题 | 建议 |

## 四、变更影响矩阵

```
入场方向判断  ← R1~R7 影响（权重重构可能导致方向判断偏移）
   ↓
入场阈值     ← S1~S2 影响（timeFactor 降低/提高阈值）
   ↓
持仓监控     ← C1~C3 影响（每周期写入 + 不可达代码）
   ↓
退出决策     ← E1~E5 影响（theta bleed 替代固定时间止损）
```

**总结**: 此次变更是方向系统和退出机制的重大重构。最关键的 bug 是 C1（不可达代码），最需要验证的风险是 E3（PnL 管辖区间隙）和方向系统权重变更后的整体方向准确率。
