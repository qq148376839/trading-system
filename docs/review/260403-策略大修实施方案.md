**日期**: 2026-04-03
**状态**: Phase 0-3 已完成部署，Phase 4 待实施
**最后更新**: 2026-04-03 — Phase 0-3 实施完成 + 入场时机数据验证结果

## Context

2026-04-01 完成的方向系统 + 退出机制重构在 4-02 实盘中产生灾难性结果: 盈亏比 0.03:1、胜率 33%、日亏 $826。
根因分析揭示三个结构性缺陷: (1) K 线分析不足 (2) PnL 多源分裂 (3) 入场滞后。

**新发现**: LongPort SDK 已内置 WebSocket 订阅能力（`subscribe()` + `setOnQuote()` + `subscribeCandlesticks()`），但系统完全未使用，仍通过 5 秒轮询获取行情。升级到订阅架构可将数据延迟从 0~5 秒降至 ~50-200ms，从根本上改善入场滞后问题。

用户确认的方向:
- K 线: 1min + 5min 双时间框架 + volume 分析
- 权重: 恢复到 20/60/20 三足鼎立，市场评分增加分钟级 K 线（不只用日 K）
- PnL: 第一性原理分析使用场景后决定 → 结论: 只需改前端分析页读 auto_trades
- 入场时机: 先拉数据验证假设，再决定具体方案
- 行情架构: 从 5 秒轮询升级到 WebSocket 订阅推送（讨论中）

## 一、PnL 第一性原理分析 — 不需要大改

### PnL 消费者地图

| 消费者 | 数据源 | 准确性 | 重要性 |
|--------|--------|--------|--------|
| **日内熔断器** | `strategy_instances.context.dailyRealizedPnL` | broker 成交价 ✅ | CRITICAL |
| **连续亏损计数** | order callback 中的 `tradePnL` | broker 成交价 ✅ | HIGH |
| **Dashboard 统计** | `auto_trades.pnl` | broker 成交价 ✅ | MEDIUM |
| **前端分析页** | `strategy_signals.metadata.netPnL` | 决策时估算 ❌ | MEDIUM |

**结论**: 3/4 的消费者已经使用正确数据。唯一错误的是前端分析页。

### 方案: 最小改动 — 分析页改读 auto_trades

**不改后端**。信号表继续存估算值（用于滑点分析）。前端分析页的 PnL 改为从 `auto_trades` 读取。

**改动范围**:
- `frontend/app/quant/analysis/page.tsx` — 配对逻辑改为 JOIN auto_trades 获取实际 PnL
- `api/src/routes/quant.ts` — 分析 API 端点返回 auto_trades.pnl 而非 signal metadata PnL

**改动量**: ~50 行前端 + ~30 行后端路由

## 二、入场时机 — 先拉数据验证

### 需要验证的假设

**核心假设**: 入场时 momentum 评分已达峰值，标的价格已完成主要移动，入场即均值回归。如果提前 3-5 分钟入场（momentum 构建阶段），盈利可提高 30%+。

### 数据验证脚本

写一个 Node.js 脚本 `scripts/analyze-entry-timing.ts`，在 NAS 生产环境执行:

**Step 1**: 拉取 3 月至今所有 EXECUTED BUY 信号 + 对应 SELL 信号 + auto_trades 实际 PnL

```sql
SELECT
  bs.id as buy_signal_id, bs.symbol, bs.created_at as entry_signal_time,
  bs.metadata->>'finalScore' as entry_score,
  eo_buy.created_at as buy_order_time,
  ss.created_at as exit_signal_time,
  ss.metadata->>'exitReason' as exit_reason,
  at.pnl as actual_pnl, at.avg_price as sell_price,
  (SELECT avg_price FROM auto_trades WHERE symbol = bs.symbol
   AND side = 'BUY' AND created_at BETWEEN bs.created_at - INTERVAL '1 min'
   AND bs.created_at + INTERVAL '5 min' LIMIT 1) as buy_price
FROM strategy_signals bs
JOIN execution_orders eo_buy ON bs.id = eo_buy.signal_id AND eo_buy.side = 'BUY'
LEFT JOIN strategy_signals ss ON ss.strategy_id = bs.strategy_id
  AND ss.symbol = bs.symbol AND ss.signal_type = 'SELL'
  AND ss.created_at > bs.created_at
  AND ss.created_at < bs.created_at + INTERVAL '4 hours'
LEFT JOIN auto_trades at ON at.symbol = bs.symbol AND at.side = 'SELL'
  AND at.created_at BETWEEN ss.created_at - INTERVAL '2 min'
  AND ss.created_at + INTERVAL '5 min'
WHERE bs.signal_type = 'BUY' AND bs.status = 'EXECUTED'
  AND bs.created_at >= '2026-03-01'
ORDER BY bs.created_at;
```

**Step 2**: 对每笔入场，查询入场前 5 分钟的系统日志（评分变化轨迹）

```sql
SELECT created_at, message, metadata
FROM system_logs
WHERE module LIKE '%OptionStrategy%'
  AND message LIKE '%${symbol}%'
  AND created_at BETWEEN '${entry_time}'::timestamp - INTERVAL '5 minutes'
  AND '${entry_time}'::timestamp
ORDER BY created_at;
```

> 注: 如果 system_logs 不存 decision log，需从 Docker 容器日志 grep。脚本应同时支持两种来源。

**Step 3**: 分析输出

对每笔交易，输出:
- 入场前 5min/3min/1min 的评分快照
- 入场时的评分峰值
- 实际 PnL
- 假设提前 N 分钟入场时标的价格（从 K 线数据反推）
- 假设 PnL（假设入场价 - 实际退出价）

**改动量**: 新增 `scripts/analyze-entry-timing.ts` ~200 行

## 三、K 线分析增强 — 1min + 5min 双框架 + Volume

### 3.1 数据层: 新增 5min K 线获取

**文件**: `api/src/services/market-data.service.ts`

在 `getIntradayVWAP()` 旁新增或扩展:
- 请求 LongPort `Period.Min_5` 获取 5 分钟 K 线（48 根 = 4 小时）
- 复用现有缓存机制（60s TTL）
- 返回 `{ klines1m: CandlestickData[], klines5m: CandlestickData[] }`

**改动**: `market-data.service.ts` +60 行

### 3.2 评分层: Volume-Weighted Momentum

**文件**: `api/src/services/option-recommendation.service.ts`

改造 `calculateMomentum()`:

**当前** (只用 close):
```
avgChange = mean(close[i]/close[i-1] - 1)  for i in last 10
momentum = (avgChange / atrPct) * 50
```

**改为** (volume-weighted):
```
// 使用全部可用 K 线（不再截断到 20 根）
vwChanges = Σ((close[i] - close[i-1]) / close[i-1] × volume[i]) / Σ(volume[i])
momentum = (vwChanges / atrPct) * 50
```

核心改动:
- 使用全部 K 线（最多 240 根 1min 或 48 根 5min），不再 slice(-20)
- close-to-close 变化率按 volume 加权
- 数据质量: 标的 K 线也过 `intradayDataFilterService`（当前只有 SPX 过了）
- **最低数据要求**: 从 5 根提高到 15 根

**改动**: `option-recommendation.service.ts` `calculateMomentum()` 重写 ~40 行

### 3.3 评分层: 5min 趋势确认

新增 `calculate5minTrendConfirmation()`:
- 输入: 5min K 线（48 根）
- 计算: 5min 级别的 volume-weighted momentum + 趋势方向
- 输出: 趋势确认分数 (-100 ~ +100)
- 当 1min 和 5min 方向不一致时，降低信号置信度（乘以 0.5-0.7 衰减系数）

**整合到 intradayScore 计算中**:
```
intradayScore 内部权重 (恢复后的 60% 总权重):
  底层1min动量(volume-weighted): 30%  (原 45%)
  5min趋势确认: 15%  (新增)
  VWAP位置: 15%  (原 20%)
  SPX分时: 25%  (原 35%)
  BTC/USD: 15%  (恢复，原 0%，降噪但不归零)
```

**改动**: `option-recommendation.service.ts` `calculateIntradayScore()` 重构 ~80 行

### 3.4 清理: 删除废弃代码

- 删除 `calculateTimeWindowAdjustment()` 废弃逻辑（已不参与评分但仍在计算）
- 删除 BTC/USD momentum 权重 0 的无效计算（恢复权重后保留计算但更新权重）
- 删除 `strategy-scheduler.service.ts:4756` 不可达代码 (C1 bug)

## 五、行情订阅架构升级 — 从 5 秒轮询到毫秒推送 ★ 新增

### 5.1 问题

当前数据路径 (`strategy-scheduler.service.ts:751-771`):
```
setInterval(5000ms) → quoteCtx.quote(idleSymbols) → priceMap → fastMomentumService.feedQuotes()
```
- 每 5 秒才能看到一次价格，两次轮询之间的价格波动完全不可见
- FastMo 的 60 秒窗口只有 12 个数据点做线性回归
- K 线数据通过 `candlesticks()` API 单独拉取，有 60 秒缓存

**实际案例 — 阶梯锁利漏损**:
```
Strategy 10 配置: threshold=10, floor=7
实际: 峰值 12.8% → 踩上 10% 台阶 → 底线锁 7%
      但下次 5s 检查时 PnL 已跌至 2.2% → 漏损 4.8%
```
底线 7% 形同虚设 — 5 秒检查间隔在 0DTE 高 gamma 环境下，等你"看见"价格时已经远离底线。
订阅推送可将检测延迟从 5s 降至 ~50-200ms，阶梯锁利的底线才能真正生效。

### 5.2 LongPort SDK 已有的订阅能力

确认自 `api/node_modules/longport/index.d.ts`:
- `quoteCtx.subscribe(symbols, [SubType.Quote], true)` — 实时报价推送
- `quoteCtx.setOnQuote(callback)` — 每笔成交/报价变动触发（毫秒级）
- `quoteCtx.subscribeCandlesticks(symbol, Period.Min_1)` — K 线推送
- `quoteCtx.setOnCandlestick(callback)` — K 线收线/实时更新推送

### 5.3 方案: 新增 QuoteSubscriptionService

**文件**: `api/src/services/quote-subscription.service.ts`（新增）

核心设计:
1. **初始化**: `setOnQuote()` 回调 → 喂入 FastMomentumService + 通知其他消费者
2. **K 线推送**: `setOnCandlestick()` 回调 → 收线时触发 momentum 重算
3. **订阅管理**: `subscribeSymbols()` / `unsubscribeSymbols()`
4. **降级**: WebSocket 断线时自动回退到 5 秒轮询，重连后恢复订阅

### 5.4 FastMomentumService 改造

- 新增 `feedSingleQuote(symbol, price)` — 单条实时报价喂入
- Buffer 容量: `BUFFER_CAPACITY` 12→120, `LONG_BUFFER_CAPACITY` 60→600

### 5.5 需要确认的点

1. LongPort 账户是否有美股 Level 1 实时行情权限
2. 期权代码是否支持 Quote 订阅
3. NAS Docker 环境的 WebSocket 连接稳定性

**详细分析**: `docs/analysis/260403-行情订阅架构升级分析.md`
