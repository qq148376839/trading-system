# 持仓同步修复 — 跨策略幽灵持仓 + 看门狗死循环

**日期**: 2026-03-17
**状态**: 已完成

## 问题

策略10卖出 SPY260316C673000.US 后，仅策略10的 `strategy_instances` 行更新为 IDLE。策略12/13 仍为 HOLDING。

0DTE看门狗每60秒检测到这些幽灵持仓并反复尝试平仓，产生120笔被拒订单死循环。

### 根因

卖出成交回调只更新执行卖出的策略行，不会同步清理其他策略中持有同一合约的行。多个策略监控同一标的（如 SPY.US）时，买入同一合约后，只有一个策略实际下了卖出单，其余成为幽灵持仓。

## 修复方案

### Fix A: 看门狗平仓前券商预检（打断死循环）

**文件**: `0dte-watchdog.service.ts`

- `forceClosePosition()` 重试循环前，调用 `calculateAvailablePosition(tradedSymbol)` 查询券商实际持仓
- 若 `actualQuantity <= 0`，直接 `stateManager.updateState → IDLE`，不再尝试下卖出单
- 修复 `result.submitted && result.orderStatus !== 'RejectedStatus'` 防止被拒订单误判成功

### Fix B: 看门狗重试全部失败后兜底

- 3次重试全部失败后，再次查询券商持仓
- 若已无持仓（可能被其他策略/手动平掉），清理DB状态

### Fix C: 卖出成交后跨策略同步（根因修复）

**文件**: `strategy-scheduler.service.ts`

卖出订单成交处理末尾（PnL日志之后、熔断检查之前）插入：

1. 查询 `strategy_instances` 中其他策略持有同一 `tradedSymbol` 的 HOLDING 行
2. 调用 `calculateAvailablePosition` 确认券商无持仓
3. 对每个幽灵行：
   - `capitalManager.releaseAllocation()` 释放资金
   - `trailingStopProtectionService.cancelProtection()` 取消保护单
   - `stateManager.updateState → IDLE`（使用 POSITION_EXIT_CLEANUP 规则 #12）
4. 整个块 try/catch 包裹，失败不阻塞主流程

## 安全性

- **规则 #7**: 纯增量代码，不移除任何安全防护
- **规则 #12**: 所有 HOLDING→IDLE 使用 POSITION_EXIT_CLEANUP 10个字段
- **容错**: 所有新代码在 try/catch 中
- **最小变更**: 仅修改2个文件，无 schema 变更
