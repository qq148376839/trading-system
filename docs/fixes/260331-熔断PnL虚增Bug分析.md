# 熔断 PnL 虚增 Bug — IRON_DOME 竞态条件

> 日期: 2026-03-31
> 严重度: **高**（直接导致策略误熔断，丧失交易机会）
> 影响范围: 所有 OPTION 策略的日内熔断判断
> 状态: **已修复**

---

## 现象

2026-03-30 交易日，策略 10 在 15:27 触发熔断：

```
[CIRCUIT_BREAKER] 策略 10 标的 TSLA.US: 日内累计亏损 $1055 >= 阈值 $609，暂停至收盘
```

但实际信号 PnL 汇总仅为 **-$115**，误差高达 **$940**。

## 数据取证

### strategy_instances 实际 dailyRealizedPnL

| 标的 | DB 值 | 信号推算值 | dailyTradeCount | 偏差 |
|---|---|---|---|---|
| TSLA.US | $7 | $7 | 4 | 0 |
| SPY.US | $120 | $120 | 1 | 0 |
| QQQ.US | $30 | $30 | 1 | 0 |
| **NVDA.US** | **-$1,212** | **-$272** | **3 (应为2)** | **-$940** |
| **合计** | **-$1,055** | **-$115** | | **-$940** |

### NVDA.US 当日交易还原

| 时间 (ET) | 操作 | 合约 | 成交价 | PnL |
|---|---|---|---|---|
| 09:51 | BUY | P167.5 × 10 | $0.94 | — |
| 09:53 | SELL | P167.5 × 10 | $0.99 | +$50 (扣费前) |
| 10:06 | BUY | P165 × 23 | $0.41 | — |
| 10:09 | SELL | P165 × 23 | $0.27 | -$322 (扣费前) |

**正确的 dailyRealizedPnL = 50 + (-322) = -272**
**DB 中的值 = -1212 = 50 + (-940) + (-322)**

多出的 **-$940** 恰好等于 P165 的 `allocationAmount`（23 × $0.41 × 100 ≈ $940）。

`dailyTradeCount = 3`（应为 2）进一步证实 IRON_DOME reconciliation 额外处理了一次退出。

## 根因分析

### 涉及代码路径

| 组件 | 文件:行号 | 作用 |
|---|---|---|
| IRON_DOME Reconciliation | `strategy-scheduler.service.ts:5463-5520` | 60s 轮询检测 "幽灵仓位"（broker=0 但 DB=HOLDING） |
| Phantom PnL 写入 | `:5507` | `dailyRealizedPnL = prev + (-allocationAmount)` |
| 竞态修正逻辑 | `:1322-1339` | 订单回调检测 BROKER_TERMINATED，撤销 IRON_DOME 估算 |
| 熔断聚合 SQL | `:1557-1562` | `SUM(dailyRealizedPnL)` 跨所有 instances |

### 竞态时序

```
  T0: P165 止损卖单成交，broker 持仓归零
  T1: IRON_DOME 60s 周期运行
      ├── SELECT ... WHERE current_state = 'HOLDING' → 包含 NVDA.US（回调还没跑）
      ├── 检测 broker 持仓 = 0 → 判定"幽灵仓位"
      ├── 写入: dailyRealizedPnL = 50 + (-940) = -890
      ├── 写入: exitReason = 'BROKER_TERMINATED'
      ├── 写入: dailyTradeCount = 2 + 1 = 3
      └── 状态: HOLDING → IDLE
  T2: 订单回调处理 P165 卖单
      ├── 读取 context → exitReason = BROKER_TERMINATED
      ├── 竞态修正: dailyRealizedPnL = -890 - (-940) = 50 ✓ (内存修正)
      ├── 计算实际 tradePnL = -322
      ├── newDailyPnL = 50 + (-322) = -272
      └── 写入: dailyRealizedPnL = -272
                                                  ← 如果修正成功，最终 = -272 ✓
```

### 问题：修正逻辑存在失败路径

竞态修正条件（`:1326-1327`）：

```typescript
const currentState = await strategyInstance.getCurrentState(instanceKeySymbol);
if (currentState === 'IDLE' && context.exitReason === 'BROKER_TERMINATED') {
  // 修正逻辑
}
```

**失败场景 1**：回调先于 IRON_DOME 运行

```
  T0: P165 卖单成交
  T1: 回调立即处理
      ├── 读取 context: exitReason = 'PROTECTION_MIT_FILLED'（来自上一笔 P167.5）
      ├── 修正检查: exitReason ≠ BROKER_TERMINATED → 跳过修正
      ├── newDailyPnL = 50 + (-322) = -272
      └── 写入: dailyRealizedPnL = -272, state = IDLE
  T2: IRON_DOME 运行（SELECT 在 T1 写入前发起）
      ├── 读到旧 state = HOLDING（事务隔离，T1 还未提交）
      ├── dailyRealizedPnL = 50 + (-940) = -890
      └── 写入覆盖: dailyRealizedPnL = -890  ← 回调的 -272 被覆盖!
```

**失败场景 2**：并发事务交叉写入

当 IRON_DOME 的 SELECT（拿到 HOLDING）和 UPDATE（写入 IDLE）与回调的 SELECT/UPDATE 交叉执行时，JSONB 合并 `context || $4::jsonb` 是 last-writer-wins 语义。任何一方的写入都可能被另一方覆盖。

在 READ COMMITTED 隔离级别下，两个并发 UPDATE 各自基于旧值计算新值，先提交者的 dailyRealizedPnL 会被后提交者覆盖——但后提交者计算时用的是旧值，导致 phantom 未被移除。

### 核心缺陷

1. **IRON_DOME 直接修改 `dailyRealizedPnL`**：把估算亏损（-allocationAmount）叠加到累计 PnL 上，污染了全局状态
2. **竞态修正是事后补救**：在回调中检测并撤销 IRON_DOME 的写入，但依赖严格的时序（IRON_DOME 先写 → 回调后读），实际时序不可控
3. **无原子保护**：两个写入路径（IRON_DOME + 回调）没有使用锁或条件更新，存在 lost-update 风险

## 修复方案

### 方案 A：IRON_DOME 不写 dailyRealizedPnL（推荐）

**原理**：IRON_DOME 的职责是检测幽灵仓位并触发熔断保护，不需要写入"估算 PnL"。直接设置熔断标志即可。

```typescript
// strategy-scheduler.service.ts:5501-5511
// 修改前:
await stateManager.updateState(strategyId, row.symbol, 'IDLE', {
  ...POSITION_EXIT_CLEANUP,
  exitReason: 'BROKER_TERMINATED',
  lastExitTime: new Date().toISOString(),
  lastTradePnL: allocationAmount > 0 ? -allocationAmount : 0,         // ← 删除
  dailyRealizedPnL: (prev + (-allocationAmount)),                      // ← 删除
  consecutiveLosses: (prev + 1),                                       // ← 保留
  dailyTradeCount: (prev + 1),                                         // ← 删除（让回调处理）
  cooldownUntil: brokerTermCooldown,
});

// 修改后:
await stateManager.updateState(strategyId, row.symbol, 'IDLE', {
  ...POSITION_EXIT_CLEANUP,
  exitReason: 'BROKER_TERMINATED',
  lastExitTime: new Date().toISOString(),
  lastTradePnL: null,                     // 不写估算值，等回调用实际值填充
  cooldownUntil: brokerTermCooldown,
  // dailyRealizedPnL: 不修改，保留当前值
  // dailyTradeCount: 不修改，让回调递增
});

// 熔断仍直接触发（保留 lines 5522-5602 的全策略熔断写入）
```

**优点**：从根本上消除竞态——dailyRealizedPnL 只有一个写入源（订单回调）
**风险**：如果回调永远不触发（真正的幽灵仓位），PnL 不会被记录。但这种场景下实际也不需要记录 PnL（仓位已消失）

同步删除回调中的竞态修正逻辑（`:1322-1339`），因为不再需要。

### 方案 B：使用 optimistic locking 防止并发覆盖

在 `strategy_instances` 表增加 `version` 列，UPDATE 时条件检查：

```sql
UPDATE strategy_instances
SET context = context || $1::jsonb,
    current_state = $2,
    version = version + 1
WHERE strategy_id = $3 AND symbol = $4 AND version = $5
```

如果返回 0 rows affected（被其他写入抢先），重新读取再试。

**优点**：彻底解决 lost-update
**缺点**：改动面大（所有 updateState 调用都要支持 version），引入重试逻辑

### 方案 C：熔断计算用信号表而非 instances

```sql
-- 替代 lines 1557-1562
SELECT COALESCE(SUM((metadata->>'netPnL')::numeric), 0) as total_pnl
FROM signals
WHERE strategy_id = $1
  AND signal_type = 'SELL'
  AND DATE(created_at AT TIME ZONE 'America/New_York') = CURRENT_DATE AT TIME ZONE 'America/New_York'
```

**优点**：signals 表是 append-only，不存在并发覆盖问题
**缺点**：需要每次聚合查询 signals 表（性能影响），且 signals 中的 PnL 可能包含 IRON_DOME 伪造的值

### 推荐：方案 A

最小改动，从根源解决。IRON_DOME 只做两件事：
1. 检测幽灵仓位 → 设置 exitReason='BROKER_TERMINATED' + 释放资金
2. 触发全策略熔断

不碰 `dailyRealizedPnL`、`dailyTradeCount`、`lastTradePnL`。

## 受影响范围

- `strategy-scheduler.service.ts`: IRON_DOME reconciliation loop（`:5463-5520`）
- `strategy-scheduler.service.ts`: 竞态修正逻辑（`:1322-1339`，可删除）
- 无需改动：熔断聚合 SQL、日重置逻辑、订单回调 PnL 计算

## 验证方法

1. 修改后重新部署
2. 在测试环境模拟：下单后立即取消（使 broker 持仓归零），观察 IRON_DOME 是否还写入 phantom PnL
3. 检查 `dailyTradeCount` 是否仍被 IRON_DOME 额外递增
4. 验证真正的幽灵仓位场景（期权到期归零）：IRON_DOME 仍能触发熔断保护，但不污染 PnL

## 临时缓解措施
