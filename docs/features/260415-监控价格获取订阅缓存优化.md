# 监控价格获取订阅缓存优化

**日期**: 2026-04-15
**状态**: 已实施

## 背景

系统已有 WebSocket 实时订阅通道（`quote-subscription.service.ts`），行情通过 `onQuote` 回调实时推送到 `priceMap`。但 HOLDING 阶段和订单监控仍每个周期无条件调用 `quoteCtx.quote()` API：

- 期权策略：5s 一次冗余 API 调用
- 正股策略：60s 一次冗余 API 调用
- 订单监控：30s/5s 再来一轮

IDLE 阶段已做了正确示范（line 821：先检查 `isWebSocketActive()`，活跃则跳过 API），但 HOLDING 和订单监控未跟进。

## 改动

文件：`api/src/services/strategy-scheduler.service.ts`

### 1. HOLDING 正股价格获取

先查 `quoteSubscriptionService.getPrice(effectiveSymbol)`，有值（>0）直接用，标记 `priceSource = 'ws-subscription'`。无值才降级到原有 API 多层 fallback（lastDone → mid → ask → bid）。

### 2. 订单监控行情获取

批量先查 `quoteSubscriptionService.getPriceMap()`，对每个待处理订单的 symbol 检查缓存。缓存命中的构造兼容对象 `{ symbol, lastDone }`，缓存缺失的收集到 `missingSymbols` 数组，只对缺失部分调 API。

### 3. 卖空持仓价格检查

同模式 1，先查缓存，miss 才降级 API。

### 4. costPrice 回退查询

同模式 1，先查缓存，miss 才降级 API。

## 不动的部分

- **期权 HOLDING 阶段**：走 `longportOptionQuoteService.getOptionPrice()`，专用服务有自己的缓存
- **IDLE 阶段**：已是正确模式
- **Iron Dome / 0DTE Watchdog**：独立安全机制，不修改

## 架构分析：间隔不需要调整

定时器间隔（期权 5s / 正股 60s / 订单监控 30s）控制的是决策频率，不是数据频率。优化后每个周期读到的是 WebSocket 最新推送值（毫秒级），数据新鲜度已从"上一周期 API 返回值"提升到"最近一次实时推送"。缩短间隔只增加系统负载，不改善决策质量。

前端监控大屏 30s 刷新间隔同样不需要调整——大屏是展示层，策略决策不经过前端。
