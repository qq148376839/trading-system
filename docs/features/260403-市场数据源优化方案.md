# 市场数据源优化方案（v2 — 三源竞速架构）

**日期**: 2026-04-03
**前置文档**: `docs/analysis/260403-双数据源冗余架构分析.md`、`docs/guides/260403-FutuAPI行情调用手册.md`
**目标**: 消除 Moomoo 爬虫单点故障，LongPort + FutuOpenD 竞速获取最低延迟，Moomoo 降级为终极兜底

---

## 1. 现状问题（第一性原理拆解）

### 1.1 Moomoo 爬虫是唯一的大盘数据通道

当前 `market-data.service.ts` 的所有大盘数据 **100% 依赖 Moomoo 爬虫**：

| 数据 | 调用链 | 故障模式 |
|------|--------|---------|
| SPX 日K | `getSPXCandlesticks()` → moomooProxy → get-kline | Cookie 过期 / 403 限频 / 反爬 |
| SPX 分时 | `getSPXHourlyCandlesticks()` → moomooProxy → get-quote-minute | 同上 |
| USD 日K+分时 | `getUSDIndexCandlesticks()` / Hourly → moomooProxy | 同上 |
| BTC 日K+分时 | `getBTCCandlesticks()` / Hourly → moomooProxy | 同上 |
| VIX | `getVIXCandlesticks()` → moomooProxy | 同上 |
| 标的 VWAP / 5min K | `getIntradayVWAP()` / `getIntraday5minKlines()` → moomooProxy | 同上 |

**三级降级（Vercel→CF→直连）只是同一个爬虫的冗余部署**，Moomoo 一旦封 Cookie 策略，三条路全断。

### 1.2 LongPort 验证结论

| Symbol | 支持状态 | 证据 |
|--------|:-------:|------|
| `.VIX.US` | **✅ 已生产运行** | `market-data.service.ts:516` 硬编码调用 |
| `.SPX.US` | **✅ 极大概率** | 同格式指数，原分析文档确认 |
| `SPY.US` | **✅ 已生产运行** | 回测配置 + 相关性分组已使用 |
| `UUP.US` / `IBIT.US` / `VXX.US` | **✅** | 标准美股 ETF，LongPort 明确支持 |

LongPort 频率限制：1 秒 ≤10 次调用，并发 ≤5。4 个 ETF 并发拉取在限制内。

### 1.3 FutuOpenD 现状

- **已部署**：NAS `futu-opend` 容器（host 网络），socat 转发 `0.0.0.0:11112`
- **可用地址**：`192.168.31.18:11112`（局域网 / bridge 网络容器）
- **协议**：TCP + Protobuf 自定义协议，**仅 Python futu SDK 可直连**
- **Node.js 不可直连** → 需要轻量 Python HTTP bridge（~80 行）

### 1.4 串行调用 + 800ms 延迟

当前 `getAllMarketData()` 为避免 Moomoo 限频，串行调用 + 每次间隔 800ms：

```
SPX日K → 800ms → USD日K → 800ms → BTC日K → VIX + 温度(并发)
→ 可选: SPX分时 → 800ms → USD分时 → 800ms → BTC分时 → 标的5min
```

一次完整调用 **耗时 3-6 秒**。竞速架构目标：降到 **<1s**。

## 2. 目标架构：三源分级

```
┌─────────────────────────────────────────────────────────┐
│                  MarketDataService (统一接口层)            │
│                                                          │
│  getAllMarketData() → getRacedData(type)                  │
│                         │                                │
│              Promise.any([longport, futu])                │
│              先到先用，两个都失败 → Moomoo 兜底             │
└──────────┬──────────────────────┬────────────┬───────────┘
           │                      │            │
     ┌─────▼──────┐        ┌─────▼──────┐  ┌──▼──────────┐
     │  LongPort   │        │ futu-bridge │  │   Moomoo    │
     │  (云端SDK)   │        │ (NAS本地)   │  │  (爬虫兜底)  │
     │  进程内调用   │        │ HTTP REST  │  │  串行+800ms  │
     └─────────────┘        └────────────┘  └─────────────┘
           │                      │
     quoteCtx.candlesticks()   fetch("http://192.168.31.18:8765/kline")
           │                      │
     LongPort 云端              futu-opend TCP:11112
```

### 2.1 三源角色定义

| 角色 | 数据源 | 延迟 | 稳定性 | 调用方式 |
|------|--------|------|--------|---------|
| **竞速源 A** | LongPort SDK | ~200ms (云端) | 高（偶发 token 过期）| 进程内 SDK 调用 |
| **竞速源 B** | FutuOpenD via bridge | <50ms (NAS 本地) | 极高（无 token 问题）| HTTP fetch 到 NAS |
| **兜底源** | Moomoo 爬虫 | 3-6s | 低（Cookie/反爬）| 现有 moomooProxy |

### 2.2 竞速逻辑

```typescript
async function getRacedKline(symbol: string, period: string, count: number): Promise<CandlestickData[]> {
  try {
    // LongPort + FutuOpenD 同时发，谁先成功用谁
    const result = await Promise.any([
      longportAdapter.getKline(symbol, period, count),
      futuBridgeAdapter.getKline(symbol, period, count),
    ]);
    return result;
  } catch (aggregateError) {
    // 两个都失败 → Moomoo 兜底
    logger.warn('LongPort + FutuOpenD 双源失败，降级 Moomoo');
    return moomooAdapter.getKline(symbol, period, count);
  }
}
```

### 2.3 数据源映射表

| 数据 | LongPort Symbol | FutuOpenD Symbol | Moomoo (兜底) |
|------|----------------|-----------------|---------------|
| SPX 日K | `.SPX.US` → Day | `US.SPY` → K_DAY | stockId=200003 |
| SPX 分时 | `SPY.US` → Min1 | `US.SPY` → K_1M | get-quote-minute |
| USD 日K+分时 | `UUP.US` → Day/Min1 | `US.UUP` → K_DAY/K_1M | stockId=72000025 |
| BTC 日K+分时 | `IBIT.US` → Day/Min1 | `US.IBIT` → K_DAY/K_1M | stockId=12000015 |
| VIX | `.VIX.US` → Day | `US.VXX` → K_DAY（方向） | Moomoo VIX |
| 标的 VWAP | `{symbol}` → Min1 自算 | `US.{ticker}` → K_1M 自算 | get-quote-minute |
| 标的 5min K | `{symbol}` → Min5 | `US.{ticker}` → K_5M | get-kline |

### 2.4 VIX 特殊处理

LongPort `.VIX.US` 已在生产运行，直接返回 VIX 绝对值 → **无需 VXX 换算**。
FutuOpenD 不支持指数，用 `US.VXX` 的**涨跌方向**作为辅助校验，不用于绝对值阈值。

## 3. futu-bridge 桥接服务设计

### 3.1 定位

FutuOpenD 使用 TCP + Protobuf 自定义协议，仅 Python futu SDK 可直连。
futu-bridge 是一个 **~100 行的轻量 HTTP 适配器**，将 Futu 协议翻译为标准 JSON REST。

### 3.2 核心接口

```
GET /kline?symbol=US.SPY&ktype=K_DAY&count=100
GET /kline?symbol=US.SPY&ktype=K_1M&count=240
GET /snapshot?symbols=US.SPY,US.UUP,US.IBIT,US.VXX
GET /health
```

### 3.3 返回格式（已归一化）

```json
{
  "source": "futu",
  "data": [
    {
      "timestamp": 1775214660000,
      "open": 646.42,
      "high": 647.47,
      "low": 646.38,
      "close": 646.86,
      "volume": 710788,
      "turnover": 459768234
    }
  ]
}
```

**字段已统一为 `CandlestickData` 格式**，Node.js 端无需再转换。

### 3.4 部署方式

```yaml
# docker-compose.yml 新增
futu-bridge:
  build: ./futu-bridge
  container_name: futu-bridge
  restart: unless-stopped
  environment:
    - FUTU_HOST=192.168.31.18
    - FUTU_PORT=11112
  ports:
    - "8765:8765"
  networks:
    - trading-net
```

trading-app 通过 `http://futu-bridge:8765/kline` 调用（Docker 内部网络）。
NAS 部署时，futu-bridge 直连 `192.168.31.18:11112`。

### 3.5 Symbol 格式转换

| 标准格式 | LongPort | FutuOpenD |
|---------|----------|-----------|
| SPY | `SPY.US` | `US.SPY` |
| UUP | `UUP.US` | `US.UUP` |
| IBIT | `IBIT.US` | `US.IBIT` |
| VXX | `VXX.US` | `US.VXX` |

转换在 bridge 内部完成，Node.js 端统一使用 LongPort 格式（`TICKER.US`）。

## 4. 实施分阶段方案

### Phase 1：futu-bridge 构建 + 基础竞速（可独立部署验证）

**目标**：构建 futu-bridge，实现 `getAllMarketData()` 的 LongPort + FutuOpenD 竞速。

**新增文件**：
```
futu-bridge/
├─ main.py            (~100行 FastAPI)
├─ requirements.txt   (futu-api, fastapi, uvicorn)
└─ Dockerfile         (~15行)
```

**改动文件**：`docker-compose.yml`, `market-data.service.ts`

**核心变更**：

```
market-data.service.ts:
├─ 新增 FutuBridgeAdapter 类
│   → fetch("http://futu-bridge:8765/kline?symbol=US.SPY&ktype=K_DAY&count=100")
│   → 解析 JSON → CandlestickData[]
│
├─ 新增 LongportMarketAdapter 类
│   → getQuoteContext() → ctx.candlesticks(".SPX.US", Period.Day, count)
│   → formatLongbridgeCandlestick()
│
├─ 新增 getRacedKline(symbol, period, count)
│   → Promise.any([longport, futu]) → catch → Moomoo fallback
│
└─ 修改 getAllMarketData():
    ├─ Promise.all([
    │     getRacedKline("SPX", "Day", 100),
    │     getRacedKline("UUP", "Day", 100),
    │     getRacedKline("IBIT", "Day", 100),
    │     getRacedKline("VIX", "Day", 100),
    │   ])
    │   → 全部并发，总耗时 ≈ max(LongPort, FutuOpenD) ≈ 200ms-1s
    │
    └─ includeIntraday 时追加:
        getRacedKline("SPY", "Min1", 240)  // SPX 分时
        getRacedKline("UUP", "Min1", 240)  // USD 分时
        getRacedKline("IBIT", "Min1", 240) // BTC 分时
```

**预期收益**：
- 延迟: 3-6s → <1s（竞速 + 并发）
- 稳定性: 双源竞速，任一源故障无感知
- Moomoo 完全故障: 系统仍正常运行

### Phase 2：WebSocket 订阅扩展到大盘 ETF

**目标**：SPY/UUP/IBIT/VXX 的实时数据通过 LongPort WebSocket 推送。

**改动文件**: `quote-subscription.service.ts`, `market-data.service.ts`

```
quote-subscription.service.ts:

```

**预期收益**：大盘数据从"周期拉取"升级到"实时推送"，评分延迟进一步降低。

---

### Phase 3：标的 VWAP/5min K 线切换（消除 Moomoo 最后依赖）

**目标**：`getIntradayVWAP()` 和 `getIntraday5minKlines()` 也走竞速通道。

**改动文件**: `market-data.service.ts`

```
├─ getIntradayVWAP() → getRacedKline(symbol, "Min1", 390) → 本地算 VWAP
└─ getIntraday5minKlines() → getRacedKline(symbol, "Min5", 78)
```

**Phase 3 完成后**：Moomoo 爬虫降级为纯兜底，日常运行零依赖。

## 5. 风险与缓解

| 风险 | 概率 | 缓解 |
|------|------|------|
| LongPort `.SPX.US` 不支持 K 线 | 中 | 用 `SPY.US` 替代（相关性 0.9998） |
| futu-bridge 容器挂掉 | 低 | `restart: unless-stopped` + LongPort 独立运行 |
| LongPort token 过期 | 低 | DB 自动刷新 + FutuOpenD 独立竞速 + Moomoo 兜底 |
| FutuOpenD 连接 NAS 断开 | 极低 | 本地部署，无外部依赖 |
| ETF 与指数微小偏差 | 极低 | SPY/SPX 相关性 0.9998 |
| LongPort K 线不含 VWAP | 确定 | Phase 3 本地计算（标准公式） |
| LongPort 频率限制（1s/10次） | 低 | 4 个 ETF 并发 < 5 并发上限 |

## 7. 与前版方案对比

| 维度 | v1（LongPort 单主） | v2（三源竞速） |
|------|-------------------|--------------|
| 数据源 | LongPort 主 + Moomoo 兜底 | LongPort + FutuOpenD 竞速 + Moomoo 兜底 |
| 延迟 | ~200ms-1s (LongPort) | min(LongPort, FutuOpenD) ≈ <200ms |
| 新增基础设施 | 无 | futu-bridge (~100行 Python + Docker) |
| 容错 | LongPort 挂 → Moomoo | LongPort 挂 → FutuOpenD 继续 → 两个都挂 → Moomoo |
| FutuOpenD 独有能力 | 不可用 | 可扩展：资金流向、条件选股 |
| 实施复杂度 | 低 | 中（多一个轻量容器） |
