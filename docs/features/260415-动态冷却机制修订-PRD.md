# 动态冷却机制修订

**日期**: 2026-04-15
**模式**: 迭代-重（重构冷却核心函数 + 新增退出原因追踪 + 价格因子引入）
**状态**: 已确认

## 业务原点 (First Principles)

- **底层物理现实**: 当前冷却系统基于固定时间，但实盘数据证明"时间够了"不等于"市场准备好了"。4/6-4/15共37笔交易，2连亏后继续入场的9笔平均PnL -9.3%，总亏 -$883；跳过这9笔则总PnL从+$524变+$1,407（+168%）。冷却机制的物理本质是"等市场条件恢复到有利状态"，不是"等够了时间"。
- **被推翻的假设**:
  1. ~~冷却越长越安全~~ — 数据反驳：>15min重入9笔avg -6.0%，而首笔(无冷却)12笔avg +9.9%。长冷却让你错过新鲜信号，进入陈旧信号。
  2. ~~同一层冷却对所有退出原因一视同仁~~ — TAKE_PROFIT后重入7笔avg -6.5%，TRAILING_STOP后重入9笔avg +3.3%。退出原因是强信号，当前系统完全忽略。
  3. ~~组级readiness模型足够~~ — 模型缺少价格变化因子和连亏因子，5因子中方向变化权重0.35但实盘显示方向变化不是最关键变量。
- **不做的后果**: 每天平均0.9笔落入"2连亏后冲动重入"，按实盘avg -$98/笔计算，每月约 -$1,764 的可避免亏损。

## 核心原子功能

| 原子 | 业务价值 | 验收标准（可执行） |
|------|---------|-----------------|
| A1: 2连亏硬门控 | 消灭最大失血口(-$883/10天) | `consecutiveLosses >= 2` 时冷却拦截日志出现 `CONSEC_LOSS_GATE`，且不入场 |
| A2: 退出原因感知冷却 | TAKE_PROFIT后加罚，TRAILING_STOP后减罚 | `GroupExitRecord.exitReason` 写入正确值，`computeCooldownUntil` 根据退出原因调整基础分钟数 |
| A3: 价格变化因子加入readiness | readiness回答"市场变了吗"而非"时间够了吗" | `calculateReentryReadiness` 新增 f6 因子，权重分配后总和=1.0 |
| A4: 跨标的冲动门控 | 防止04/13 TSLA→QQQ 0.6min情况 | 任意标的退出后全局2min硬底线（不限同组） |

## 资金与风控影响

- **资金流变化**: 无。不改资金分配、仓位大小、订单金额。只改"是否允许入场"的门控。
- **风控规则变化**:
  - 现有 circuitBreaker（4连亏 / 30%日亏熔断至收盘）不动，那是最后防线
  - 新增 A1 是前置门控：2连亏后当天剩余时间该标的禁入，次日自动解除（首笔0冷却规则不变）
  - A1 和 circuitBreaker 的关系：A1 是标的级（2连亏），circuitBreaker 是策略级（4连亏）。A1 先拦，拦不住才到 circuitBreaker
- **失败处理**: 所有新门控采用"fail-open"原则 — 数据异常时放行而非阻止（宁可多入一笔，不可系统僵死）

## 业务逻辑与规则

### 正常路径

#### 改造1: `GroupExitRecord` 新增 `exitReason` 字段

```typescript
interface GroupExitRecord {
  // ...现有字段保持不变
  exitReason: string | null;  // 新增: STOP_LOSS | TAKE_PROFIT | TRAILING_STOP | TIME_STOP | EMERGENCY_* | FORCED_CLOSE_* | BROKER_* | PROTECTION_MIT_*
}
```

`recordSymbolExit` 新增可选参数 `exitReason?: string`，所有调用点传入对应退出原因。

#### 改造2: `computeCooldownUntil` — 退出原因感知

**当前逻辑**: 0DTE非首笔固定1min，非0DTE固定10min。

**新逻辑** (0DTE路径):

| 退出原因 | 基础冷却 | 理由（数据支撑） |
|---------|---------|---------------|
| 首笔(dailyTradeCount=0) | 0min（不变） | 首笔12笔avg +9.9%，不拦 |
| TRAILING_STOP | max(1, configValue) min（不变） | 后续9笔avg +3.3%，追踪止损=趋势走完，正常冷却 |
| STOP_LOSS | max(1, configValue) * 1.5 min | 后续9笔avg -2.8%，止损=判断错误，多给50%冷却 |
| TAKE_PROFIT | max(1, configValue) * 2.0 min | 后续7笔avg -6.5%，止盈=趋势结束主动离场，最需冷却 |
| EMERGENCY_* / FORCED_CLOSE_* | max(1, configValue) * 2.0 min | 异常退出=市场混乱，多给时间 |
| 其他/未知 | max(1, configValue) min | 保守默认 |

非0DTE路径：同样乘以上述系数，基数改为 `configMinutes ?? 10`。

公式：`cooldownMinutes = baseCooldown * exitReasonMultiplier`

```typescript
function getExitReasonMultiplier(exitReason: string | null): number {
  switch (exitReason) {
    case 'TRAILING_STOP': return 1.0;
    case 'STOP_LOSS': return 1.5;
    case 'TAKE_PROFIT': return 2.0;
    case 'EMERGENCY_STOP_LOSS':
    case 'EMERGENCY_CLOSE':
    case 'FORCED_CLOSE_BEFORE_MARKET_CLOSE': return 2.0;
    default: return 1.0;
  }
}
```

#### 改造3: 2连亏硬门控 (A1)

**位置**: `processIdleState` 的冷却检查段（L2318-2326），在现有 `cooldownUntil` 检查之后。

**逻辑**:
```
if (context.consecutiveLosses >= 2) {
  // 硬门控：2连亏后该标的当天禁入
  // 次日 POSITION_CONTEXT_RESET 会重置 consecutiveLosses=0
  log CONSEC_LOSS_GATE
  return (不入场)
}
```

**为何是2不是3**: 实盘数据 — 2连亏后9笔avg -9.3%（-$883），3连亏后次日首笔6笔avg +25.4%（+$1,442）。2是分水岭。

**解除条件**: 次日交易日开始时 `POSITION_CONTEXT_RESET` 已有 `consecutiveLosses: 0` 重置逻辑（L646, L2296, L2739），无需改动。

**与circuitBreaker的协同**: consecutiveLosses 在每次止损退出时+1。2连亏 → A1门控该标的 → 如果其他标的也连亏到4 → circuitBreaker策略级熔断。两层不冲突，A1是更早的标的级拦截。

#### 改造4: `calculateReentryReadiness` — 新增价格变化因子 + 权重重分配

**当前5因子**:
- f1 方向变化 0.35
- f2 分数变化 0.25
- f3 时间衰减 0.20
- f4 盈亏x方向 0.12
- f5 连续同向 0.08

**问题**: 没有价格因子。退出后价格纹丝不动也能因为时间到了放行。实盘 >15min桶9笔avg -6.0% 就是这个问题 — 时间到了但市场没变。

**新增 f6 价格变化因子**:

```typescript
// f6: 退出后标的价格变化百分比（通过 fastMomentumService 获取当前价）
// 价格变化越大 → 市场已经"重新定价"→ 旧信号失效 → 新信号可能有效
const currentPrice = fastMomentumService.getLatestPrice(symbol);  // 需新增方法
const exitPrice = exitRecord.exitPrice;  // 需新增字段
if (currentPrice && exitPrice && exitPrice > 0) {
  const priceChangePct = Math.abs((currentPrice - exitPrice) / exitPrice) * 100;
  f6 = Math.min(1.0, priceChangePct / 0.5);  // 0.5%变化=满分
} else {
  f6 = 0.5;  // 数据缺失时中性值（fail-open）
}
```

**新权重分配**:
| 因子 | 旧权重 | 新权重 | 调整理由 |
|------|-------|-------|---------|
| f1 方向变化 | 0.35 | 0.25 | 降低：方向变化重要但被高估，实盘显示同向追踪止损后重入表现不差 |
| f2 分数变化 | 0.25 | 0.20 | 微降 |
| f3 时间衰减 | 0.20 | 0.10 | 大幅降低：时间是最不可靠的信号，数据已证明 |
| f4 盈亏x方向 | 0.12 | 0.15 | 微升：退出盈亏是强信号 |
| f5 连续同向 | 0.08 | 0.05 | 微降 |
| f6 价格变化 | - | 0.25 | 新增：直接回答"市场变了吗" |

校验：0.25 + 0.20 + 0.10 + 0.15 + 0.05 + 0.25 = 1.00

`baseCooldownMinutes` 从硬编码8改为策略配置项 `reentryReadinessBaseCooldownMinutes`，默认值8不变。

#### 改造5: 跨标的冲动门控 (A4)

**位置**: `processIdleState` 冷却检查段，在 cooldownUntil 检查之前。

**逻辑**:
```typescript
// 全局冲动门控：任意标的退出后2min内，所有标的禁入
const crossState = this.getCrossSymbolState(strategyId);
for (const [, exitRec] of crossState.recentExits) {
  if (now - exitRec.exitTime < 2 * 60_000) {
    log GLOBAL_IMPULSE_GATE
    return (不入场)
  }
}
```

**为何2分钟**: `calculateReentryReadiness` 已有同组2min硬底线。A4 把这个逻辑提升到跨组级别。04/13 TSLA→QQQ 0.6min跨组入场就是当前逻辑的漏洞（不同组不受2min限制）。

**与现有2min硬底线的关系**: 现有的在 `calculateReentryReadiness` 内部（同组）。A4 在 `processIdleState` 入口处（全组）。前者可以保留作为冗余，或在A4生效后去掉（减少重复检查）。建议保留，defense in depth。

### 异常/边界

| 场景 | 处理 |
|------|------|
| exitReason 为 null（旧数据/兼容） | `getExitReasonMultiplier` 返回 1.0，等同当前行为 |
| fastMomentumService 无当前价格 | f6 = 0.5（中性值），不阻止也不加速 |
| consecutiveLosses 字段不存在 | `parseInt(String(context.consecutiveLosses ?? 0), 10)` 已有兜底，返回0 |
| 2连亏门控 + circuitBreaker 同时触发 | 两者独立，A1先拦（标的级），circuitBreaker后拦（策略级），不冲突 |
| 开盘首笔 dailyTradeCount=0 | A1不触发（consecutiveLosses已被日重置为0），computeCooldownUntil 返回 null（0冷却），与当前行为一致 |

## 失败模式清单

| # | 场景 | 系统行为 | 验证方式 |
|---|------|---------|---------|
| 1 | 2连亏门控过于激进，拦掉了本可盈利的第3笔 | 接受。实盘数据：2连亏后9笔33%胜率avg -9.3%，即使拦掉2/9的盈利笔，净期望仍为正。次日首笔自动解除。 | 对比启用前后10个交易日的"2连亏后入场笔数"和总PnL |
| 2 | exitReason传递遗漏（某退出路径未传入exitReason） | `getExitReasonMultiplier(null)` 返回 1.0，等同当前行为，不会比现在更差 | grep所有 `recordSymbolExit` 调用点，确认每个都传入了 exitReason |
| 3 | fastMomentumService.getLatestPrice 返回null（WebSocket断连/数据延迟） | f6 = 0.5（中性值），readiness计算不受极端影响，等同"不知道价格变没变"的保守态度 | 断开WebSocket后观察日志中f6值是否为0.5 |
| 4 | 跨标的2min门控在高波动日拦掉快速轮动机会 | 接受。2分钟是极短窗口，04/13案例证明<2min入场3笔avg -13.3%。如果2min后市场仍有机会，信号会重新生成。 | 统计被 GLOBAL_IMPULSE_GATE 拦截的次数和后续市场走势 |
| 5 | 新权重分配导致readiness阈值0.50的实际含义改变 | 有可能。f3从0.20降到0.10意味着纯靠时间很难过0.50，必须有方向变化或价格变化。这是预期行为。如果过度拦截，调低阈值到0.40。 | 对比启用前后readiness分布直方图和入场频率 |

## 回滚方案

每个改造独立控制，可单独回滚：

1. **A1 2连亏门控**: 注释掉 processIdleState 中的 `consecutiveLosses >= 2` 检查（约5行代码），不影响其他逻辑
2. **A2 退出原因感知**: `getExitReasonMultiplier` 全部返回1.0，等同当前行为
3. **A3 价格因子**: f6 权重设为0，其他5因子按比例放大回原权重
4. **A4 跨标的门控**: 注释掉 processIdleState 中的 `GLOBAL_IMPULSE_GATE` 检查（约7行代码）

全部回滚 = 恢复到当前行为。每一步都是安全的，不涉及数据库变更，不涉及资金逻辑。
