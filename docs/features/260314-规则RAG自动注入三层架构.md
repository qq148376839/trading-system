# 规则 RAG 自动注入 — 三层架构

> 创建时间：2026-03-14
> 状态：已实施，待 RAG 重新索引

---

## 问题

CLAUDE.md 膨胀到 155 行，15 条规则只有一行摘要无法触发行动，写详解又突破可维护上限。结果：规则形同虚设。例证：R5v2 Phase B 日志从未按规则 #15 降级。

根因：规则的"存在"和"被执行"之间缺一个主动推送机制——靠 agent 自觉读 CLAUDE.md 中的一行摘要来决定是否遵守规则，不可靠。

## 方案设计

三层架构，各层职责分明：

```
Layer 3: UserPromptSubmit Hook (主动注入)
  ↓ 每次用户提问自动调 RAG → 匹配规则 → 注入上下文
Layer 2: Agent 内嵌规则 (角色级)
  ↓ developer/reviewer/debugger 各带自己必查的规则门控
Layer 1: CLAUDE.md 精简 (导航层)
  ↓ ~97 行，只保留项目概览 + 规则索引表 + 指向详情的链接
```

**数据流**：用户提问 → hook 读 stdin `{ prompt }` → RAG 查询（3s 超时）→ score > 0.40 且来源为错误规则集 → 格式化为 `additionalContext` 注入。RAG 不可用时 fallback 到本地 15 条规则的关键词映射。

## 关键设计决策

### 1. Hook 用 `.mjs` + Node.js 而非 bash

bash 拼 JSON 有转义地狱，Node.js 原生支持 fetch + JSON，且项目已依赖 Node.js 环境。

### 2. RAG 优先 + 本地两层 fallback

RAG 能语义匹配，但依赖网络且需 `错误规则集.md` 被索引。本地 fallback 采用两层关键词匹配：
- **exact**：高精度词（如 `jsonb`、`0dte`），单独命中即触发
- **context**：歧义词（如 `日志`、`删除`、`退出`），需与上下文词共现才触发

这解决了旧版单关键词匹配的误触发问题（如"信号日志"误触发日志降级规则 #15）。

### 3. MIN_PROMPT_LENGTH = 6

短指令（如 `继续`、`好的`）不需要规则检查，跳过避免噪音。阈值从 10 降至 6，因为中文字符在 JS `.length` 中每字占 1，9 个中文字就是有意义的完整句子。

### 4. CLAUDE.md 只保留索引表

规则详情的 single source of truth 是 `docs/guides/260307-错误规则集.md`，CLAUDE.md 只做导航。通过 hook 主动推送替代被动阅读。
