> 日期: 2026-04-12

> 优先级: P0（启用正股策略的前提）

---

## 原始问题

系统存在**零跨策略感知**：
- `validateSellPosition()` 用券商聚合持仓验证，不区分策略（`strategyId` 参数从未使用）
- `requestAllocation()` 只检查本策略资金
- `scoringAuction()` 只在单策略内竞价
- 持仓缓存全局共享（30s 脏读窗口）
- 现有 `CROSS_STRATEGY_SYNC` 是事后补救，不是事前预防

---

## 第一性原理重审：5-Phase Coordinator 是过度工程

### 被挑战的假设

原方案假设"N 个策略可能交易同一标的"，因此需要通用协调层。但审计实际场景：

**期权 vs 正股天然不冲突**：

| 维度 | 期权 vs 正股 | 结论 |
|------|-------------|------|
| 券商持仓 | 完全不同的 position entries（合约 vs 股票） | **不冲突** |
| 卖出验证 | `validateSellPosition` 按 symbol 匹配，期权合约名 ≠ 股票代码 | **不冲突** |
| 资金 | 已有 `SELECT FOR UPDATE` per-strategy 隔离 | **已解决** |
| 状态机 | `UNIQUE(strategy_id, symbol)` — 不同 symbol 不干涉 | **不冲突** |
| 订单通道 | 共享但无影响（不同品种不同接口方法） | **不冲突** |
| 缓存 | 正股 60s 周期 > 30s TTL，且不同 symbol 不互相影响 | **无实际风险** |

**结论**：期权和正股是不同品种，在券商层面就是完全独立的持仓、独立的 symbol、独立的订单。5-Phase 方案解决的"同 symbol 跨策略冲突"在期权+正股场景中不存在。

### 真正的冲突源：两个期权策略之间

OPTION_INTRADAY_V1 和 OPTION_SCHWARTZ_V1 都跑 5s 周期、都交易期权合约 — 这才是冲突可能发生的地方。但解法不是建协调层，而是**砍到只剩一个期权策略**。

## 方案 B: 架构简化 — 1 期权 + 1 正股 [采纳]

### 核心思路

不建协调层，而是从根源消除冲突：

1. **砍掉 Schwartz 策略** — 期权策略从 2 个降为 1 个（OPTION_INTRADAY_V1）
2. **Schwartz 4 个独有过滤器合入 Intraday** — 作为可配置开关（默认关闭）
3. **1 期权 + 1 正股** — 不同品种天然不冲突，无需协调层

### 为什么这是更优解

- **零新增服务** — 不需要 CrossStrategyCoordinator（省 250 行新代码 + 4 文件改动）
- **零冲突面** — 期权合约和股票在券商层就是完全独立的持仓
- **保护已验证数据** — 生产 Intraday 行为零变化，Schwartz 过滤器默认关闭
- **可渐进验证** — 测试环境开启 Schwartz 过滤器，对比后再决定生产启用

### Schwartz vs Intraday 对比

两者共享完全相同的信号源（`optionRecommendationService.calculateOptionRecommendation()`），区别只在过滤层：

| 能力 | Schwartz 独有 | Intraday 独有 | 共享 |
|------|-------------|-------------|------|
| EMA 硬过滤（逆趋势一刀切） | ✓ | | |
| CHOP 震荡检测（阈值 2x） | ✓ | | |
| IV Rank 过滤（高 IV 拒买方） | ✓ | | |
| 大赚后仓位缩减 | ✓ | | |
| 智能反向（regime detection） | | ✓ | |
| 加速度 bonus（2 阶导提前入场） | | ✓ | |
| 多策略类型（跨式/价差/铁鹰） | | ✓ | |
| 资金预检缓存 | | ✓ | |
| FastMo gate | | | ✓ |
| 开盘冲量守卫 | | | ✓ |
| VIX 动态阈值 | | | ✓ |
| 0DTE 尾盘衰减 | | | ✓ |

**结论**：Schwartz 是 Intraday 的子集 + 4 个额外过滤器。合并而非并行。
