# Phase 4: 入场时机微调 — 动量加速度检测

**日期**: 2026-04-03
**状态**: 待实施
**前置**: Phase 0-3 已部署（数据验证 + PnL修复 + WebSocket订阅 + K线增强）

## 一、问题定义

### 数据事实（Phase 0 验证结果，166笔交易）

| 提前时间 | 平均PnL | 改善率 | 解读 |
|----------|---------|--------|------|
| T-5min | -$12.96 | 58% | 方向未确立，入场即赌博 |
| **T-3min** | **+$14.31** | **71%** | **方向已确立，动量在构建，最优窗口** |
| T-1min | +$5.20 | 64% | 接近峰值，仍有改善空间 |
| T-0 (实际) | -$9.54 | — | 动量已达峰值，入场即均值回归 |

### 根因（第一性原理分解）

信号延迟 = 数据延迟 + K线收线延迟 + **分数累积延迟** + 调度延迟

| 延迟源 | 量级 | Phase 2 后 | 是否主因 |
|--------|------|-----------|---------|
| 数据采集（轮询→WebSocket） | ~~0-5s~~ ~200ms | ✅ 已解决 | 否 |
| K线收线（1min bar 关闭） | 0-60s | 不变 | 次要 |
| **分数累积（连续同向 bar 才过阈值）** | **2-4min** | **不变** | **主因** |
| 调度间隔（timer tick） | ~15s | 不变 | 否 |

**核心矛盾**：评分系统要求 K线动量累积到 `directionalScoreMin`（CONSERVATIVE=12）才触发。累积需要多根同向 1min bar。但在 T-3 时刻，方向信息量已足够（71%准确率），这 3 分钟的等待只带来更差的价格，没有带来方向判断的提升。

### 为什么选方案 C（加速度检测）

| 方案 | 解决什么 | 缺陷 |
|------|---------|------|
| A: FastMo 升级为信号源 | 调度延迟 ~15s | 主因是累积延迟 2-4min，不是调度延迟 |
| B: 降低 directionalScoreMin | 累积延迟 | 全局降低，不区分"加速构建"和"震荡噪声" |
| **C: 加速度检测 bonus** | **累积延迟** | **自校准：只在动量构建期加分，峰值时自动消失** |

## 二、设计方案

### 2.1 物理模型

```
动量 = 价格变化速度（一阶导）
加速度 = 动量变化速度（二阶导）

T-5min: 动量≈0, 加速度≈0     → 方向未确立
T-3min: 动量>0, 加速度>0(峰值) → 构建阶段 ← 最优入场
T-0min: 动量>0(峰值), 加速度≈0 → 平台期
T+2min: 动量>0(衰减), 加速度<0 → 衰退阶段
```

加速度 bonus 的数学性质：
- 只在动量增长期（加速度>0）生效
- 动量达峰时加速度自然归零，bonus 消失
- 震荡市动量忽正忽负，加速度均值≈0，不产生持续 bonus

### 2.2 数据源选择：FastMo 短 buffer（60s, 120点）

为什么不用 K线计算加速度：
- K线是 1min 粒度，计算 5-bar 加速度 = 5分钟窗口 → 加速度本身就迟到 5 分钟
- FastMo 短 buffer 有 120 个实时价格点（WebSocket ~200ms/点），60s 窗口，亚秒粒度

FastMo 已有的加速度信息：
```typescript
// checkGate() 中已计算:
const mid = Math.floor(n / 2);
const firstReg = linearRegression(prices.slice(0, mid));   // 前30s
const secondReg = linearRegression(prices.slice(mid));      // 后30s
const decelRatio = |secondReg.slope| / |firstReg.slope|;
// decelRatio > 1.0 = 加速中, < 1.0 = 减速中
```

### 2.3 注入层选择：策略层（非评分层）

```
当前架构:
  recommendation.calculateOptionRecommendation() → finalScore
  strategy.evaluateDirectionalBuyer(finalScore vs threshold) → pass/fail
  strategy.fastMo.checkGate() → direction/decel filter

改造后:
  recommendation.calculateOptionRecommendation() → baseFinalScore
  ★ strategy.fastMo.getAccelerationInfo(symbol) → accelBonus
  ★ adjustedScore = baseFinalScore + accelBonus
  strategy.evaluateDirectionalBuyer(adjustedScore vs threshold) → pass/fail
  strategy.fastMo.checkGate() → direction/decel filter（不变）
```

**为什么不在 recommendation 层注入**：
1. recommendation 是纯 K线评分服务，不依赖 FastMo — 保持解耦
2. 加速度 bonus 是实时微调，属于策略决策层的职责
3. 策略层已同时使用 recommendation 和 FastMo，是自然的整合点

### 2.4 参数配置化 — 前端可调

核心可调参数从硬编码常量改为策略配置项，存储在 `strategies.config` JSONB 中，通过前端 UI 调整。

**可调参数（前端配置）**：

| 参数 | 配置键 | 默认值 | 含义 | 调参场景 |
|------|--------|--------|------|---------|
| 总开关 | `enabled` | `true` | 启用/禁用加速度 bonus | 紧急关闭、A/B对比 |
| 加速阈值 | `accelRatioThreshold` | `1.15` | decelRatio 超过此值才算加速 | 调灵敏度：低→更易触发，高→更保守 |
| Bonus 系数 | `bonusMultiplier` | `5.0` | magnitude → bonus 映射倍数 | 调 bonus 强度 |
| Bonus 上限 | `bonusCap` | `8.0` | finalScore 最大 bonus | 安全阀：限制单靠加速度的贡献 |

**不可调参数（硬编码，内部机制）**：

| 参数 | 值 | 原因 |
|------|-----|------|
| `ACCEL_MIN_DATA_POINTS` | 20 | 数据质量门槛，不应降低 |
| `ACCEL_MIN_SLOPE` | 0.0005 | 噪声地板，改了可能导致虚假方向 |
| staleness 阈值 | 10s | 基础设施参数，跟 WebSocket 实现绑定 |

### 3.2 `fast-momentum.service.ts` — 新增 `getAccelerationInfo()`

**新增接口**:
```typescript
export interface AccelerationInfo {
  /** 是否处于有效加速状态 */
  isAccelerating: boolean;
  /** 加速度幅度: max(0, decelRatio - 1.0), capped at 3.0 */
  magnitude: number;
  /** 动量方向 (基于全段 slope) */
  direction: 'CALL' | 'PUT' | null;
  /** 全段 slope（用于日志） */
  slope: number | null;
  /** decelRatio 原始值（用于日志） */
  decelRatio: number | null;
  /** 数据是否可靠（WebSocket 活跃 + 数据充足） */
  reliable: boolean;
  /** 数据点数 */
  dataPoints: number;
}
```

**新增方法**（接受外部传入的 accelRatioThreshold）:
```typescript
getAccelerationInfo(symbol: string, accelRatioThreshold: number = 1.15): AccelerationInfo {
  const buf = this.buffers.get(symbol);

  // 数据不足 → 不可靠
  if (!buf || buf.size() < ACCEL_MIN_DATA_POINTS) {
    return { isAccelerating: false, magnitude: 0, direction: null,
             slope: null, decelRatio: null, reliable: false,
             dataPoints: buf?.size() ?? 0 };
  }

  const samples = buf.getAll();
  const prices = samples.map(s => s.price);
  const n = prices.length;

  // 数据新鲜度检查: 最新数据不超过 10 秒
  const latestTs = samples[samples.length - 1].timestamp;
  if (Date.now() - latestTs > 10_000) {
    return { isAccelerating: false, magnitude: 0, direction: null,
             slope: null, decelRatio: null, reliable: false, dataPoints: n };
  }

  // 全段 slope → 方向
  const fullReg = linearRegression(prices);
  const direction: 'CALL' | 'PUT' | null =
    fullReg.slope > ACCEL_MIN_SLOPE ? 'CALL' :
    fullReg.slope < -ACCEL_MIN_SLOPE ? 'PUT' : null;

  if (!direction) {
    return { isAccelerating: false, magnitude: 0, direction: null,
             slope: fullReg.slope, decelRatio: null, reliable: true, dataPoints: n };
  }

  // 前后半段 slope 比 → decelRatio
  const mid = Math.floor(n / 2);
  const firstReg = linearRegression(prices.slice(0, mid));
  const secondReg = linearRegression(prices.slice(mid));

  if (Math.abs(firstReg.slope) < NEAR_ZERO_SLOPE) {
    // 前半段无趋势，后半段有 → 启动加速（特殊处理）
    const startupMagnitude = Math.min(Math.abs(secondReg.slope) / ACCEL_MIN_SLOPE, 3.0);
    return {
      isAccelerating: startupMagnitude > 0.5,
      magnitude: Math.max(0, startupMagnitude - 0.5),
      direction, slope: fullReg.slope,
      decelRatio: null, reliable: true, dataPoints: n,
    };
  }

  const decelRatio = Math.abs(secondReg.slope) / Math.abs(firstReg.slope);

  // 方向一致性: 前后半段 slope 必须同向
  const sameDirection = (firstReg.slope > 0 && secondReg.slope > 0) ||
                        (firstReg.slope < 0 && secondReg.slope < 0);
  if (!sameDirection) {
    return { isAccelerating: false, magnitude: 0, direction,
             slope: fullReg.slope, decelRatio, reliable: true, dataPoints: n };
  }

  // 加速度判定（使用外部传入的阈值）
  const isAccelerating = decelRatio > accelRatioThreshold;
  const magnitude = isAccelerating ? Math.min(decelRatio - 1.0, 3.0) : 0;

  return { isAccelerating, magnitude, direction,
           slope: fullReg.slope, decelRatio, reliable: true, dataPoints: n };
}
```

**新增常量**（内部机制，不可配置）:
```typescript

```

### 3.3 `option-intraday-strategy.ts` — generateSignal() 加速度 bonus 注入

**插入位置**: Step 2（获取推荐）和 Step 4（策略评估）之间

```typescript
// --- 2.5) 动量加速度 bonus — 在构建阶段提前入场 ---
let accelBonus = 0;
const accelCfg = this.cfg.accelerationBonus;
const accelEnabled = accelCfg?.enabled ?? true;

if (accelEnabled) {
  const accelThreshold = accelCfg?.accelRatioThreshold ?? 1.15;
  const accelMultiplier = accelCfg?.bonusMultiplier ?? 5.0;
  const accelCap = accelCfg?.bonusCap ?? 8.0;

  const accelInfo = fastMomentumService.getAccelerationInfo(symbol, accelThreshold);

  if (accelInfo.reliable && accelInfo.isAccelerating && accelInfo.direction) {
    const rawBonus = accelInfo.magnitude * accelMultiplier;

    accelBonus = accelInfo.direction === 'CALL' ? cappedBonus : -cappedBonus;

      { module: 'OptionStrategy.AccelBonus', strategyId: this.strategyId }

  // 记录到 logData（无论是否触发）

    config: { accelThreshold, accelMultiplier, accelCap },
```

## 四、典型场景推演

**场景 1: 构建阶段（T-3, 最优窗口）**
```
FastMo: slope=+0.003, decelRatio=2.0 (后半段比前半段快2倍)
→ magnitude = 2.0 - 1.0 = 1.0
→ rawBonus = 1.0 × 5.0(bonusMultiplier) = 5.0
→ accelBonus = +5.0 (CALL 方向)

K线 baseFinalScore = 7 (低于阈值 12)
adjustedFinalScore = 7 + 5 = 12 → 刚好触发 ✅
提前 ~3 分钟入场
```

**场景 2: 峰值阶段（T-0, 当前实际入场点）**
```
FastMo: slope=+0.005, decelRatio=0.8 (减速中)
→ decelRatio < 1.15(accelRatioThreshold) → isAccelerating = false
→ accelBonus = 0

K线 baseFinalScore = 15 (已过阈值)
adjustedFinalScore = 15 + 0 = 15 → 正常触发
(即使没有 bonus 也会触发, bonus 不影响)
```

**场景 3: 震荡市（无方向）**
```
FastMo: slope=+0.0001 (接近零), decelRatio=1.8
→ slope < ACCEL_MIN_SLOPE(0.0005) → direction = null
→ accelBonus = 0

K线 baseFinalScore = 3
adjustedFinalScore = 3 + 0 = 3 → 不触发 ✅
```

**场景 4: 太早（T-5）**
```
FastMo: slope=+0.001, decelRatio=1.3
→ magnitude = 0.3
→ rawBonus = 0.3 × 5.0 = 1.5
→ accelBonus = +1.5

K线 baseFinalScore = 2
adjustedFinalScore = 2 + 1.5 = 3.5 → 远不够阈值 12 ✅
```

**场景 5: WebSocket 降级（5s 轮询）**
```
FastMo buffer: 数据间隔大, staleness > 10s
→ reliable = false → accelBonus = 0
→ 回退到纯 K线评分（当前行为不变） ✅
```

**场景 6: 前端关闭 bonus（enabled=false）**
```
→ 整个加速度逻辑跳过
→ adjustedFinalScore = baseFinalScore（零改动行为） ✅
```

### 调参指南

| 想达到的效果 | 调什么 | 方向 |
|-------------|--------|------|
| 更早入场（更激进） | accelRatioThreshold ↓ 或 bonusMultiplier ↑ | 更多交易会提前触发 |
| 更晚入场（更保守） | accelRatioThreshold ↑ 或 bonusMultiplier ↓ | 只有强加速才加分 |
| 完全禁用 | enabled = false | 回到 Phase 3 行为 |
| 限制 bonus 贡献 | bonusCap ↓ | 单靠加速度永远不够过阈值 |

## 五、安全边界

### 5.1 四重安全机制

```
Layer 1: bonusCap 上限（默认8，前端可调，最低2） — 不可能单靠加速度过阈值
Layer 2: FastMo checkGate 不变 — decel<0.5 或 >5.0 仍然拦截
Layer 3: 开盘冲量守卫不变 — ATR 超标拦截
Layer 4: WebSocket 降级检测 — 不可靠数据自动禁用 bonus
```

### 5.2 不动的代码

- 熔断器（dailyRealizedPnL）— 不涉及
- 止损逻辑 — 不涉及
- 资金检查 — 不涉及
- 阶梯锁利 — 不涉及
- directionalScoreMin 阈值 — 不修改

### 5.3 可观测性

所有 bonus 计算写入 decision log（logDecision）：
```json
{
```
