## 二、P0: 信号PnL vs 实际成交PnL 偏差

### 2.1 根因

PnL 记录在 `strategy-scheduler.service.ts:4552` 的 `logSellSignal()` 调用中。此时的执行流程：

```
1. calculatePnL(positionCtx)  ← 使用当前市场中间价
2. logSellSignal(metadata={netPnL, ...})  ← 记录到 strategy_signals 表
3. executeSellOrder()  ← 提交 ELO(增强限价单)
4. 订单在交易所成交  ← 实际成交价可能显著不同
```

**问题**: 步骤2记录的PnL是基于步骤1的市场价估算,而非步骤4的实际成交价。前端分析页面读取的是步骤2的数据。

## 三、P0: auto_trades.pnl 缺少期权乘数

### 位置
`api/src/services/basic-execution.service.ts:1335`

```typescript
const pnl = (avgPrice - openPrice) * Math.min(filledQuantity, openQuantity);
// 缺少 * multiplier (期权为100)
```

### 影响
- `auto_trades` 表中所有期权交易的 PnL 缩小 100 倍
- Dashboard 统计如果读取 `auto_trades.pnl`,期权收益会被严重低估
- 例: SPY 3/27 实际亏损 -$15, auto_trades 记录为 -$0.15

## 四、P1: BUY 信号配对未过滤状态

### 位置
`frontend/app/quant/analysis/page.tsx:205`

```typescript
const buySignals = signals.filter(s => s.signal_type === 'BUY')
// ❌ 缺少 && s.status === 'EXECUTED'
```

### 影响
- 当前数据中有 **27 条 FILTERED** BUY 信号(从未实际下单)
- 这些信号如果被匹配到 SELL 信号,会导致:
  - 错误的入场得分(score)
  - 错误的方向标签(direction)
  - 错误的策略类型(strategyName)
- 在当前数据集中,由于 EXECUTED BUY 信号时间更接近 SELL,多数情况下 FILTERED 信号不会被优先匹配
- 但当某日 BUY(EXECUTED) 数量 < SELL 数量时,会回退到匹配 FILTERED 信号 — **产生幽灵交易**

### 验证
```
3/21: buy=5(0 executed) sell=0 → 无 SELL,暂无影响
3/22: buy=5(0 executed) sell=0 → 无 SELL,暂无影响
3/26: buy=13(4 exec, 9 filtered) sell=4 → 4 SELL 匹配 4 exec BUY,刚好够用
3/27: buy=15(5 exec, 10 filtered) sell=5 → 5 SELL 匹配 5 exec BUY,刚好够用
```
当前数据恰好未触发错误配对,但边界极为脆弱。

## 五、P2: netPnLPercent 字段语义错误

### 位置
`api/src/services/strategy-scheduler.service.ts:4561`

```typescript
netPnLPercent: pnl.grossPnLPercent, // Fix 9: 使用 grossPnLPercent 避免 NaN
```

### 影响
- 前端读取 `metadata.netPnLPercent` 作为"盈亏百分比"显示
- 实际值是 **毛盈亏百分比**(不含手续费)
- 对于小额交易,手续费占比可达 5-10%,误差显著
- 例: NVDA P180 3/25, 信号 netPnLPercent=1.6%(实为 gross), 而 netPnL=$0.45 对应实际 net 仅 0.05%

### 背景
Fix 9 是为了避免 T+1 手续费数据不完整导致 NaN。这是一个合理的防御性编码,但字段命名有误导性。

## 六、P2: 反向分析数据源不一致

### 问题
反向分析 tab 的对比:
- **原始 PnL**: 来自 `strategy_signals` 的信号配对(决策时价格,含前述偏差)
- **反向 PnL**: 来自 `option_trade_analysis` 表(基于 K线数据计算)
- 两个数据源的价格基准和时间窗口不同,比较结果不公平

### 具体表现
- `reverseComparison.origCount` 来自信号配对(filteredTrades.length)
- `reverseComparison.revCount` 来自分析表(analysis.filter(a => a.reverse_pnl !== null).length)
- 两个计数可能不一致

## 七、修复建议

### 7.1 P0-信号PnL → 使用实际成交价重算 (推荐方案)

**方案A: 后端订单成交后更新信号PnL (最准确)**
- 在 `basicExecutionService.recordTrade()` 中,当 SELL 订单成交后
- 用实际成交价重新计算 PnL
- 更新对应 `strategy_signals` 记录的 metadata

**方案B: 前端从订单匹配计算PnL (改动最小)**
- 在前端配对逻辑中,将信号与订单交叉匹配
- 用订单实际成交价计算 PnL
- 不依赖信号 metadata 中的 PnL

### 7.2 P0-auto_trades乘数

```typescript
// 修复: 检测期权并乘以 multiplier
const isOption = /^[A-Z]+\d{6}[CP]\d+/.test(symbol);
const multiplier = isOption ? 100 : 1;
const pnl = (avgPrice - openPrice) * Math.min(filledQuantity, openQuantity) * multiplier;
```

### 7.3 P1-BUY状态过滤

```typescript
const buySignals = signals.filter(s => s.signal_type === 'BUY' && s.status === 'EXECUTED')
```

### 7.4 P2-字段语义

选项:
1. 重命名字段为 `grossPnLPercent`(需前端同步改)
2. 或在前端显示时标注"(毛)"

## 八、数据流全景

```
策略调度器 ─→ 生成 BUY 信号(market price) ─→ 提交买入订单 ─→ 成交(execution price)
         │                                              ↓
         │                                    positionCtx.entryPrice = 实际成交价
         │
         ├→ 监控持仓 ─→ calculatePnL(market mid) ─→ 满足退出条件
         │                    ↓
         │              logSellSignal(netPnL=估算值)  ← ⚠️ 此处记录的PnL是估算
         │                    ↓
         │              提交卖出订单 ─→ 成交(execution price)  ← 实际价格可能不同
         │                                     ↓
         │                              recordTrade(pnl=价差*qty, ❌缺少*100)
         │
         └→ 前端分析页 ─→ 读取 strategy_signals ─→ 信号配对(❌含FILTERED)
                             ↓
                        显示 PnL = 信号估算值(❌非实际)
                        显示 PnL% = grossPnLPercent(❌标为net)
```
