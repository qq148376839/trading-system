# MU 期权资金预检问题分析

## 问题描述

2026-03-17 策略10 对 MU 生成了 172 条 PENDING 信号，全部因资金校验失败无法执行。

错误日志：
```
资金申请被拒绝 - 标的 MU.US 持仓超过限制:
已用0.00 + 申请2096.63 = 2096.63 > 上限1000.00
(策略总资金 2000.00 / 1 个标的, 集中度上限50%)
```

## 根本原因

### 两阶段预算不一致

| 阶段 | 调用 | survivorCount | 预算 |
|---|---|---|---|
| Phase B: `generateSignal()` | `getMaxPositionPerSymbol(strategyId)` — 无参数 | 无 | ~$2000 (BY_GROUP, 1组) |
| Phase D: `requestAllocation()` | `resolveDenominator(..., survivorCount, maxConcentration)` | 有 | ~$1000 (50%集中度) |

### 强制最低1张合约

`generateSignal()` L862 (修复前):
```typescript
contracts = Math.max(1, n - 1);
```

当 MU 期权1张合约成本 ~$2096 超过 Phase B 预算 ~$2000 时，`n-1 = 0`，但 `Math.max(1, 0) = 1`，强制生成1张合约的信号。

信号写入 DB (`logSignal` → `INSERT INTO strategy_signals ... status='PENDING'`)，然后到 Phase D 被 `requestAllocation` 拒绝，留下大量 PENDING 信号。

### 无退避机制

每个策略周期(~15秒)都重新评估 MU，每次都生成一条无法执行的 PENDING 信号，累积至 172 条。

## 修复内容（260317）

### 1. 资金预检拦截

位置: `option-intraday-strategy.ts` L870-889

当 `MAX_PREMIUM` 模式下1张合约都超过预算时，直接返回 `null`（拒绝信号），不再强制 `contracts=1`。

```typescript
if (n === 1) {
  // 1张合约成本超预算，拒绝信号
  this.capitalPrecheckRejectUntil.set(symbol, Date.now() + 5 * 60_000);
  logData.rejectionCheckpoint = 'capital_precheck';
  return null;
}
```

### 2. 拦截缓存（5分钟退避）

位置: `option-intraday-strategy.ts` L244-245, L494-498

新增 `capitalPrecheckRejectUntil: Map<string, number>` 缓存，被拦截的标的5分钟内不再重复评估，消除信号刷屏问题。

### 3. 分组竞价自然排除

因为 `generateSignal()` 返回 `null`，`evaluateIdleSymbol()` 不会将该标的加入候选列表，竞价排名自动在 TSLA/NVDA 之间进行，无需额外踢出逻辑。

## 效果

| 指标 | 修复前 | 修复后 |
|---|---|---|
| MU PENDING 信号/天 | ~172 条 | 0 条 |
| 无效 `generateSignal` 调用 | ~172 次 | ~1 次(首次)+缓存跳过 |
| 竞价参与标的 | MU(必败)+TSLA+NVDA | TSLA+NVDA |
| 日志噪音 | 高（每周期一条 warn） | 低（首次 warn + 5分钟静默） |

## 与 TSLA 换仓修复的关系

同日修复的 TSLA 换仓问题（`recentExits` 机制）和本次 MU 修复是独立的：
- TSLA 修复：退出后同组标的冷却联动，防止卖出后立即换仓
- MU 修复：信号阶段资金预检，防止生成无法执行的信号

两者互不冲突，分别解决不同维度的问题。
