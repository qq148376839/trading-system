> 核心原则: **关键参数不能写死，必须基于市场状态自适应**

## 一、为什么写死参数必然失败

### 第一性原理：市场是非平稳的

写死参数的隐含假设是：**最优参数值在所有市场条件下恒定不变**。

这与现实矛盾：

```
VIX=12 的低波动牛市:
  → BTC 与 SPY 高度正相关 (+0.7) → BTC 信号有用
  → 趋势清晰 → marketScore 可靠
  → 期权波动小 → VWAP 2% 偏离 = 极端事件

VIX=35 的高波动恐慌市:
  → BTC 与 SPY 相关性断裂 (-0.2) → BTC 信号是噪音
  → 趋势剧烈震荡 → marketScore 频繁反转
  → 期权波动大 → VWAP 2% 偏离 = 正常波动
```

同一个参数值 (BTC 权重 15%) 在一个环境下是信号，在另一个环境下是噪音。**写死 = 在两个环境中各错一半**。

### 量化影响

从 3/17-3/31 数据：
- BTC 权重 15% 全程固定 → 至少 2-3 天的方向错误受 BTC 噪音驱动
- VWAP 饱和点 2% 全程固定 → NVDA (日均波动 3-4%) 和 SPY (日均波动 0.8%) 用同一个饱和点
- marketScore 20% 权重全程固定 → 趋势市和震荡市贡献相同权重

### 正确的心智模型

```
写死参数:  optimal_param = constant
正确模型:  optimal_param = f(market_state)

其中 market_state = {
  volatility_regime,     // VIX 水平和变化趋势
  trend_clarity,         // 趋势强度（ADX 类指标）
  correlation_structure,  // 跨资产相关性矩阵
  recent_signal_quality,  // 信号的近期准确率
  intraday_character      // 当日的特征（gap、range、volume）
}
```

## 二、动态参数架构：三层防护

### 设计原则

动态化不是"让参数随意变化"，而是**在安全边界内，基于可观测信号自适应**。

```
┌─────────────────────────────────────────┐
│  Layer 3: 安全硬限 (Hard Clamps)          │
│  → 参数的绝对最小/最大值，永不突破        │
│  → 防止极端市场条件导致参数失控            │
│                                           │
│  ┌─────────────────────────────────────┐  │
│  │  Layer 2: 连续自适应 (Adaptation)    │  │
│  │  → 基于实时可观测量（相关性、ATR等）  │  │
│  │  → 平滑调整，EMA 衰减避免过拟合       │  │
│  │                                       │  │
│  │  ┌─────────────────────────────────┐  │  │
│  │  │  Layer 1: 基准值 (Baseline)      │  │  │
│  │  │  → 来自策略配置的默认值           │  │  │
│  │  │  → 相当于当前的硬编码值           │  │  │
│  │  └─────────────────────────────────┘  │  │
│  └─────────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

**最终参数 = clamp(baseline × adaptiveFactor, hardMin, hardMax)**

## 三、六组核心动态参数

### 参数组 1: 信号分量权重（intradayScore 内部）

#### 当前（写死，option-recommendation.service.ts）

```
underlying_1m:  30%  (L437, momentum * 0.3)
vwap:           15%  (L450, rawScore * 0.15)
spx_hourly:     25%  (L463, spxMomentum * 0.25)
btc_hourly:     15%  (L480, btcHourlyMomentum * 0.15)
usd_hourly:     15%  (L492, -usdHourlyMomentum * 0.15，反向)
```

#### 应该是什么

每个分量的权重应反映它**当前**对标的方向的预测力。

**驱动量: 滚动相关性**

```
对于每个分量 C:
  relevance_C = abs(pearson(C_returns, underlying_returns, window=3days))

  weight_C = relevance_C / sum(all_relevances)  // 归一化到总和=100%
```

具体计算:

```typescript
// 每个调度周期（~15秒）可计算
const btcRelevance = Math.abs(
  pearsonCorrelation(btcHourlyReturns, underlyingReturns)  // 已有函数
);
const usdRelevance = Math.abs(
  pearsonCorrelation(usdHourlyReturns, underlyingReturns)
);
const spxRelevance = Math.abs(
  pearsonCorrelation(spxHourlyReturns, underlyingReturns)
);
// underlying_1m 和 vwap 是直接信号，给固定基础权重
const underlyingBaseRelevance = 0.40;  // 始终是主力信号
const vwapBaseRelevance = 0.15;        // 始终有结构意义

// 归一化
const crossAssetTotal = btcRelevance + usdRelevance + spxRelevance;
const crossAssetBudget = 1.0 - underlyingBaseRelevance - vwapBaseRelevance; // = 0.45

const weights = {
  underlying_1m: underlyingBaseRelevance,                         // ~40%
  vwap:          vwapBaseRelevance,                                // ~15%
  spx:           (spxRelevance / crossAssetTotal) * crossAssetBudget,  // 动态分配
  btc:           (btcRelevance / crossAssetTotal) * crossAssetBudget,  // 动态分配
  usd:           (usdRelevance / crossAssetTotal) * crossAssetBudget,  // 动态分配
};
```

**实际效果举例**:

```
场景 A: BTC-NVDA 相关性高（0.6）, USD 低（0.1）, SPX 中（0.4）
  → btc: 0.6/(0.6+0.1+0.4) × 0.45 = 24.5%
  → usd: 0.1/1.1 × 0.45 = 4.1%
  → spx: 0.4/1.1 × 0.45 = 16.4%

场景 B: BTC 不相关（0.05）, USD 不相关（0.08）, SPX 高（0.7）
  → btc: 0.05/0.83 × 0.45 = 2.7%
  → usd: 0.08/0.83 × 0.45 = 4.3%
  → spx: 0.7/0.83 × 0.45 = 37.9%
```

场景 B 就是 3/17-3/31 的实际情况 — BTC/USD 自动降权到 ~7%（从固定 30% 降到 7%），噪音大幅减少。

#### 三层防护

| 层 | 参数 | 值 |
|----|------|-----|
| Layer 1 基准 | 初始权重 | underlying=40%, vwap=15%, 跨资产=45% |
| Layer 2 适应 | 滚动相关性窗口 | 3 个交易日（~390 根 1min bars） |
| Layer 3 硬限 | btc/usd 单项最大 | 20%（永不让单个跨资产信号占比超过 20%） |
| Layer 3 硬限 | btc/usd 单项最小 | 0%（允许完全关闭） |
| Layer 3 硬限 | underlying 最小 | 30%（始终是主信号） |

#### 数据可行性: ✅

- `calculatePearsonCorrelation()` 已实现（option-recommendation.service.ts L613-629，私有方法）
- 当前仅用于 `checkBTCSPXCorrelation`（日K返回率），需扩展为通用工具函数
- 底层标的 1min K线已有（20根）→ 需扩展到 ~400 根（3天）或用 5min 汇总
- SPX/BTC/USD 分时数据已有（~500根）

### 参数组 2: finalScore 合成权重

#### 当前（写死，L161-164）

```
marketScore × 0.2 + intradayScore × 0.6 + timeWindowAdj × 0.2
```

#### 应该是什么

marketScore 的权重应反映**当前趋势的可读性**。趋势越清晰，marketScore 越可靠，权重越高。

**驱动量: 趋势清晰度（ADX 概念）**

```
// 用 SPX 日K线计算趋势一致性
const recentDays = spxDaily.slice(-5);
const sameDirection = recentDays中连续同向天数;
const avgDailyMove = 最近5天平均|日涨跌幅|;

// trendClarity: 0（完全混沌）到 1（完美趋势）
const trendClarity = (sameDirection / 5) × Math.min(avgDailyMove / 1.0, 1.0);
// 5天全涨且平均涨1%+ → trendClarity ≈ 1.0
// 涨跌交替且幅度小 → trendClarity ≈ 0.1
```

权重分配:

```
marketWeight = 0.05 + trendClarity × 0.25
// 趋势清晰: 0.05 + 0.25 = 0.30 (marketScore 占 30%)
// 趋势混沌: 0.05 + 0.00 = 0.05 (marketScore 几乎不参与)

intradayWeight = 1.0 - marketWeight  // 剩余全给 intraday
// 趋势清晰: 0.70
// 趋势混沌: 0.95 (几乎完全靠日内信号)
```

**关键改动: timeWindowAdjustment 不再作为得分分量**

timeWindow 从 finalScore 公式中移除，改为**阈值修正系数**:

```
// 新公式（两项，非三项）
finalScore = marketScore × marketWeight + intradayScore × intradayWeight

// timeWindow 作为阈值修正
effectiveThreshold = baseThreshold × timeThresholdFactor
// 开盘第1小时: factor = 0.85 (降低阈值，更容易入场，但不偏向任何方向)
// 正常时段:    factor = 1.0
// 尾盘:        factor = 1.5~2.5
```

**实际效果 — 4/1 NVDA 案例回测**:

```
当前写死 (VIX≈20, vixFactor=1.0):
  finalScore = -0.7×0.2 + 10.3×0.6 + 20×0.2 ≈ 10.0 (精确值 10.018)
  策略阈值 = 9 × 1.0 = 9 (directionalScoreMin × vixFactor, L298)
  10.0 > 9 → 入场 (其中 timeWindow 贡献 +4.0 = 40%，是 CALL 偏置)

动态化后（假设 trendClarity=0.3，弱趋势）:
  marketWeight = 0.05 + 0.3×0.25 = 0.125
  intradayWeight = 0.875
  finalScore = -0.7×0.125 + 10.3×0.875 = -0.09 + 9.01 = 8.9
  effectiveThreshold = 9 × 1.0 × 0.85 (vixFactor × 开盘因子) = 7.65
  8.9 > 7.65 → 入场 ✓

  注意: 入场决策相同，但驱动力完全来自 intradayScore，不是 timeWindow 偏置
```

#### 三层防护

| 层 | 参数 | 值 |
|----|------|-----|
| Layer 1 基准 | marketWeight | 0.15 |
| Layer 2 适应 | trendClarity × 0.25 | 0~0.25 |
| Layer 3 硬限 | marketWeight 范围 | [0.05, 0.30] |
| Layer 3 硬限 | timeThresholdFactor 范围 | [0.7, 3.0] |

### 参数组 3: marketScore 内部时间尺度

#### 当前（写死，analyzeMarketTrend() → L523）

```
SPX 趋势 = (currentPrice - avg20) / avg20 × 100 × 5  // L523: 放大5倍
// 然后 × 0.4 权重进入 marketScore (L318)
// marketScore 最终 × 0.2 进入 finalScore (L162)
```

#### 应该是什么

趋势信号的时间窗口应该匹配当前趋势的实际周期。

**驱动量: 市场状态 → 选择时间窗口**

```
// 不用单一窗口，用多时间窗口加权
const trend3d  = (current - avg3)  / avg3  × 100;  // 3天动量
const trend10d = (current - avg10) / avg10 × 100;  // 10天趋势
const trend20d = (current - avg20) / avg20 × 100;  // 20天趋势

// 权重由趋势一致性决定
if (sign(trend3d) === sign(trend10d) === sign(trend20d)) {
  // 所有时间窗口同向 → 强趋势 → 中长期权重高
  weights = { d3: 0.3, d10: 0.4, d20: 0.3 };
} else if (sign(trend3d) !== sign(trend20d)) {
  // 短期与长期反向 → 可能是转折点 → 短期权重高
  weights = { d3: 0.7, d10: 0.2, d20: 0.1 };
} else {
  // 混合状态
  weights = { d3: 0.5, d10: 0.3, d20: 0.2 };
}

trendScore = trend3d × weights.d3 + trend10d × weights.d10 + trend20d × weights.d20;
```

**实际效果 — 3/31 转折日**:

```
3/31: SPX 大涨 +1.73%
  trend3d  = (6529-6369)/6369 = +2.5%  (3天前=3/27收盘6369)
  trend10d = (6529-6624)/6624 = -1.4%  (10天前=3/18收盘6624)
  trend20d = (6529-6716)/6716 = -2.8%  (20天前=3/04估计6716)

  sign(trend3d)=+ ≠ sign(trend20d)=- → 转折点 → d3 权重 0.7
  trendScore = 2.5×0.7 + (-1.4)×0.2 + (-2.8)×0.1 = 1.75 - 0.28 - 0.28 = +1.19

当前写死(20日均线):
  trendScore = -2.8 × 5 × 0.4 = -5.6 (负值 → 推 PUT → 错误)

动态多窗口:
  trendScore = +1.19 × 5 × 0.4 = +2.4 (正值 → 不推 PUT → 正确！)
```

**3/31 的方向错误被修复**。

#### 三层防护

| 层 | 参数 | 值 |
|----|------|-----|
| Layer 1 基准 | 窗口 | [3, 10, 20] 天 |
| Layer 2 适应 | 窗口间权重 | 由趋势一致性决定 |
| Layer 3 硬限 | 短期窗口最大权重 | 0.8（不完全忽视长期） |
| Layer 3 硬限 | 长期窗口最小权重 | 0.05 |

### 参数组 4: VWAP 饱和点

#### 当前（写死，L448-449）

```
const distancePct = ((lastPrice - vwap) / vwap) * 100;     // L448
const rawScore = Math.max(-100, Math.min(100, distancePct * 50));  // L449
// amplifier=50, 2% 偏离即饱和（100/50=2%），所有标的同一标准
```

#### 应该是什么

饱和点应该与标的的**当日实际波动率**成正比。

**驱动量: 底层标的的日内 ATR**

```
// 用底层标的的 1min K线计算当日 ATR
const intradayATR = calculateATR(underlying1mBars, 14);  // technical-indicators.ts L21-48
const atrPct = (intradayATR / currentPrice) × 100;       // 转为百分比

// 饱和点 = 当日 ATR 的 3 倍
// 直觉: VWAP 偏离达到 3×ATR 才算"极端"
const saturationPct = Math.max(1.0, atrPct × 3);

// 放大系数 = 100 / saturationPct
const amplifier = 100 / saturationPct;
const vwapScore = clamp(distancePct × amplifier, -100, 100);
```

**实际效果**:

```
SPY (低波动, 日内 ATR ≈ 0.3%):
  saturationPct = 0.3 × 3 = 0.9%
  amplifier = 100/0.9 = 111
  偏离 0.5% → score = 55 (中等信号)
  偏离 1.0% → score = 100 (饱和 → 极端事件)

NVDA (高波动, 日内 ATR ≈ 1.2%):
  saturationPct = 1.2 × 3 = 3.6%
  amplifier = 100/3.6 = 28
  偏离 0.5% → score = 14 (微弱信号)
  偏离 2.0% → score = 56 (中等信号)
  偏离 3.6% → score = 100 (饱和 → 极端事件)
```

当前写死 (amplifier=50, saturation=2%):
- SPY 偏离 0.5% → score = 25 ← 低估了（SPY 偏离 0.5% 很不寻常）
- NVDA 偏离 2% → score = 100 ← 饱和了（NVDA 偏离 2% 很正常）

**动态化后，SPY 的 0.5% 偏离获得比 NVDA 更高的得分 — 这才是对的。**

#### 三层防护

| 层 | 参数 | 值 |
|----|------|-----|
| Layer 1 基准 | amplifier | 50（≈ 2% 饱和） |
| Layer 2 适应 | 100 / (intradayATR% × 3) | 实时计算 |
| Layer 3 硬限 | saturationPct 范围 | [0.8%, 8%] |
| Layer 3 硬限 | amplifier 范围 | [12.5, 125] |

### 参数组 5: VIX 阈值

#### 当前（写死，L340-355 在 calculateMarketScore 内部）

```
VIX > 35 → score -= 50  (极度恐慌)    // 影响链: marketScore -50 → finalScore -10
VIX > 25 → score -= 20  (高恐慌)      // 影响链: marketScore -20 → finalScore -4
VIX < 15 → score += 10  (低恐慌)      // 影响链: marketScore +10 → finalScore +2
// 注: 这些是在 marketScore 空间的绝对值，× 0.2 后才进入 finalScore
// VIX>35 的 -50 可压倒 SPX 趋势的 40% 权重（SPX trendStrength 通常 ±30 范围）
```

#### 应该是什么

VIX 的"高/低/极端"应该相对于**近期 VIX 分布**定义，不是绝对值。

**驱动量: VIX 的滚动分位数**

```
// 用最近 60 个交易日的 VIX 数据
const vixHistory = spxDaily.map(d => d.vix).slice(-60);
const vixMean = mean(vixHistory);
const vixStd = stddev(vixHistory);

const extremeThreshold = vixMean + 2.0 × vixStd;
const highThreshold    = vixMean + 1.0 × vixStd;
const lowThreshold     = Math.max(10, vixMean - 1.0 × vixStd);
```

**实际效果**:

```
低波动期 (VIX 均值=14, 标准差=2):
  extreme = 14 + 4 = 18 ← VIX 18 就算极端了
  high = 14 + 2 = 16
  low = 12

高波动期 (VIX 均值=28, 标准差=5):
  extreme = 28 + 10 = 38 ← VIX 38 才算极端
  high = 28 + 5 = 33
  low = 23

当前写死 (extreme=35, high=25, low=15):
  低波动期: VIX 25 被认为"高恐慌"，但实际只是回到正常水平 ← 过度惩罚
  高波动期: VIX 33 不算"高恐慌"，但相对于均值28已经偏高 ← 反应不足
```

#### 惩罚分数也应自适应

```
// 惩罚力度应与 VIX 变化速度成正比，而非绝对水平
const vixChangeRate = (currentVix - vixMean) / vixStd;  // Z-score

const vixPenalty = (() => {
  if (vixChangeRate > 2.0) return -50;   // 极端（2σ事件）
  if (vixChangeRate > 1.0) return -20;   // 偏高
  if (vixChangeRate < -1.0) return +10;  // 偏低（市场平静）
  return 0;  // 正常范围
})();
```

**这样 VIX 的惩罚机制自动适应任何 VIX 环境**。2022 年 VIX 均值 25 和 2024 年 VIX 均值 14 使用同一套逻辑，无需手动调阈值。

#### 三层防护

| 层 | 参数 | 值 |
|----|------|-----|
| Layer 1 基准 | 阈值 | extreme=35, high=25, low=15 |
| Layer 2 适应 | VIX 60日 Z-score | 实时计算 |
| Layer 3 硬限 | extreme 范围 | [20, 60]（物理意义边界） |
| Layer 3 硬限 | 最大惩罚 | -60 分（不让 VIX 单项否决所有信号） |

### 参数组 6: 入场阈值 (baseThreshold)

#### 当前（部分动态）

```
策略10 DB 覆盖: directionalScoreMin = 9, spreadScoreMin = 7

已有自适应机制 (option-intraday-strategy.ts L283-298):
  effectiveThreshold = Math.round(directionalScoreMin × vixFactor)  // L298
  vixFactor = clamp(VIX / 20, 0.5, 2.5)    // L285

  VIX=10 → threshold = Math.round(9 × 0.5) = 5 (低波动，宽松入场)
  VIX=20 → threshold = Math.round(9 × 1.0) = 9 (基准)
  VIX=40 → threshold = Math.round(9 × 2.0) = 18 (高波动，严格入场)

另有尾盘递增 (option-recommendation.service.ts L913-935):
  推荐层: dynamicThreshold = 15 × timeFactor (1.0→3.0)
  但这作用在推荐层，不影响策略层实际入场决策
```

**问题: VIX 自适应是正确的方向，但维度单一。** 缺少信号质量反馈。

#### 应该增加什么

入场阈值还应反映**信号的近期可靠性**。在已有 VIX 自适应基础上叠加准确率反馈：

**驱动量: 近期信号准确率**

```
// 数据来源: strategy_signals 表（已有）
// 查询最近 N 笔 EXECUTED 信号的胜率
const recentSignals = await db.query(
  'SELECT metadata->>'netPnL' as pnl FROM strategy_signals
   WHERE strategy_id=10 AND status='EXECUTED' AND signal_type='SELL'
   ORDER BY created_at DESC LIMIT 20'
);
const recentWR = recentSignals.filter(s => Number(s.pnl) > 0).length / recentSignals.length;

// 阈值调整系数
// WR=50% → factor=1.0（基线）
// WR=30% → factor=1.4（提高阈值40%，信号不可靠要更严格）
// WR=70% → factor=0.7（降低阈值30%，信号可靠可以更宽松）
const accuracyFactor = 1.0 + (0.50 - recentWR) × 2.0;

// 完整公式 = 已有VIX自适应 × 新增准确率反馈 × timeWindow修正
const effectiveThreshold = baseThreshold × vixFactor × accuracyFactor × timeThresholdFactor;
```

**实际效果** (以 VIX=20, vixFactor=1.0 为例):

```
3/17-3/31 (WR=44%):
  accuracyFactor = 1.0 + (0.50-0.44)×2 = 1.12
  effectiveThreshold = 9 × 1.0 × 1.12 = 10.1 → 更严格，减少错误入场

假设修复后 WR=55%:
  accuracyFactor = 1.0 + (0.50-0.55)×2 = 0.90
  effectiveThreshold = 9 × 1.0 × 0.90 = 8.1 → 更宽松，抓住更多机会

如果同时 VIX=30 (市场恐慌):
  effectiveThreshold = 9 × 1.5 × 0.90 = 12.2 → VIX效应压倒准确率宽松
```

**这是一个自稳定的反馈环**：信号差 → 门槛高 → 减少入场 → 只有高确信度的信号通过 → 表现改善 → 门槛逐渐降低。

#### 跨日 vs 当日

- **跨日准确率**（20笔窗口 ≈ 4-5 天）: 捕捉宏观信号质量变化
- **当日准确率**（当日已完成交易）: 捕捉今天的市场是否适合策略

```
// 当日已有交易时，融合两个维度
const crossDayWR = ...;  // 最近20笔
const todayWR = ...;     // 今日已完成交易（如果 >= 2 笔）

const blendedWR = todayTrades >= 2
  ? crossDayWR × 0.6 + todayWR × 0.4  // 当日样本可信时融合
  : crossDayWR;                         // 当日样本不足时只用跨日
```

#### 三层防护

| 层 | 参数 | 值 |
|----|------|-----|
| Layer 1 基准 | baseThreshold | 9 (DB entryThresholdOverride) |
| Layer 2 适应(已有) | vixFactor | 0.5~2.5 (VIX/20, L285) |
| Layer 2 适应(新增) | accuracyFactor | 0.7~1.5 |
| Layer 3 硬限 | effectiveThreshold 范围 | [5, 20] |
| Layer 3 硬限 | 最小样本量 | 10 笔才启用 accuracyFactor |

## 四、反过拟合机制

### 问题：动态参数可能比写死更糟

如果适应太快 → 对噪音反应过度 → 参数剧烈波动 → 比写死更差。

### 三道防线

#### 防线 1: EMA 平滑（防抖动）

所有动态参数的更新使用 EMA（指数移动平均）而非瞬时值:

```
// α = 平滑因子, 越小越稳定
// 推荐: α = 2/(N+1), N=适应周期的观测数

const α_correlation = 2 / (60 + 1);  // 相关性: ~60个观测周期（≈1小时@15s间隔）
const α_vix = 2 / (20 + 1);          // VIX: ~20个交易日

// 每次更新
newWeight = α × instantWeight + (1 - α) × prevWeight;
```

**效果**: BTC 相关性从 0.5 突然跌到 0.1 时，权重不会立刻从 15% 变到 3%，而是在 ~1 小时内平滑过渡。

#### 防线 2: 最小变化阈值（防微调）

```
// 参数变化小于 minDelta 时不更新
const minDelta = currentValue × 0.05;  // 5% 的最小变化

if (Math.abs(newValue - currentValue) < minDelta) {
  return currentValue;  // 不变
}
```

**效果**: 避免权重在 14.8% ↔ 15.2% 之间无意义地振荡。

#### 防线 3: 变化速率限制（防突变）

```
// 每次更新最多变化 maxChangeRate
const maxChangeRate = 0.10;  // 10%/周期

const change = (newValue - currentValue) / currentValue;
if (Math.abs(change) > maxChangeRate) {
  return currentValue × (1 + Math.sign(change) × maxChangeRate);
}
```

**效果**: 即使市场剧变，参数也只能以 10%/周期 的速度调整，不会跳变。

### 组合效果

```
瞬时信号: BTC-SPX 相关性从 0.6 骤降到 -0.2（如恐慌事件）

无防护: btcWeight 立即从 20% 变到 0% → 可能是噪音导致的假信号
有防护:
  1. EMA 平滑: 0.6 → 0.58 → 0.55 → ... → 逐渐下降
  2. 最小变化: 如果变化<5%, 不更新
  3. 速率限制: 每周期最多变化10%

  实际路径: 20% → 18% → 16.2% → ... → 约20分钟后到 5%
  如果相关性恢复 → 权重也会逐渐恢复
  如果相关性持续低 → 权重最终到达 ~3%（接近 hardMin）
```

## 五、参数之间的耦合与一致性

### 耦合矩阵

参数不是独立的。改一个可能需要改另一个:

```
                信号权重  合成权重  marketScore窗口  VWAP饱和  VIX阈值  入场门槛
信号权重          -         →          ×              ×         ×        →
合成权重          ←         -          →              ×         ×        →
marketScore窗口   ×         ←          -              ×         ×        ×
VWAP饱和          ×         ×          ×              -         ×        ×
VIX阈值           ×         ×          ×              ×         -        →
入场门槛          ←         ←          ×              ×         ←        -

→ = 前者变化影响后者
← = 后者变化影响前者
× = 独立
```

### 一致性规则

1. **权重归一化**: 无论怎么动态调整，intradayScore 内部权重必须和 = 100%
2. **finalScore 范围稳定**: 无论权重怎么变，finalScore 的范围应保持在 [-100, +100] 附近
3. **阈值与得分尺度一致**: 如果 intradayScore 的分布因权重变化而改变，入场阈值应自动补偿

### 规则 3 的实现

```
// 跟踪 finalScore 的滚动均值和标准差
const scoreMean = ema(finalScore, 100);
const scoreStd = ema(|finalScore - scoreMean|, 100);

// 阈值基于 Z-score 而非绝对值
const zScoreThreshold = 0.8;  // 要求信号强度 > 0.8σ
const effectiveThreshold = scoreMean + zScoreThreshold × scoreStd;
```

**效果**: 如果参数变化导致 finalScore 整体偏移或缩放，阈值自动跟随。

  AND signal_type = 'SELL'

## 八、已有动态机制与本框架的关系

### 系统已有的自适应能力

代码中已存在两个动态机制，但互相独立且不完整：

| 机制 | 位置 | 作用 | 局限 |
|------|------|------|------|
| VIX 自适应阈值 | option-intraday-strategy.ts L285 | `threshold × VIX/20` | 仅调阈值，不调信号权重 |
| 尾盘阈值递增 | option-recommendation.service.ts L913-935 | `baseThreshold × timeFactor` | 作用在推荐层，策略层不受影响 |
| BTC 相关性条件 | option-recommendation.service.ts L329 | 相关性 >0.5 时 BTC 权重 ×2 | 仅二值（0.1 或 0.2），非连续 |

本框架不是从零开始，而是**将这些孤立的自适应点统一为系统化的动态参数体系**。

### 此框架如何替代之前的"去噪"方案

之前的方案（方向准确率分析文档）:
> BTC/USD 权重设为 0%

这是**正确方向但过于粗暴**。动态框架的做法:
> BTC/USD 权重由相关性驱动，在不相关时**自动趋近 0%**，在相关时**保留信号价值**

两者在 3/17-3/31 期间效果几乎相同（BTC/USD 相关性低 → 权重自动降到 ~3-5%）。但动态框架在 BTC-SPX 高度相关时期能保留 BTC 信号的价值。

### 此框架如何替代"timeWindow 改阈值修正"

之前的方案:
> timeWindow 从得分分量改为阈值修正，用固定曲线

动态框架的做法:
> 相同方向（改为阈值修正），但 timeThresholdFactor 也可以基于历史数据自适应

### 此框架如何替代"marketScore 用 3日动量"

之前的方案:
> 替换 20 日均线为 3 日动量

动态框架的做法:
> 不是替换，而是**多窗口加权**。趋势一致时长期窗口权重高，转折时短期窗口权重高。比单一窗口更鲁棒。

### 独立于本框架的 Phase 3 方案（仍计划实施）

以下两项来自[方向准确率分析](260401-方向准确率第一性原理分析.md) Phase 3C，属于**新增信号**，不在动态参数调优范畴内：

- **动量确认门槛**: 入场前检查标的最近 5 根 1min K线方向，信号与动量反向时推迟入场
- **ORB（开盘区间突破）**: 开盘 30 分钟建立 range，突破后确认方向

这两项可与 Phase 4 并行或独立实施。

## 九、实施路线建议

### Phase 4A: 配置提取（前置条件）

在任何动态化之前，先把 90+ 个 magic number 提取到 `DynamicScoringConfig` 接口中。**不改逻辑，只改结构**。

这样后续每个动态化改动都是：修改一个配置字段的计算方式，而非重构整个系统。

### Phase 4B: 逐个动态化

按影响力排序:

| 序号 | 参数组 | 影响力 | 已有基础 | 新增工作 |
|------|-------|--------|---------|---------|
| 1 | timeWindow → 阈值修正 | 消除 CALL 偏置 | 尾盘递增已有(推荐层) | 改到策略层 + 消除方向偏置 |
| 2 | 信号分量权重（相关性驱动） | 消除 BTC/USD 噪音 | pearsonCorrelation(L613) + BTC二值条件(L329) | 扩展为连续权重分配 |
| 3 | VWAP 动态饱和 | 区分高低波动标的 | calculateATR(technical-indicators.ts L21) | 用 ATR 替换固定 amplifier=50 |
| 4 | marketScore 多窗口 | 加快方向响应 | SPX 日K线(100根) | 增加 3d/10d 窗口 + 一致性权重 |
| 5 | VIX 自校准 | 适应不同波动率环境 | VIX 历史(日K含VIX) | Z-score 替代固定阈值 |
| 6 | 入场阈值自适应 | 信号质量反馈 | VIX 自适应(L285) | 增加 accuracyFactor 维度 + DB 查询 |

### Phase 4C: 反过拟合层

在所有动态参数都实现后，统一加上 EMA 平滑 + 最小变化阈值 + 速率限制。

## 十、一句话总结

**写死参数是用一把固定尺子量所有市场。动态参数是让尺子随市场自动伸缩 — 但必须给尺子装上刻度（安全硬限）和阻尼器（反过拟合），否则尺子本身会成为噪音源。**
