# TSLA 卖出后30秒买入问题分析

## 问题描述

2026-03-17 策略10 对 TSLA 期权执行了如下操作：

| 时间 (北京) | 动作 | 合约 | 成交价 | 原因 |
|---|---|---|---|---|
| 21:45:42 | BUY 5股 | TSLA260318C402500 | $1.87 | 牛市价差得分9.8 |
| 22:03:46 | SELL 5股 | TSLA260318C402500 | $1.72 | 阶梯锁利：峰值19.8%→跌破0%底线 |
| 22:04:16 | BUY 1股 | TSLA260318C395000 | $5.18 | 牛市价差得分9.3 |

**卖出→买入间隔仅30秒，整体亏损约$78。**

## 逻辑链分析

### 1. Symbol Pool 构成

期权策略的 symbol pool 来自 `stockSelector.getSymbolPool()`（L422），返回的是**底层标的**（如 `TSLA.US`、`NVDA.US`），不是期权合约名。

### 2. 分组机制 (`getCorrelationGroup`)

```
位置: strategy-scheduler.service.ts L103-106

function getCorrelationGroup(symbol, correlationMap) {
  const map = correlationMap || DEFAULT_CORRELATION_GROUPS;
  return map[symbol] || symbol;  // 不在映射中的标的 → 自身作为组名
}
```

分组映射由 `buildCorrelationMap()` 从策略配置的 `correlationGroups.groups` 构建。
输入格式: `{ GROUP_0: ['TSLA.US', 'NVDA.US', 'MU.US'] }`
输出格式: `{ 'TSLA.US': 'GROUP_0', 'NVDA.US': 'GROUP_0' }`

**关键:** 这里的 key 是底层标的名（`TSLA.US`），不是期权合约名。

### 3. activeEntries 映射

```
位置: L146-153, getCrossSymbolState()

activeEntries: Map<string, number>  // key=symbol, value=entryTime
```

当期权买入成交后，`activeEntries.set(symbol, Date.now())` 中的 `symbol` 是底层标的名（因为 symbol pool 用的就是底层标的）。

退出时 `activeEntries.delete(symbol)` 也是删除底层标的。

### 4. 冷却期机制 (`cooldownUntil`)

```
位置: L75-93, computeCooldownUntil()

0DTE 规则:
- 首笔: 0分钟冷却
- 2-4笔: >= 1分钟
- 5+笔: >= 3分钟
```

冷却期存储在 `strategy_instances.context.cooldownUntil`（数据库 JSONB 字段），检查发生在 `evaluateIdleSymbol` L2516-2523。

**冷却期是按 strategy_instance（策略+底层标的）维度存储的。**

### 5. 退出时发生了什么

当 TSLA260318C402500 通过阶梯锁利退出时（L4321-4326）：

```
1. activeEntries.delete('TSLA.US')     ← 从分组活跃列表中移除
2. state → CLOSING（写入 cooldownUntil）
3. 卖单成交后 → processClosingPosition → state → IDLE
4. IDLE 时 context 保留了 cooldownUntil
```

### 6. 下一个周期发生了什么

约30秒后，下一个策略周期运行：

```
Phase A: 状态分类
  → TSLA.US 已变回 IDLE

Phase B: evaluateIdleSymbol('TSLA.US')
  → L2516: 检查 cooldownUntil
  → 如果 dailyTradeCount=1（第二笔），cooldown=1分钟
  → 但 TSLA 的 strategy_instance 刚被写入 cooldownUntil
```

## 根本问题

**问题不在分组机制，而在冷却期的执行窗口。**

从日志看，TSLA 在 22:03:46 卖出，22:04:16 买入，间隔 30 秒。按 0DTE 规则，第二笔（dailyTradeCount=1）应有 >= 1 分钟冷却。

可能原因：

1. **CLOSING → IDLE 状态转换耗时**：卖出信号发出后，需要等券商确认成交，`processClosingPosition` 才把状态转回 IDLE。如果这个过程刚好在冷却期内完成，IDLE 状态的 cooldownUntil 可能已经过期（实际退出可能更早，冷却期从退出时算起）。

2. **冷却期可能只有1分钟**：如果 `dailyTradeCount=1` 且配置的 `reentryCooldownMinutes` 未设置或 =1，冷却正好1分钟。但状态转换可能在50秒时完成，剩余10秒不足以阻止下一周期。

3. **更关键的设计问题 — 退出信号和入场信号不协同**：
   - 阶梯锁利说"TSLA 动能衰竭"
   - 牛市价差评分模块依然给出 9.3 高分
   - 两个模块独立运行，退出上下文没有传递给入场评分

## 建议修复方向

### 方案：退出后同底层标的强制冷却（与分组踢出联动）

在 `activeEntries.delete(symbol)` 时，不是立即删除，而是标记一个退出时间戳。在 `evaluateIdleSymbol` 的分组检查阶段，除了检查 HOLDING 的同组标的，还检查**最近退出的同组标的**。

具体实现点位:

1. **`getCrossSymbolState` 扩展**（L146-153）：新增 `recentExits: Map<string, number>`
2. **所有 `activeEntries.delete(symbol)` 处**（8处）：改为同时写入 `recentExits.set(symbol, Date.now())`
3. **`evaluateIdleSymbol` 分组检查**（L2540-2554）：新增对 `recentExits` 的检查，同组标的退出后 N 分钟内不参与竞价
4. **冷却时间**：复用 `computeCooldownUntil` 的逻辑，或使用固定值（如5分钟）
5. **新交易日重置**（L505）：`crossSymbolState.delete()` 已自动清除

### 与现有冷却机制的关系

| 机制 | 维度 | 作用 | 问题 |
|---|---|---|---|
| `cooldownUntil` | per symbol instance (DB) | 阻止同一 symbol 重入 | 同底层不同合约不阻止 |
| `activeEntries` | per symbol (内存) | 阻止同组其他标的入场 | 退出后立即删除 |
| **新增 `recentExits`** | per symbol (内存) | 退出后阻止同组标的入场 | 补上漏洞 |

**注意**: 对于期权策略，symbol pool 用底层标的名，所以 cooldownUntil 实际上已经绑定到底层标的维度（TSLA.US 只有一个 instance），问题可能更细微 — 需要确认卖出 TSLA260318C402500 后，TSLA.US 这个 instance 的 cooldownUntil 是否被正确检查。如果期权合约名和 instance symbol 不同，冷却就绕过了。
