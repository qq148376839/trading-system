## 诊断一: 评分机制是个瞎子 — 有 K 线数据但当它不存在

### 结论先行

**`calculateMomentum()` 拿到了完整的 OHLCV K 线数据，却只看了 close 价格。** 这就像拿到了一本完整的财报，但只看了净利润那一行。

### 代码实锤

`option-recommendation.service.ts:645-686`:

```typescript
private calculateMomentum(data: CandlestickData[]): number {
  // data 包含: open, high, low, close, volume, timestamp
  // 实际使用: 仅 close + high/low(ATR用)
  // 完全忽略: volume ❌

  const changes: number[] = [];
  for (let i = 1; i < Math.min(data.length, 10); i++) {
    // 只取最近 10 根中的 close-to-close 变化率
    const change = (data[i].close - data[i - 1].close) / data[i - 1].close;
    changes.push(change);
  }
  const avgChange = changes.reduce((sum, c) => sum + c, 0) / changes.length;
  // ATR 归一化后输出
}
```

### 七宗罪

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| **K1** | **Volume 完全不参与评分** | CRITICAL | K 线里有 volume 字段，VWAP 计算用了它，但 momentum 评分完全忽略。一根成交量 100 的 K 线和成交量 100 万的 K 线，对方向判断的权重完全相同。**你无法区分真突破和假突破。** |
| **K2** | **240 根 K 线只用了 20 根** | HIGH | `getIntradayVWAP()` 从 LongPort 拉了 240 根 1 分钟 K 线（4 小时数据），但传给 momentum 的只有 `slice(-20)`，再被 `Math.min(data.length, 10)` 砍到**只看最后 10 根**。90% 以上的数据被扔掉了。 |
| **K3** | **最低 5 根就敢出评分** | HIGH | `if (!data \|\| data.length < 5) return 0;` — 5 根 1 分钟 K 线 = 5 分钟数据就敢给出方向判断。4 个 close-to-close 变化率取平均 = **统计学上的笑话**。 |
| **K4** | **标的 K 线无数据质量过滤** | MEDIUM | SPX hourly 数据过了 `intradayDataFilterService.filterData()`（Z-score 异常检测 + volume 校验 + EMA 平滑），但标的 1 分钟 K 线**直接裸传**给 momentum 计算。同一套评分系统，SPX 数据洗过脸，标的数据满脸泥。 |
| **K5** | **只有 1 分钟一个时间框架** | MEDIUM | 整个方向判断只有 1 分钟 K 线。没有 5 分钟确认、没有 15 分钟趋势。**单一时间框架的动量信号在 0DTE 高 gamma 环境下就是纯噪声。** |
| **K6** | **Open/High/Low 不参与趋势判断** | MEDIUM | K 线有 OHLC 四个价格，但趋势判断只用 Close。没有上下影线分析、没有吞没形态检测、没有 pin bar 识别。白白丢弃了 75% 的价格信息。 |
| **K7** | **静默降级到过期缓存** | LOW | `getIntradayVWAP()` API 失败时用 5 分钟前的过期缓存，但不告知评分系统。评分系统以为自己在用实时数据，实际在用 5 分钟前的快照做决策。 |

### 数据利用率全景

```
LongPort API 返回:
  240 根 K 线 × 6 字段(O/H/L/C/V/T) = 1440 个数据点

实际使用:
  10 根 K 线 × 1 字段(Close) = 10 个数据点
  + ATR 用了 H/L（但只是归一化系数）

利用率: 10/1440 = 0.7%
```

**说直白了: 这个评分系统对标的走势的理解，相当于一个交易员闭着眼睛、堵着耳朵，每分钟摸一下价格，摸 10 次，然后告诉你方向。**

### 应该怎么做

1. **Volume-weighted momentum**: 成交量大的 K 线变动权重应该更高
2. **多时间框架**: 至少用 1min + 5min 双框架确认，不一致时降低信号置信度
3. **K 线形态**: 利用 OHLC 关系检测关键反转形态（吞没、pin bar、十字星）
4. **最低数据要求**: 从 5 根提高到至少 20 根，且有效 volume 占比 > 70%
5. **数据质量对齐**: 标的 K 线也要过 `intradayDataFilterService`

## 诊断二: PnL 记录是个精神分裂患者 — 两套账本谁也不认谁

### 结论先行

**系统有两条 PnL 计算路径，一条用估算值，一条用实际成交价，但它们之间没有任何回写或校正机制。** 前端分析页看的是估算值，实际盈亏可能完全不同 — 甚至盈亏方向都能翻转。

### 这不是新发现 — 是 3 月 28 日就记录的 P0 bug

`docs/analysis/260328-交易分析模块深度诊断.md` 已经明确记录:

> **P0-致命: 信号 PnL 基于决策时价格，非实际成交价**
> - SPY P640 3/27: 信号显示 **+$10.59 盈利**，实际成交 **-$15.00 亏损**，偏差 $25.59，**盈亏方向翻转**
> - 平均绝对偏差: $8.95/笔
> - 信号胜率显示 48%，实际只有 44%，**高估 4 个百分点**

**问: 这个 P0 bug 过了 6 天修了吗?**
**答: 没有。**

### 两条 PnL 路径的精神分裂

```
路径 A: 信号 PnL（前端分析页读取的）
=========================================
1. processHoldingPosition() 检测到退出条件
2. calculatePnL(positionCtx) ← 用当前市场中间价估算
3. logSellSignal(metadata={netPnL: 估算值}) ← 写入 strategy_signals 表
4. executeSellOrder() ← 提交卖出订单
5. 订单成交，实际价格可能不同
6. ❌ 无人回写 strategy_signals.metadata.netPnL

路径 B: dailyRealizedPnL（熔断器读取的）
=========================================
1. 卖出订单成交，broker 返回 avgPrice（实际成交价）
2. tradePnL = (avgPrice - entryPrice) × quantity × multiplier - fees
3. dailyRealizedPnL += tradePnL ← 写入 strategy_instances.context
4. ✅ 基于实际成交价，相对准确

路径 C: auto_trades.pnl（Dashboard 读取的）
=========================================
1. recordTrade() 用 broker 实际 avgPrice 计算
2. pnl = (avgPrice - openPrice) × qty × multiplier ← 已修复乘数 bug
3. ✅ 基于实际成交价
```

### 问题根源

```typescript
// strategy-scheduler.service.ts:4595-4607
const sellSignalId = await this.logSellSignal(
  strategyId,
  effectiveSymbol,
  currentPrice,     // ← 当前市场中间价，不是成交价
  `option_exit:${action}`,
  {
    netPnL: pnl.netPnL,          // ← 基于 currentPrice 的估算值
    netPnLPercent: pnl.grossPnLPercent,  // ← 字段名还是错的(gross 标为 net)
    totalFees: pnl.totalFees,
  }
);
// ↑ 信号写完就不管了。后面订单成交后，没有任何代码回来更新这条信号。
```

### 为什么"再算一次"必然出错

**第一性原理**: PnL 的唯一真实来源是券商的成交确认（Execution Report）。任何其他计算方式都是近似值，近似值必有误差。

当前系统的做法是:
1. 用市场中间价**估算** PnL → 记录到信号表
2. 用 broker 成交价**实际** PnL → 记录到 strategy_instances
3. 两个数字不一致，没有校正机制

**正确做法只有一种**: PnL 有且只有一个来源 — broker 成交确认。所有下游系统（信号分析、Dashboard、熔断器）都应该从这个唯一来源读取。

### 影响范围

| 消费者 | 读取来源 | 数据质量 |
|--------|---------|---------|
| 前端分析页 | `strategy_signals.metadata.netPnL` | **估算值，可能方向翻转** |
| 前端盈亏百分比 | `strategy_signals.metadata.netPnLPercent` | **字段名是 net 但值是 gross** |
| 日内熔断器 | `strategy_instances.context.dailyRealizedPnL` | 基于实际成交，相对准确 |
| Dashboard trades | `auto_trades.pnl` | 基于实际成交（乘数已修复） |
| 交易日报 | 混合来源 | **取决于查哪个表，数字可能不同** |

### 应该怎么做

**唯一正确方案: 订单成交后，用 broker 实际成交数据回写信号表。**

```
卖出订单成交（broker 返回 avgPrice + fees）
    ↓
计算实际 PnL = (avgPrice - entryPrice) × qty × multiplier - totalFees
    ↓
UPDATE strategy_signals SET metadata = metadata ||
  '{"actualNetPnL": X, "actualPnLPercent": Y, "executedPrice": Z}'
WHERE id = sellSignalId
    ↓
所有下游统一读取 actualNetPnL，不再读 netPnL
```

不要再搞"两套账本"。一个交易的 PnL 只应该有一个数字，那就是 broker 告诉你的数字。

## 诊断三: 入场时机是个追星族 — 永远在巅峰时到场

### 结论先行

**系统不是在"预判趋势"，是在"确认趋势已经发生"。** 当 momentum 评分达到入场阈值时，价格移动已经发生了，你买的是尾巴，不是头部。

### 信号管道不是瓶颈

信号生成 → 订单提交的延迟只有 1-2 秒:
```
T+0ms:     调度器周期启动
T+100ms:   Phase A 状态检查
T+200ms:   Phase B 信号评估（generateSignal）
T+400ms:   Phase C 竞价筛选
T+500ms:   Phase D 订单执行
T+1500ms:  broker API 返回 orderId
```

**瓶颈不在管道速度，在评分系统本身是一个滞后指标。**

### calculateMomentum 是一个滞后 10 分钟的指标

```typescript
// option-recommendation.service.ts:668-672
for (let i = 1; i < Math.min(data.length, 10); i++) {
  const change = (data[i].close - data[i - 1].close) / data[i - 1].close;
  changes.push(change);
}
const avgChange = changes.reduce((sum, c) => sum + c, 0) / changes.length;
```

这段代码做的事: **取最近 10 根 1 分钟 K 线的 close-to-close 变化率，求平均。**

第一性原理分析:

1. **为什么平均变化率会达到入场阈值？** 因为过去 10 分钟内，大多数 K 线的收盘价都在朝同一方向变化。
2. **什么时候大多数 K 线都朝同一方向变化？** 当一波趋势性行情已经**持续了至少 7-8 分钟**（10 根中 70-80% 同向）。
3. **一波持续 7-8 分钟的趋势行情，在 0DTE 环境下通常意味着什么？** 行情可能已经接近或到达短期极值，即将均值回归。

**你的 momentum 评分在说"过去 10 分钟市场一直在跌"的时候，市场大概率正在准备反弹。**

### 实盘数据完美印证了这个滞后

4 月 2 日交易数据:

| 交易 | 入场得分 | 入场后 | 结果 |
|------|---------|--------|------|
| QQQ P574 | -17.0（强烈看跌） | 3 分钟触发 -22.8% 止损 | 价格反弹 |
| TSLA P365 #1 | -11.4（看跌） | 1 分钟 trailing stop | 短暂对，但不持续 |
| TSLA P362.5 | -22.3（极强看跌） | 1 分钟触发 -23.7% 止损 | 价格反弹 |
| TSLA P365 #2 | -12.5（看跌） | 1.5 分钟触发 -24.2% 止损 | 价格反弹 |
| NVDA C177.5 | +13.2（看涨） | 28 秒 trailing stop | 短暂对，但不持续 |
| SPY P655 | -11.0（看跌） | 1.5 分钟触发 -22.4% 止损 | 价格反弹 |

**模式一目了然**: 评分越极端（-17, -22），入场后反转越快。这不是方向判断错了 — 方向是对的（过去确实在跌），但**时机错了**（跌到头了）。

### 时间轴还原

```
T-10min: 标的开始下跌
T-8min:  1min K线连续收跌，momentum 开始转负
T-5min:  momentum 累积到 -5~-8，低于入场阈值(±5 aggressive)
T-3min:  momentum 达到 -10~-15，满足入场条件
T-2min:  竞价通过，FastMo gate 确认方向一致
T-1min:  订单提交，买入 Put
T+0:     入场！此时标的已经跌了 10 分钟
T+1min:  标的开始反弹（均值回归）
T+2min:  Put 亏损 -15%~-20%
T+3min:  触发止损退出

如果在 T-5min 入场（momentum 刚转负但未达阈值）:
  - Put 价格更便宜（标的还没跌完）
  - 后续 5 分钟的下跌 = 利润空间
  - 即使 T+0 反弹，已经有 5 分钟的利润缓冲
```

**用户的观察是 100% 正确的: 提前 3-5 分钟入场，意味着在 momentum 构建阶段而非峰值阶段入场，本质上是从"趋势确认后追入"变为"趋势早期跟进"。**

### FastMomentumService — 有实时数据但只当门卫

`fast-momentum.service.ts` 有一个实时报价的 `feedQuotes()`，每个调度周期都喂入最新价格，做线性回归。

**但它只是一个 GATE（通过/拒绝），不是信号生成器。**

```
fastMomentumService.checkGate(symbol, direction)
→ 方向一致? ✓
→ 没有减速? ✓
→ 没有异常加速? ✓
→ pass: true ← 放行，但不影响入场时机
```

**讽刺的是**: FastMo 用的是真正的实时数据（5秒级别），但它只做 pass/fail 决策。而决定入场时机的 `calculateMomentum()` 用的是 1 分钟级别的滞后数据。**最敏感的数据给了最不重要的角色。**

### 为什么 "提前 3-5 分钟" 能提高 30%+ 盈利

数学证明:

假设一个典型的 0DTE 方向性行情周期:
```
Phase 1 (T-10 → T-5): 趋势启动期
  - 标的匀速移动 0.3%（Put 价值变化约 +15%）
  - momentum 评分: -3 → -8（未达阈值 ±5×VIX=6.5）

Phase 2 (T-5 → T-0): 趋势加速期
  - 标的继续移动 0.5%（Put 价值变化约 +25%）
  - momentum 评分: -8 → -15（超过阈值，触发入场）

Phase 3 (T+0 → T+5): 均值回归期
  - 标的回撤 0.4%（Put 价值变化约 -20%）
  - 已入场，吃到 -20% 亏损
```

**在 T-5 入场**:
- 吃到 Phase 2 的 +25% 利润
- Phase 3 回撤 -20%
- 净结果: **+5%**

**在 T+0 入场**（当前系统行为）:
- 完全错过 Phase 1 和 Phase 2
- 只吃到 Phase 3 的 -20%
- 净结果: **-20%**

**差异: 25 个百分点。** 这就是为什么"提前 3-5 分钟"能将盈利提高 30%+。

### 应该怎么做

| 方案 | 思路 | 复杂度 | 预期效果 |
|------|------|--------|---------|
| **A: 降低入场阈值** | 从 ±5(aggressive) 降到 ±3，更早触发 | LOW | 提前 ~2 分钟，但可能增加噪声交易 |
| **B: 用 momentum 变化率替代 momentum** | 检测 "momentum 正在加速" 而非 "momentum 已经很强" | MEDIUM | 从趋势确认型变为趋势预判型 |
| **C: FastMo 升级为信号生成器** | 用实时 5 秒报价的线性回归斜率做 early signal | MEDIUM | 最直接利用已有的实时数据 |
| **D: K 线形态预判** | 检测大阳/大阴线 + 放量突破作为 early entry trigger | HIGH | 最接近交易员直觉 |
| **E: 双触发机制** | 低阈值 early entry（半仓）+ 高阈值 confirmation entry（加仓） | HIGH | 兼顾早入场和确认 |

**推荐**: 先做 A + C（低成本、高收益），验证效果后再做 B/D。

## 追加诊断: 退出精度同样被 5 秒轮询毒害 — 阶梯锁利底线形同虚设

> 补充于 2026-04-03，行情订阅架构分析后发现

三个诊断聚焦于入场问题，但退出侧有同样严重的精度缺陷:

### 实际案例

```
Strategy 10 阶梯锁利配置: threshold=10, floor=7
实际退出日志: "峰值12.8%踩上10%台阶，底线7%，回落至2.2%触发"
```

时间线:
```
检查 T+0s:  PnL = 12.8% → 记录峰值, 底线锁 7%
            (中间 ≤5s: 0DTE gamma 加速, 期权跌穿 7% 底线)
检查 T+5s:  PnL = 2.2% → 终于"看见" → TRAILING_STOP
```

**漏损 4.8%**（7% 底线 - 2.2% 实际退出），仓位 $500 即多亏 $24。

### 根因

`option-dynamic-exit.service.ts:490` 的检查逻辑完全正确:
```typescript
if (pnl.grossPnLPercent <= lockedFloor) → TRAILING_STOP
```
但它只在调度周期（5s）触发 — 5 秒对 0DTE 高 gamma 来说是永恒。

### 这是第四把刀

```
K线分析不足 (诊断一) → 信号延迟 → 入场已是尾巴
PnL 多源分裂 (诊断二) → 分析失真 → 越调越差
入场滞后 (诊断三) → 追星族入场 → 入场即反转
退出精度不足 (诊断四) → 底线形同虚设 → 利润回吐失控 ← NEW
```

诊断三和诊断四的根因相同: **5 秒轮询对 0DTE 高 gamma 来说太慢**。行情订阅架构升级（WebSocket 推送）同时解决入场和退出两个维度。

详细分析: `docs/analysis/260403-行情订阅架构升级分析.md`

### SQL 查询 1: 3 月至今所有 BUY 信号及其前 5 分钟的日志

```sql
-- 在 NAS 生产数据库执行
-- 1) 拉取所有 EXECUTED 的 BUY 信号
SELECT
  ss.id as signal_id,
  ss.symbol,
  ss.price,
  ss.reason,
  ss.metadata,
  ss.created_at as signal_time,

WHERE ss.strategy_id IN (
  SELECT id FROM strategies WHERE type LIKE '%OPTION%'

  AND ss.created_at >= '2026-03-01'
  AND ss.signal_type = 'BUY'
  AND ss.status = 'EXECUTED'
ORDER BY ss.created_at;
```

### SQL 查询 2: 每笔交易实际 PnL vs 信号估算 PnL

```sql
-- 对比两条 PnL 路径
SELECT
  buy_sig.symbol,
  buy_sig.created_at as entry_time,
  sell_sig.created_at as exit_time,
  (sell_sig.metadata->>'netPnL')::numeric as signal_estimated_pnl,
  at.pnl as actual_trade_pnl,
  ABS((sell_sig.metadata->>'netPnL')::numeric - at.pnl) as pnl_deviation
FROM strategy_signals buy_sig
JOIN execution_orders buy_eo ON buy_sig.id = buy_eo.signal_id AND buy_eo.side = 'BUY'
JOIN strategy_signals sell_sig ON sell_sig.strategy_id = buy_sig.strategy_id
  AND sell_sig.symbol = buy_sig.symbol
  AND sell_sig.signal_type = 'SELL'
  AND sell_sig.created_at > buy_sig.created_at
  AND sell_sig.created_at < buy_sig.created_at + INTERVAL '4 hours'
LEFT JOIN auto_trades at ON at.symbol = buy_sig.symbol
  AND at.created_at BETWEEN buy_sig.created_at AND sell_sig.created_at + INTERVAL '5 minutes'
WHERE buy_sig.strategy_id IN (
  SELECT id FROM strategies WHERE type LIKE '%OPTION%'
)
  AND buy_sig.created_at >= '2026-03-01'
  AND buy_sig.signal_type = 'BUY'
  AND buy_sig.status = 'EXECUTED'
ORDER BY buy_sig.created_at;
```

### SQL 查询 3: 每笔入场前 5 分钟的 FILTERED 信号（评分变化轨迹）

```sql
-- 每笔 EXECUTED 信号前 5 分钟内，同标的的所有信号（含 FILTERED）
SELECT
  exec_sig.id as executed_signal_id,
  exec_sig.created_at as entry_time,
  all_sig.id as preceding_signal_id,
  all_sig.status,
  all_sig.created_at as preceding_time,
  EXTRACT(EPOCH FROM (exec_sig.created_at - all_sig.created_at)) / 60.0 as minutes_before,
  (all_sig.metadata->>'finalScore')::numeric as score_at_time,
  all_sig.reason as filter_reason
FROM strategy_signals exec_sig
JOIN strategy_signals all_sig ON
  all_sig.strategy_id = exec_sig.strategy_id
  AND all_sig.symbol = exec_sig.symbol
  AND all_sig.signal_type = 'BUY'
  AND all_sig.created_at BETWEEN exec_sig.created_at - INTERVAL '5 minutes' AND exec_sig.created_at
WHERE exec_sig.strategy_id IN (
  SELECT id FROM strategies WHERE type LIKE '%OPTION%'
)
  AND exec_sig.created_at >= '2026-03-01'
  AND exec_sig.signal_type = 'BUY'
  AND exec_sig.status = 'EXECUTED'
ORDER BY exec_sig.created_at, all_sig.created_at;
```

> 注: 查询 3 的前提是系统每个周期（5秒）都会产生 FILTERED/REJECTED 信号记录。如果只记录 EXECUTED 信号，则需要从系统日志中提取评分变化。

## 总结: 四把刀

| 问题 | 一句话 | 影响 |
|------|--------|------|
| **K 线分析缺失** | 有 1440 个数据点只用了 10 个 | 方向判断 = 盲人摸象 |
| **PnL 精神分裂** | 两套账本，前端看的是估算值 | 分析基础全是错的 |
| **入场滞后** | 等 momentum 峰值才入场 = 追顶 | 盈亏比 0.03:1 的直接原因 |
| **退出精度不足** | 底线 7% 实际退出 2.2%，漏损 4.8% | 利润回吐失控 |

**四个问题环环相扣。K 线分析不足导致方向信号质量差，方向信号质量差导致需要高阈值确认，高阈值确认导致入场滞后，入场滞后导致盈亏比崩塌，退出时 5 秒间隔导致底线保护失效，PnL 估算错误导致分析页误导性数据，误导性数据导致调优方向错误，形成恶性循环。**

打破循环的切入点: **先修 PnL 数据源（让分析数据可信），再升级行情订阅架构（同时解决入场滞后 + 退出精度），然后补 K 线分析（volume-weighted + 多时间框架 + 实时推送数据）。**
