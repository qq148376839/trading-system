**日期**: 2026-03-17
**严重程度**: 高危
**影响范围**: 多策略共享同一券商账户时的持仓状态同步

---

## 问题概述

## 根因分析

### 核心架构缺陷：Per-Strategy 状态隔离 vs 共享券商账户

```
┌─────────────────────────────────────────────────────────┐
│  strategy_instances 表 (数据库)                           │
├───────────┬──────────┬───────────────────────┬──────────┤
│ strategy  │ symbol   │ tradedSymbol          │ state    │
│    10     │ SPY.US   │ SPY260316C673000.US   │ IDLE ✓   │
│    12     │ SPY.US   │ SPY260316C673000.US   │ HOLDING ✗│ ← 过期数据
│    13     │ SPY.US   │ SPY260316C673000.US   │ HOLDING ✗│ ← 过期数据
└───────────┴──────────┴───────────────────────┴──────────┘

┌──────────────────────────────────────────┐
│ 券商账户 (实际持仓)                        │
│ SPY260316C673000.US: 0 股               │
│ (策略10已在13:58全部卖出)                  │
└──────────────────────────────────────────┘
```

### 5个根因

| # | 根因 | 文件位置 |
|---|------|---------|
| 1 | **trackPendingOrders() 只更新卖出策略自身的状态** — 策略10卖出后仅更新 `(10, SPY.US)` → IDLE，不通知策略12/13 | strategy-scheduler.service.ts:1190-1302 |
| 2 | **0DTE看门狗用数据库而非券商数据做持仓检测** — `WHERE current_state = 'HOLDING'` 查的是 strategy_instances 表 | 0dte-watchdog.service.ts:118-126 |
| 3 | **skipPositionValidation 跳过了券商实际持仓验证** — 看门狗用此标志绕过验证，导致对不存在的持仓发起卖单 | basic-execution.service.ts:621-628 |
| 4 | **多策略买入同一合约但持仓是共享的** — 3个策略各自记录持有16股，但券商账户只有16股实际持仓 | 架构设计问题 |
| 5 | **看门狗平仓失败后不更新策略状态** — `RejectedStatus` 订单完成后，看门狗认为"平仓成功"但实际被拒 | 0dte-watchdog.service.ts:237 |

### 关键代码路径

**订单完成回调** (`strategy-scheduler.service.ts:1190-1211`):
```sql
SELECT symbol, context FROM strategy_instances
WHERE strategy_id = $1  -- ← 只查询卖出策略自身
  AND (context->>'tradedSymbol' = $2 OR context->'intent'->>'symbol' = $2)
```

**0DTE看门狗扫描** (`0dte-watchdog.service.ts:118-126`):
```sql
SELECT si.strategy_id, si.symbol, si.context
FROM strategy_instances si
WHERE si.current_state = 'HOLDING'  -- ← 查所有策略的HOLDING状态
  AND si.context->>'tradedSymbol' IS NOT NULL
```

## 修复建议

### 紧急修复（P0）

1. **订单完成时同步清除同合约其他策略的持仓状态**
   - 在 `trackPendingOrders()` 中，当一个策略成功卖出某合约后，查询所有策略中 `tradedSymbol` 相同的 HOLDING 记录并清除
   - 使用 POSITION_EXIT_CLEANUP 清理机制

2. **0DTE看门狗在强制平仓前验证券商实际持仓**
   - 即使 `skipPositionValidation=true`，也应先调用 `calculateAvailablePosition()` 确认实际持仓>0
   - 若实际持仓=0，直接将数据库状态更新为 IDLE（而非发起卖单）

3. **看门狗增加平仓失败后的状态修正**
   - `RejectedStatus` 不应被标记为"平仓成功"
   - 连续N次被拒后，应检查券商实际持仓并修正数据库

### 架构优化（P1）

4. **引入定时持仓同步任务**
   - 定时（如每5分钟）拉取券商实际持仓
   - 对比数据库 HOLDING 状态，自动修正不一致

5. **同合约多策略去重**
   - 如果多个策略买入同一合约，应在数据库层面合并持仓记录
   - 或限制同一合约只能被一个策略操作

6. **PENDING信号清理机制**
   - 对已平仓标的的 PENDING 信号批量标记为 CANCELLED
   - 限制同标的同方向信号的密度
