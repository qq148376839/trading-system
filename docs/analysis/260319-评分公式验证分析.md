## 四、最终评分验证

### 公式结构（源码 L169-172）

```
finalScore = marketScore * 0.2 + intradayScore * 0.6 + timeWindowAdjustment * 0.2
```

### SPY.US 验证

已知:
- marketScore = -38.8 (精确约 -38.82)
- intradayScore = 1.6 (精确约 1.61)
- finalScore 日志 = -2.79

**反推 timeWindowAdjustment:**

```
-2.79 = -38.82 * 0.2 + 1.61 * 0.6 + timeWindow * 0.2
-2.79 = -7.764 + 0.966 + timeWindow * 0.2
-2.79 = -6.798 + timeWindow * 0.2
timeWindow * 0.2 = -2.79 + 6.798 = 4.008
timeWindow = 4.008 / 0.2 = 20.04
```

### 时间窗口验证（源码 L635-667）

13:51 UTC, 使用 EST (UTC-5):
- etHour = (13 - 5 + 24) % 24 = 8 ... 等等，这需要更仔细计算。
- UTC 13:51 -> EST = 13 - 5 = 8:51?

**但 3 月 19 日美国已进入夏令时 (EDT = UTC-4)！**

实际美东时间应该是 13:51 - 4 = **9:51 ET (EDT)**

而代码使用硬编码 `UTC - 5`，计算得到:
- etHour = (13 - 5 + 24) % 24 = 8
- etMinutes = 8 * 60 + 51 = 531

按源码逻辑:
- marketOpen = 570 (9:30)
- 531 < 570，不在任何 if 分支中

**但 531 < 570 意味着被认为是"开盘前"——所有 if 分支都不匹配，adjustment = 0！**

然而实际 timeWindow = +20 且反推也证实 timeWindow = +20.04。

**重新核实**: 等一下，让我重新看 if 条件:

```javascript
if (etMinutes < marketOpen + 60) {        // < 630
    adjustment = 20;
```

531 < 630 确实成立！所以 adjustment = 20。

**但逻辑上存在问题**: 代码认为 `etMinutes < 630` 就给 +20 分，没有检查 `etMinutes >= marketOpen (570)`。这意味着：
- 如果 etMinutes = 531（代码认为的"盘前"），也会拿到 +20
- 实际上开盘前时间不应该获得"开盘黄金时段"的加分

不过在实际场景中，策略只在交易时段运行（由 `isWithinTradeWindow` 控制），所以这个边界条件不太可能被真正触发为问题。但仍然是一个逻辑漏洞。

**实际验证结果**:
- 代码计算 etMinutes = 531
- 531 < 630 (marketOpen + 60) -> adjustment = +20
- 反推得 timeWindow = 20.04，**与代码行为匹配**

### 动态阈值验证

源码 L180-182:
```javascript
const baseThreshold = 15;
const entryTimeFactor = this.get0DTETimeThresholdFactor();
const dynamicThreshold = baseThreshold * entryTimeFactor;
```

`get0DTETimeThresholdFactor()` 使用 `Intl.DateTimeFormat('America/New_York')` 正确处理夏令时:
- 真实 ET = 9:51 -> etMinutes = 591
- 591 <= 720 (midDay = 12:00) -> return 1.0

dynamicThreshold = 15 * 1.0 = **15**

finalScore = -2.79, 在 [-15, 15] 区间内 -> **HOLD**，符合日志。

### 最终评分结论

- **finalScore 算术正确**，各层权重 (0.2, 0.6, 0.2) 应用正确。
- timeWindowAdjustment = +20 的结果正确（虽然路径有瑕疵，见下方问题分析）。
- 动态阈值 15 正确，HOLD 决策正确。

## 六、发现的问题

### P1: calculateTimeWindowAdjustment 硬编码 EST，夏令时偏移 1 小时

**位置**: `option-recommendation.service.ts` L641-642

```javascript
// 简化：使用EST
const etHour = (hour - 5 + 24) % 24;
```

**问题**: 3 月第二个周日到 11 月第一个周日期间美国使用 EDT (UTC-4)。代码始终使用 EST (UTC-5)，导致**计算的美东时间比实际慢 1 小时**。

**影响**:
- 真实 9:51 ET 被计算为 8:51 ET
- 真实 10:30 ET 被计算为 9:30 ET（开盘时间）
- 真实 10:31 ET 被计算为 9:31 ET（仍在"黄金 1 小时"区间）
- 实际效果：夏令时期间，"开盘黄金 1 小时" +20 分的窗口被延长到开盘后 2 小时

**对本次数据的影响**:
- 真实 ET 9:51, 代码计算为 8:51 (531 分钟)
- 两种情况都命中 `< marketOpen + 60 (630)` 分支，结果相同 = +20
- **本次数据点恰好不受影响**，但在 10:30-11:30 ET 区间会产生错误评分

**对比**: `get0DTETimeThresholdFactor()` 使用 `Intl.DateTimeFormat('America/New_York')` **正确处理夏令时**，同一文件内两个时间函数使用了不同的时区处理方式，存在不一致。

**风险评估**: 中等。夏令时期间每天 10:30-11:30 ET 这 1 小时内，timeWindowAdjustment 会多给 +30 分（应该是 -10 但给了 +20），可能导致错误入场。

### P2: calculateTimeWindowAdjustment 缺少下界检查

**位置**: `option-recommendation.service.ts` L652

```javascript
if (etMinutes < marketOpen + 60) {
    adjustment = 20;  // 没有检查 etMinutes >= marketOpen
}
```

如果因为 P1 或其他原因导致 `etMinutes` 处于盘前（<570），仍会得到 +20 的加分。应该增加 `etMinutes >= marketOpen` 的下界检查。

**风险评估**: 低。策略有独立的交易时间窗口检查（`isWithinTradeWindow`），盘前不会真正产生交易信号。但评分本身不准确可能影响监控面板的参考价值。

### P3: 10:30-11:30 ET 时间窗口存在评分真空

**位置**: `option-recommendation.service.ts` L652-664

```javascript
if (etMinutes < marketOpen + 60) {          // < 630  -> +20
} else if (etMinutes > forceCloseTime) {     // > 930  -> -50
} else if (etMinutes > forceCloseTime - 60) { // > 870 -> -30
} else if (etMinutes > marketOpen + 120) {   // > 690  -> -10
}
```

分钟在 [630, 690] 之间（即 10:30-11:30 ET）时，**没有任何分支命中**，adjustment = 0。

这不算 bug（0 分是"中性"），但存在一个阶梯断崖：
- 10:29 ET (629分钟) -> +20
- 10:30 ET (630分钟) -> 0
- 11:30 ET (690分钟) -> 0
- 11:31 ET (691分钟) -> -10

从 +20 直接跳到 0，再从 0 跳到 -10，形成两个断崖。回测版本 (`option-backtest.service.ts` L317-356) 使用了**平滑曲线版**来消除断崖，但实盘仍使用阶梯版。

**风险评估**: 低。主要影响是 10:30 时刻 finalScore 突然下降 4 分（20*0.2=4），可能导致恰好在阈值附近的信号在 10:29 触发但 10:30 不触发，或反之。

## 七、总结

### 数学准确性

| 评分层 | 验证结果 | 说明 |
|--------|---------|------|
| 大盘评分 | **正确** | 2 个时间点均匹配，所有组件权重、VIX 阶梯、温度逻辑均正确 |
| 分时评分 (SPY) | **正确** | 精确值 1.610, 日志 1.6, 匹配 |
| 分时评分 (NVDA) | **正确** | 精确值 -1.450, 日志 -1.4, 匹配 (IEEE 754 取整行为) |
| 分时评分 (TSLA) | **正确** | 精确值 -1.480, 日志 -1.5, 匹配 |
| 分时评分 (QQQ) | **正确** | 精确值 2.855, 日志 2.9, 匹配 |
| 最终评分 (SPY) | **正确** | 反推 timeWindow=20.04, 与代码行为一致 |
| 动态阈值 | **正确** | 15 * 1.0 = 15, HOLD 决策正确 |
| Regime Detection | **正确** | MEAN_REVERSION HIGH, 触发条件逐一验证通过 |

### 发现的问题

| 优先级 | 问题 | 影响 |
|--------|------|------|
| **P1** | `calculateTimeWindowAdjustment` 硬编码 EST，夏令时偏移 1 小时 | 夏令时期间 10:30-11:30 ET 评分错误 (+20 应为 0 或 -10) |
| P2 | 同函数缺少盘前下界检查 | 理论上盘前也会得到 +20 (被 isWithinTradeWindow 保护) |
| P3 | 实盘 timeWindow 使用阶梯函数，回测使用平滑曲线，存在不一致 | 回测与实盘在阶梯边界处行为不同 |

### 关键观察

1. **BTC 小时 K 动量快速恶化**：从 -0.5 -> -4.6 -> -10.7，在分时评分中贡献了持续的负向压力（-1.6 分）。这是分时评分无法翻正的重要原因之一。

2. **VIX 25.9 触发 -20 惩罚是当前大盘评分极度负面的主因**：SPX 趋势 -13.8 + VIX -20 = -33.8，这两项就占了大盘评分的 87%。如果 VIX 降到 25 以下，大盘评分会瞬间从 -38.8 跳到 -18.8（+20 分跳跃），这是 VIX 阶梯函数的固有问题。

3. **Regime Detection 正确识别了均值回归机会**：大盘极端看跌（-38.8）但日内微弱看涨（+1.6），分歧度 1.05 远超阈值 0.3。如果策略启用了 smartReverse，会翻转方向做 PUT 而非 CALL。但由于 finalScore = -2.79 未超过阈值 15，无论是否翻转都不会实际入场，这是正确的保守行为。

4. **权重结构合理性**：大盘 20% + 分时 60% + 时间 20% 的权重分配意味着日内动量是主导因素。在当前场景下，大盘 -38.8 * 0.2 = -7.76 的负面贡献被时间窗口 +20 * 0.2 = +4 和分时 +1.6 * 0.6 = +0.96 部分对冲，最终得分 -2.79 较为中性，HOLD 决策合理。
