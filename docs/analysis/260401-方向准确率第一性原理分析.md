## 一、核心数学：需要多少准确率才能盈利？

### 当前状态（3/17-3/31 实盘）

```
交易级胜率 WR = 44%, avgWin = $103, avgLoss = $176
E[PnL] = 0.44 × 103 - 0.56 × 176 = -$52/笔
Kelly = -50.6%  →  不应下注

天级方向准确率 = 4/11 交易日方向正确 = 36%
（注: WR 是单笔交易盈亏比，方向准确率是当日主要信号方向正误，两者相关但不等价）
```

### A+D 退出优化后的预期

A+D 修复让亏损交易更快退出（STOP_LOSS 接管）+ 让赢家有更多时间发展：

```
预期 avgLoss: $176 → ~$140（止损更精准，不再被 TIME_STOP 提前打断）
预期 avgWin: $103 → ~$160（赢家有更多时间到达目标，如 NVDA 4/1 案例）
```

### 盈亏平衡点

```
E[PnL] = 0 时:
WR × avgWin = (1-WR) × avgLoss
WR × 160 = (1-WR) × 140
160·WR = 140 - 140·WR
300·WR = 140
WR = 46.7%
```

**只需 47% 的方向准确率就能盈亏平衡**（假设 A+D 修复生效）。

如果同时改善方向准确率到 55%：

```
E[PnL] = 0.55 × 160 - 0.45 × 140 = $88 - $63 = +$25/笔
日均 3 笔 × $25 = +$75/天
```

**结论：交易级 WR 距盈亏平衡只差 3pp（44%→47%）。方向信号改善同时提升 WR（更多正确交易）和 R 值（赢家更大、输家更小），协同效应让策略翻正。**

## 二、方向判断系统解剖

### finalScore 合成公式（L161-164）

```
finalScore = marketScore × 0.2 + intradayScore × 0.6 + timeWindowAdjustment × 0.2
```

方向决策经过两层阈值门控：

```
层1 — 推荐服务 (option-recommendation.service.ts L170-173):
  baseThreshold = 15, entryTimeFactor = get0DTETimeThresholdFactor()
  dynamicThreshold = 15 × entryTimeFactor (9:30-12:00 → 1.0, 尾盘递增到 3.0)
  finalScore > dynamicThreshold → 标记 CALL; < -dynamicThreshold → 标记 PUT; 否则 HOLD

层2 — 策略服务 (option-intraday-strategy.ts L369):
  effectiveThreshold = entryThresholdOverride.directionalScoreMin × vixFactor
  策略10 DB覆盖: directionalScoreMin=9, spreadScoreMin=7
  vixFactor = clamp(VIX/20, 0.5, 2.5)    // L285
  finalScore ≥ effectiveThreshold → 实际入场
```

**实际入场由层2决定。** 层1的标记仅用于日志。策略10有效阈值 = `Math.round(directionalScoreMin × vixFactor)`（L298），VIX=20→9，VIX=30→14，VIX=12→5（clamp 下限 VIX=10→5）。

### 4/1 NVDA 信号反算验证

```
NVDA 4/1 BUY 信号 (10:18 ET，开盘第一小时):
  marketScore = -0.689
  intradayScore = 10.259
  finalScore = 10.018

反算 (L161-164):
  finalScore = -0.689 × 0.2 + 10.259 × 0.6 + timeWindow × 0.2
  10.018 = -0.138 + 6.156 + timeWindow × 0.2
  timeWindow × 0.2 = 4.0
  timeWindow = +20 (开盘第一小时, L660-662)

阈值验证 (策略层 L369):
  directionalScoreMin = 9 (DB override)
  VIX ≈ 20 → vixFactor = 20/20 = 1.0
  有效阈值 = 9 × 1.0 = 9
  10.018 ≥ 9 → 入场 ✓
```

**关键发现**: 如果没有 timeWindow 的 +4 分加成，finalScore = 6.018 < 9（有效阈值），不入场。**时间窗口加成是决定性因素，不是方向判断。**

### 三层权重分解

```
finalScore (100%) =
  ├── marketScore (20%)  ← calculateMarketScore() L307-381
  │   ├── SPX 20日均线偏离 × 5 倍放大 × 0.4    (线性权重 40%)
  │   ├── USD 指数趋势 × 0.2 (反向)             (线性权重 20%)
  │   ├── BTC 趋势 × 0.1~0.2 (相关性>0.5 时 0.2)(线性权重 10-20%)
  │   ├── VIX 离散跳变: >35→-50, >25→-20, <15→+10  ⚠️ 非比例权重，可压倒其他分量
  │   └── 市场温度: >50→加分, <20→减分              ⚠️ 条件性，非固定权重
  │
  ├── intradayScore (60%)  ← 主驱动, L417-502
  │   ├── 底层标的 1min 动量 × 0.30 (L437)   ← 18% of final ✅ 正确时间尺度
  │   ├── VWAP 位置 × 0.15 (L450)            ← 9% of final  ✅ 合理
  │   ├── SPX 小时线动量 × 0.25 (L463)       ← 15% of final ⚠️ 太慢
  │   ├── BTC 小时线动量 × 0.15 (L480)       ← 9% of final  ❌ 噪音
  │   └── USD 小时线动量 × 0.15 反向 (L492)  ← 9% of final  ❌ 噪音
  │
  └── timeWindowAdjustment (20%)  ← calculateTimeWindowAdjustment() L650-675
      ├── 9:30-10:30 开盘第1小时: +20        ← +4 to final  ❌ 系统性CALL偏置
      ├── 10:30-11:30:             0         ← 中性
      ├── 11:30-14:30:            -10        ← -2 to final  轻度PUT偏置
      ├── 14:30-15:30 收盘前90-30: -30       ← -6 to final
      └── 15:30-16:00 收盘前30分:  -50       ← -10 to final
```

## 三、第一性原理诊断：五个结构性问题

### 问题 1: 18% 的 finalScore 是纯噪音（BTC + USD）

**第一性原理**: 2-15 分钟持仓的 0DTE 期权，标的是 NVDA/TSLA/SPY/QQQ。BTC 和 USD 的小时线走势对这些标的在未来 15 分钟的方向有多少预测力？

**答案: 接近于零。**

- BTC 与 SPY 的分钟级相关性极不稳定，时而正、时而负、时而无
- USD 对个股的影响是宏观层面的（天/周级别），不是分钟级的
- 这 30% 的 intradayScore（= 18% 的 finalScore）本质上是随机噪音

**注意**: BTC 对 finalScore 的影响有两条路径：intradayScore（L480, 固定 15%）和 marketScore（L329-336, 相关性>0.5 时 0.2 否则 0.1）。合计 BTC 影响 ≈ 11-13% 的 finalScore。去噪方案需同时处理两条路径。

**实际影响**: 在 3/20（CALL on -1.21% day，11笔交易），如果 BTC 那小时碰巧涨了、USD 跌了，就会推高 intradayScore → 产生错误的 CALL 信号。

### 问题 2: timeWindowAdjustment 制造非对称方向偏置

**第一性原理**: 时间窗口调整的本意是"在好时段鼓励交易、在坏时段抑制交易"。但实际实现把**时间偏好变成了方向偏好**：

```
完整偏置表 (marketScore=0, intradayScore=0 时 finalScore 的值):
  9:30-10:30:   +20 × 0.2 = +4.0  → 推 CALL
  10:30-11:30:    0 × 0.2 =  0.0  → 中性
  11:30-14:30:  -10 × 0.2 = -2.0  → 推 PUT
  14:30-15:30:  -30 × 0.2 = -6.0  → 强推 PUT
  15:30-16:00:  -50 × 0.2 = -10.0 → 极度推 PUT
```

问题不仅是开盘 CALL 偏置，而是**一天中方向偏置的分布不对称**：开盘推 CALL，然后全天递增推 PUT。但策略 10 通常只在开盘第一小时交易（`firstHourOnly` 配置），所以**实际暴露的几乎全是 +4 的 CALL 偏置**。

**结果**: 3/17-3/23 期间，市场连续下跌，但开盘时策略一直发 CALL 信号 — 因为 timeWindow 给了 +4 的 CALL 偏置。

### 问题 3: SPX 小时线对分钟级交易太慢

SPX 小时线动量占 intradayScore 的 25%（= finalScore 的 15%）。

**一根小时线代表 60 分钟的信息**。对于平均持仓 2.5 分钟的盈利交易，这就像用日线图做 5 分钟的 scalp — 信号在 N-1 小时前就已确定，对"接下来 5 分钟"几乎没有增量预测。

**而且 SPX 小时线还经过了 intradayDataFilter 的 EMA 平滑**，进一步增加滞后。

### 问题 4: marketScore 20日均线在趋势市中制造持续性偏差

```
SPX 3/17: 6716, 20日均线 ≈ 6750 → marketScore ≈ -5
SPX 3/20: 6506, 20日均线 ≈ 6720 → marketScore ≈ -30
SPX 3/27: 6369, 20日均线 ≈ 6650 → marketScore ≈ -45
SPX 3/31: 6529, 20日均线 ≈ 6600 → marketScore ≈ -15（仍然负值！）
SPX 4/1:  6570, 20日均线 ≈ 6590 → marketScore ≈ -0.7（终于接近零）
```

3/31 市场大涨 +1.73%，但 marketScore 仍为负值 → 推 PUT → **方向完全错误**。

问题本质: **20日均线偏离是一个回归信号（"市场相对均线偏低/偏高"），但回归的时间尺度是天~周，不是分钟**。用它来做分钟级方向判断，会在趋势市中产生持续性的错误偏差。

### 问题 5: 没有动量确认门槛

当前系统：finalScore 超过阈值就入场，但不检查标的**当前**是否在朝信号方向移动。

```
可能发生的场景:
  - BTC 涨了 → intradayScore 偏正 → finalScore 超过阈值 → CALL
  - 但 NVDA 此刻正在下跌
  - 入场 CALL → 立即亏损
```

**好的方向判断应该有一个"当下确认"**: 信号说 CALL，标的也在涨 → 入场。信号说 CALL，标的在跌 → 等待或放弃。

## 四、信号贡献度量化

用 4/1 NVDA 信号反推各因子对 finalScore 的实际贡献:

| 因子 | 得分 | × 权重路径 | 对 finalScore 贡献 | 贡献比例 |
|------|------|-----------|-------------------|---------|
| 底层 1min 动量 | ~+17 | × 0.30 × 0.6 | +3.1 | 31% |
| VWAP | ~+8 | × 0.15 × 0.6 | +0.7 | 7% |
| SPX 小时 | ~+5 | × 0.25 × 0.6 | +0.8 | 8% |
| BTC 小时 | ~+3 | × 0.15 × 0.6 | +0.3 | 3% |
| USD 小时 | ~+2 | × 0.15 × 0.6 | +0.2 | 2% |
| marketScore | -0.7 | × 0.2 | -0.1 | -1% |
| **timeWindow** | **+20** | **× 0.2** | **+4.0** | **40%** |
| **合计** | | | **+10.0** | **100%** |

**timeWindow (+20) 贡献了 finalScore 的 40%**。这不是方向信号，这是时间偏置。去掉它 → finalScore = 6.0。

策略10在 VIX≈20 时有效阈值 = `Math.round(9 × 1.0)` = 9。6.0 < 9 → 不入场。VIX=15 时阈值 = `Math.round(9 × 0.75)` = 7，仍然不入场。VIX≤14 时阈值降到 6（`Math.round(9 × 0.7)=6`），6.0 ≥ 6 才可能入场。

**4/1 NVDA 这笔方向正确的交易，实际上是被 timeWindow 偏置推过阈值的**，不是被方向信号驱动的。

## 五、修复方案：按第一性原理重构权重

### 原则

**方向信号应该只包含对标的未来 2-15 分钟方向有预测力的因子。时间、费用、流动性等非方向因素应该作为过滤器，不是得分分量。**

### 方案 1：去噪 — 移除无效因子（最高优先级，最低风险）

```
当前 intradayScore 构成:
  底层 1min 动量:  30% ✅
  VWAP:           15% ✅
  SPX 小时:       25% ⚠️ (太慢但有方向含义)
  BTC 小时:       15% ❌ (噪音)
  USD 小时:       15% ❌ (噪音)

改为:
  底层 1min 动量:  45% ✅ (核心信号，权重提升)
  VWAP:           20% ✅ (略提升)
  SPX 5min 动量:  35% ✅ (替换小时线，用5分钟线)
  BTC:             0%
  USD:             0%
```

**效果**: 消除 18% 的 finalScore 噪音。intradayScore 完全由标的自身和大盘的短期动量驱动。

**实现**: 修改 `option-recommendation.service.ts` 中 intradayScore 的权重。SPX 数据源从小时线改为 5 分钟线（如果可用）。

### 方案 2：修复 timeWindowAdjustment（高优先级）

**问题**: timeWindow 是正方向加分（+20），制造 CALL 偏置。

**修复**: 将 timeWindowAdjustment 从 **得分分量** 改为 **阈值修正**。

```
当前:
  finalScore = mkt×0.2 + intraday×0.6 + timeWindow×0.2  ← timeWindow 影响方向
  策略阈值 = directionalScoreMin × vixFactor             ← VIX 自适应已有

改为:
  finalScore = mkt×0.2 + intraday×0.8  // timeWindow 不再参与方向得分
  策略阈值 = directionalScoreMin × vixFactor × timeThresholdFactor

  timeThresholdFactor:
    开盘第1小时: 0.8 (降低阈值 → 更容易入场，但不偏向任何方向)
    正常时段:    1.0
    尾盘:        1.5~2.5 (提高阈值 → 更难入场)
```

**注意**: 尾盘阈值递增机制 (`get0DTETimeThresholdFactor`) 已在推荐服务中实现（L913-935），但它操作的是推荐层 baseThreshold=15，不影响策略层的入场门控。方案 2 将两种时间调整统一到策略层。

**效果**: 开盘第一小时仍然"更容易交易"（低阈值），但不再系统性偏向 CALL。方向完全由市场信号决定。

### 方案 3：降低 marketScore 权重 + 加快响应速度（中优先级）

**当前**: marketScore 20% 权重，基于 20 日均线，非常滞后。

**选项 A: 降权到 10%**
```
finalScore = mkt×0.1 + intraday×0.9
```
简单但仍有偏差。

**选项 B: 替换为 3 日动量（推荐）**
```
shortMomentum = (SPX_today_open - SPX_3day_close) / SPX_3day_close * 100
marketScore = shortMomentum × 10  // 归一化到 ±50 范围

特点: 3/31 SPX 大涨日:
  3日前收盘 ≈ 6369 (3/27), 当日开盘 6418
  shortMomentum = +0.77% → marketScore = +7.7 → 正确推向 CALL

  vs 当前 20日均线:
  marketScore ≈ -15 → 错误推向 PUT
```

**选项 C: 改为 Gap 信号（最直接）**
```
gapScore = (todayOpen - prevClose) / prevClose * 100

3/31: gap = (6418-6344)/6344 = +1.17% → gapScore = +11.7 → CALL 信号
4/1:  gap = (176-174.4)/174.4 = +0.92% → gapScore = +9.2 → CALL 信号

这比 20日均线 对当日方向的预测力强得多
```

### 方案 4：加入动量确认门槛（中优先级）

```
入场前增加检查:
  momentumConfirmation = 标的最近 5 根 1min K线的方向

  if (signal == CALL && 5bar_momentum < 0) → 推迟入场
  if (signal == PUT  && 5bar_momentum > 0) → 推迟入场
```

**效果**: 避免在标的正在反方向移动时入场。4/1 NVDA 入场时 NVDA 可能正在下跌（10:18 附近，低点 $174.76），如果有这个确认，可能会推迟入场到股价反弹时 → 更好的入场价。

### 方案 5：开盘区间突破 ORB（低优先级，需要更多开发）

```
开盘前 30 分钟:
  rangeHigh = max(high of 9:30-10:00)
  rangeLow = min(low of 9:30-10:00)

10:00 之后:
  if (price > rangeHigh) → CALL 偏置
  if (price < rangeLow) → PUT 偏置
  if (price in range) → HOLD

这是日内交易最经典的方向判断方法之一，有数十年实证
```

## 八、关键洞察总结

### 1. 不需要"预测市场"，只需要"不逆市"

当前 36% 准确率意味着策略**经常逆市交易**。目标不是精准预测方向，而是：
- **市场明显涨的时候别做 PUT**（3/31 的错误）
- **市场明显跌的时候别做 CALL**（3/18、3/20 的错误）
- **不确定时别交易**（提高阈值 → HOLD）

这通过去噪 + Gap 信号 + 动量确认就能基本实现。

### 2. 最大杠杆点：去掉 18% 的 finalScore 噪音

BTC+USD 贡献 18% 的 finalScore 但几乎 0% 的预测力。这是最高性价比的改动 — 删除代码就能提升性能。

### 3. timeWindow 偏置是隐藏最深的 bug

开盘 +20 看起来是"鼓励在好时段交易"，但实际上是"在开盘时系统性偏向 CALL"。在 3/17-3/23 连续下跌周，这个偏置让策略在每天开盘时发出错误的 CALL 信号。

### 4. 方向准确率和 R 值是乘法关系

A+D 提升 R 值（让赢家跑、限制输家），Phase 3 提升方向准确率。**两者的改善是乘法效应**：
- 单独 A+D: E[PnL] 从 -$52 → -$20（仍负）
- 单独 Phase 3: E[PnL] 从 -$52 → -$10（仍负）
- **A+D + Phase 3 = E[PnL] 正值**

所以这不是二选一的问题，而是必须一起做。

### 5. 已有但未充分利用的自适应机制

系统已有两个动态机制：
- **VIX 自适应阈值**: `effectiveThreshold = directionalScoreMin × VIX/20`（option-intraday-strategy.ts L283-298）
- **尾盘阈值递增**: `get0DTETimeThresholdFactor()` 从 1.0 递增到 3.0（option-recommendation.service.ts L913-935）

但两者互相独立且作用于不同层。Phase 3 应统一并扩展这些机制，而非从零开始设计动态参数。

### 6. 方向系统的终极瓶颈

即使做完 Phase 3 所有改动，方向准确率大概在 55-60%。要突破 60%，需要：
- 更好的日内数据源（Level 2 订单簿、options flow）
- 机器学习模型（但需要更多训练数据）
- 或者放弃方向预测，转向波动率策略（straddle/strangle）

当前阶段应聚焦 Phase 3，在策略转正后积累更多数据再考虑更高级方法。
