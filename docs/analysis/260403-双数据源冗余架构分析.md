# LongPort + FutuOpenD 双数据源冗余架构分析

> 日期: 2026-04-03
> 背景: 当前系统依赖 Moomoo 爬虫获取大盘指数（SPX/VIX/USD/BTC），不稳定且延迟高。
> 目标: 评估 LongPort + FutuOpenD 双源冗余方案的可行性。

---

## 一、当前数据源现状

| 数据 | 当前来源 | 方式 | 问题 |
|------|----------|------|------|
| SPX 日K | Moomoo 爬虫 | HTTP 轮询 | Cookie 过期/403/反爬 |
| SPX 分时 | Moomoo `get-quote-minute` | HTTP 轮询 | 延迟 5s+，不稳定 |
| USD Index | Moomoo 爬虫 | HTTP 轮询 | 同上 |
| BTC | Moomoo 爬虫 | HTTP 轮询 | 同上 |
| VIX | Moomoo 爬虫 + LongPort | HTTP 轮询 | 同上 |
| 标的 1min/5min K线 | LongPort SDK | HTTP 拉取 | 稳定，延迟 ~1-2s |
| 标的实时报价 | LongPort SDK | HTTP 拉取 | 5s 轮询 |

**核心问题**: Moomoo 爬虫是单点故障，一旦 Cookie/反爬策略变化，大盘数据全断。

## 四、双源竞速架构设计

### 4.1 核心思路：Promise.race — 同时调用，先到先用

大盘评分需要低延迟，不能等一个超时再切另一个。正确做法是**双源并发 + 竞速**：

```
┌──────────────────────────────────────────────────┐
│           MarketDataService (统一接口层)            │
│                                                   │
│  getRacedKline(symbol, period)                    │
│  getRacedSnapshot(symbol)                         │
│  subscribeRealtime(symbol, callback)              │
└────────┬───────────────────────┬──────────────────┘
         │                       │
    ┌────▼────┐            ┌────▼─────┐
    │ LongPort │            │ FutuOpenD │
    │ adapter  │            │ adapter   │
    └────┬────┘            └────┬─────┘
         │                       │
    getQuoteContext()         futu-bridge
    (DB token,进程内)        (socat:11112)
         │                       │
         └──── Promise.race ────┘
                    │
                    ▼
            formatToStandard()
            (统一 CandlestickData)
```

### 4.2 拉取类接口：Promise.race 竞速

```typescript
async function getRacedKline(symbol: string, period: KLinePeriod): Promise<StandardCandlestickData[]> {
  const longportPromise = longportAdapter.getKline(symbol, period)
    .then(data => ({ source: 'longport', data }))
    .catch(err => ({ source: 'longport', data: null, error: err }));

  const futuPromise = futuAdapter.getKline(symbol, period)
    .then(data => ({ source: 'futu', data }))
    .catch(err => ({ source: 'futu', data: null, error: err }));

  // 两个同时发，谁先回来用谁
  const result = await Promise.any([longportPromise, futuPromise]);
  // Promise.any: 第一个成功的返回；全部失败才抛 AggregateError

  return result.data;
}
```

**关键区别**：
- ~~主备模式~~（超时 3s 再切换）→ 错过机会窗口
- **竞速模式**（同时发出，先到先用）→ 延迟 = min(LongPort延迟, FutuOpenD延迟)
- 任一源故障 = 自动降级到另一源，无感知

### 4.3 推送类接口：双活 + 去重

```
LongPort subscribe ──┐
                     ├── 去重合并(symbol + timestamp) → 回调
FutuOpenD subscribe ─┘
```

- 两个源同时订阅，以先到的数据为准
- 相同 symbol + 相同时间戳（精确到秒）的数据去重
- 任一源断开不影响数据流

### 4.4 LongPort 调用方式修正

LongPort token 存储在数据库 `system_config` 表中，通过 `getQuoteContext()` 获取（`api/src/config/longport.ts`）。
初始化链路：`getQuoteContext()` → `ConfigService.get('longport_access_token')` → DB → 构建 Config → QuoteContext。
NAS 环境变量 `SKIP_LONGPORT_INIT=true` 仅影响启动时的预初始化，不影响运行时通过 DB token 的按需初始化。

### 5.3 Symbol 格式映射

| 标准 | LongPort | FutuOpenD |
|------|----------|-----------|
| SPY | `SPY.US` | `US.SPY` |
| QQQ | `QQQ.US` | `US.QQQ` |
| 腾讯 | `700.HK` | `HK.00700` |

**转换函数**：
```typescript
function toLongportSymbol(symbol: string): string {
  // US.SPY → SPY.US, HK.00700 → 700.HK
  const [market, code] = symbol.split('.');
  return `${code}.${market}`;
}

function toFutuSymbol(symbol: string): string {
  // SPY.US → US.SPY, 700.HK → HK.00700
  const [code, market] = symbol.split('.');
  return `${market}.${code}`;
}
```

## 六、集成方案：FutuOpenD 桥接服务

FutuOpenD 是 Python SDK，trading-app 是 Node.js。需要桥接层。

### 方案 B：轻量 Python HTTP 桥接 (推荐)

```
trading-app (Node.js) ──HTTP──→ futu-bridge (Python FastAPI/Flask)
                                     └── TCP → futu-opend:11112
```

- futu-bridge 暴露 REST API，返回 JSON（已统一字段名）
- trading-app 用 `fetch()` 调用，与 LongPort 的调用方式对齐
- 可打包为独立 Docker 容器，部署在 NAS 上

**futu-bridge 核心接口**：
```
GET /kline?symbol=US.SPY&period=1m&count=50
GET /snapshot?symbols=US.SPY,US.QQQ
GET /quote?symbols=US.SPY
WS  /subscribe?symbols=US.SPY&types=quote,kline_1m
```

**返回格式**（已归一化，trading-app 无需再转换）：
```json
{
  "source": "futu",
  "symbol": "SPY.US",
  "klines": [
    {"timestamp": 1775214660000, "open": 646.42, "high": 647.47, "low": 646.38, "close": 646.86, "volume": 710788, "turnover": 459768234}
  ]
}
```

## 七、可行性结论

**完全可行，且是正确方向。**

### 收益

1. **消除 Moomoo 爬虫** — 去掉最不稳定的数据源（Cookie/反爬/403）
2. **双源竞速** — `Promise.race` 同时调用，延迟 = min(两源)，不浪费任何时间
3. **自动容错** — 任一源故障，另一源无缝接管（`Promise.any` 语义）
4. **低延迟推送** — 两个 SDK 都支持 WebSocket 订阅，延迟从 5s → ~200ms
5. **数据更丰富** — FutuOpenD 独有资金流向、条件选股能力
6. **VIX 双通道** — LongPort 可直接拿 VIX 指数（绝对值），FutuOpenD 用 VXX（趋势方向）

### 风险

1. **FutuOpenD 桥接层** — 需新增 Python 微服务，增加运维复杂度
2. **VXX 与 VIX 不等价** — VXX 阈值需重新标定（VIX>30 veto 不能直接套用到 VXX）
3. **Symbol 格式差异** — 需要双向转换（`SPY.US` ↔ `US.SPY`），必须覆盖所有标的

### 实施优先级

| 阶段 | 内容 | 说明 |
|------|------|------|
| P1 | 构建 futu-bridge 桥接服务 | Python FastAPI + Docker，暴露 REST/WS |
| P2 | 新增 `formatFutuCandlestick` + symbol 转换 | 统一到 StandardCandlestickData |
| P3 | MarketDataService 改造：双源竞速 | `Promise.any` 替代单源调用 |
| P4 | 将 Moomoo 爬虫的 SPX/USD/BTC 切换到 ETF | SPY/UUP/IBIT 替代 |
| P5 | 启用双源 WebSocket 订阅推送 | 双活 + 去重 |
