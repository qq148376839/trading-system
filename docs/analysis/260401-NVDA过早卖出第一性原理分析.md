## 二、第一性原理拆解：退出机制的职责边界

每个退出机制解决**一个特定问题**：

| 退出机制 | 解决的问题 | 触发条件 |
|---------|-----------|---------|
| STOP_LOSS | **方向错了**，市场不认可这笔交易 | 亏损达到阈值 |
| TIME_STOP | **方向不明**，theta 在侵蚀死钱 | 持仓超时 + 未盈利 |
| TRAILING_STOP | **方向对了但在回头**，保住已有利润 | 从高点回撤达到阈值 |
| TAKE_PROFIT | **方向对了且够了**，锁定目标利润 | 盈利达到阈值 |

### TIME_STOP 的实际代码行为

代码位于 `option-dynamic-exit.service.ts` L509-521：

```typescript
// 3.5 最大持仓时间止损
if (holdingMinutesForMaxHold >= maxHold && pnl.grossPnLPercent < 5) {
  return { action: 'TIME_STOP', ... };
}
```

实际触发条件是 **`holdTime >= maxHold AND pnlPercent < 5%`**。已排除盈利超 5% 的持仓，但对深度亏损持仓无下界保护——从 -100% 到 +4.99% 全部触发。

### 关键发现：退出检查的优先级顺序

代码中各退出条件的检查顺序：

```
0. 0DTE 强制平仓（收盘前 180 分钟）
1. 收盘前时间止损（10 分钟）
2. TAKE_PROFIT 止盈
2.8 阶梯锁利 (TRAILING_STOP/ratchet)
3. 移动止损 (TRAILING_STOP/trailing)
3.5 maxHoldMinutes TIME_STOP        ← 在 STOP_LOSS 之前
4. 0DTE 兜底止损 (STOP_LOSS)
5. 常规止损 (STOP_LOSS)
6. 安全阀 40% 硬止损
```

**TIME_STOP（步骤 3.5）在 STOP_LOSS（步骤 4/5）之前检查。** 当持仓处于 TIME_STOP 和 STOP_LOSS 之间的"灰色地带"时（如 -18.7% 亏损、止损 -20%），TIME_STOP 会抢先触发。

4/1 NVDA 正是这种情况：

- 15 分钟时 PnL = -18.7%，离 20% 止损只差 1.3%
- TIME_STOP 条件成立（15min 且 <5%）→ 立即触发
- STOP_LOSS 条件不成立（-18.7% > -20%）→ 即使检查也不会触发
- 但此时 **NVDA 正在反弹**，既不会触发止损，也会开始恢复

**这是一个覆盖间隙：-10% 到 -20% 之间的持仓没有合适的退出机制。TIME_STOP 不应管（不是"theta 吃死钱"），STOP_LOSS 未到阈值，结果 TIME_STOP 越俎代庖。**

## 三、根因分析：Phase 1 修复改变了 TIME_STOP 的前提

### Phase 1 修复清单（已部署）

| 修复项 | 变更 | 解决的原始问题 |
|--------|------|--------------|
| stopLossPercent | 35% → 20% | 亏损交易出血时间过长 |
| maxHoldMinutes | 无 → 15min | 3笔超30min亏损合计 -$728 |
| profitLockSteps | 收窄 floors | 利润回吐 |
| maxDailyTradesPerUnderlying | 无 → 1 | 重复入场摧毁利润 |
| minEntryPrice | 无 → $1.00 | 低delta彩票期权闪崩 |

### 隐含的交互效应

原始诊断中引入 TIME_STOP 的 3 笔案例：

```
3/26 NVDA C175000: 49min, -$228 → SmartReverse 导致方向错误
3/26 TSLA C385000: 33min, -$220 → SmartReverse 导致方向错误
3/24 NVDA P175000: 35min, -$280 → marketScore 滞后导致方向偏差
```

**这 3 笔全部是方向判断错误造成的。** "持仓太久"是表象，方向错误是根因。

Phase 1 同时修了因（SmartReverse 禁用）和果（TIME_STOP），但当因已消除时，果的修复变成过度纠正：

```
前提1: 3笔超长亏损的根因是 SmartReverse + 方向错误
前提2: Phase 1 已禁用 SmartReverse
前提3: Phase 1 已将 stopLoss 35%→20%（加速错误方向的退出）
结论: "超长亏损滞留"问题已被根因修复消除。
      TIME_STOP 正在修复一个不再存在的问题。
```

4/1 NVDA 验证了这个推理：

- 没有 SmartReverse → 方向正确（CALL on bullish day）
- 20% 止损已设置 → -18.7% 时未触发 = 还没判定为"方向错"
- TIME_STOP 在 15 分钟强制退出了一个**方向正确、暂时回撤、即将反弹**的持仓

### "到达 15 分钟的亏损交易"性质已变

| 维度 | Phase 1 前（3/17-3/31） | Phase 1 后（4/1） |
|------|----------------------|-----------------|
| SmartReverse | 活跃，制造方向错误 | 已禁用 |
| stopLossPercent | 35%（太宽，亏损持仓滞留） | 20%（加速退出） |
| 达到15min的亏损交易 | 大概率方向错误 → **不可恢复** | 更可能是暂时回撤 → **可能恢复** |

TIME_STOP 的决策前提已变，但触发条件没变。

## 四、TIME_STOP 的第一性原理重设计

### 当前设计的问题

```typescript
// 当前代码（简化）
if (holdTime >= 15min && pnlPercent < 5%) → 退出
```

隐含假设：**超过 15 分钟且未盈利的 0DTE 持仓，theta 衰减 > 可能的方向性收益。**

这个假设在 Phase 1 之前可能成立（方向经常错），Phase 1 之后不再成立。

### 应该是什么模型？

TIME_STOP 应只处理"方向不明、theta 吃死钱"的场景。它不应替 STOP_LOSS 做判断：

```
正确的职责划分:
- 亏损不大（如 > -10%）+ 时间到了 → theta 在吃死钱 → TIME_STOP 退出
- 深度亏损（如 < -10%）+ 时间到了 → STOP_LOSS 会处理 → TIME_STOP 不介入
- 盈利中 → TRAILING_STOP / TAKE_PROFIT 处理（当前已正确排除 >5%）
```

### 4/1 NVDA 在正确模型下的表现

```
15 分钟时: pnlPercent = -18.7%
条件: -18.7% < -10% → 不触发 TIME_STOP
→ 持仓继续
→ STOP_LOSS 在 -20% 待命（但 NVDA 开始反弹，没到 -20%）
→ 反弹后 TRAILING_STOP 接管
→ 预计退出价 ~$1.80-2.00，PnL ≈ +$150 ~ +$270
```

## 五、四个方案的评估

### 方案 A：TIME_STOP 加亏损下界（推荐度: ★★★★）

```typescript
if (holdTime >= maxHold && pnlPercent < 5 && pnlPercent >= -10) → 退出
```

- ✅ 解耦 TIME_STOP 和 STOP_LOSS 的职责，最小改动
- ⚠️ -10% 阈值选择需数据验证；-5%~-20% 之间的持仓只由 STOP_LOSS 管

### 方案 B：基于 delta 调整 maxHoldMinutes（推荐度: ★★★）

```
delta > 0.6: 25min | delta 0.3~0.6: 15min | delta < 0.3: 10min
```

- ✅ 高 delta 更像股票，theta 慢，给更多时间合理
- ❌ 没解决核心问题（TIME_STOP 不应在深度亏损时抢 STOP_LOSS 的活）

### 方案 C：盈利延时（推荐度: ★★）

```
曾达 +10% → 延 10min，曾达 +20% → 再延 10min
```

- ❌ **完全不解决 NVDA 问题**（NVDA 从未盈利，不获得延时）

### 方案 D：放宽到 25 分钟（推荐度: ★★★）

- ✅ 最简单，4/1 NVDA 25min 时大概率已反弹
- ⚠️ 治标不治本，下次在 25min 时同样问题

## 六、综合判断

### 核心矛盾一句话

**TIME_STOP 被设计来止血"方向错误的长亏损"，但 Phase 1 已通过禁用 SmartReverse + 收紧止损消除了"方向错误"的根因。加上代码中 TIME_STOP 优先级高于 STOP_LOSS，它抢先杀死了"方向对但暂时回撤"的持仓。**

### 推荐方案：A+D 组合

```typescript
// Phase 2 修改建议 — option-dynamic-exit.service.ts L509-521
// 1. maxHoldMinutes: 15 → 25（策略配置）
// 2. TIME_STOP 增加亏损下界

const maxHold = Number(exitRulesOverride.maxHoldMinutes);
if (!isNaN(maxHold) && maxHold > 0
    && holdingMinutesForMaxHold >= maxHold
    && pnl.grossPnLPercent < 5
    && pnl.grossPnLPercent >= -10) {  // 新增：深度亏损交给 STOP_LOSS
  return { action: 'TIME_STOP', ... };
}
```

**参数依据**：
- `maxHoldMinutes: 25` — 给趋势更多时间，Phase 1 已消除 30-50min 极端滞留的根因
- `pnlPercent >= -10%` — 止损阈值 20% 的一半，留 10% 空间让 STOP_LOSS 判断；-5%~-10% 之间的"温水煮青蛙"持仓正是 TIME_STOP 应处理的

### 验证指标（实施后观察 2 周）

| 指标 | 基线 (4/1) | 预期方向 |
|------|-----------|---------|
| TIME_STOP 触发次数 | 1/4 笔 | 减少 |
| TIME_STOP 触发时 PnL% | -18.7% | > -10%（设计保证） |
| 被 TIME_STOP 杀后反弹比例 | 1/1 (100%) | 需更多数据 |
| 策略 E[PnL] | -$52/笔 | 接近 $0 或正 |

### 远期思考

退出优化有天花板。如果 Phase 1 + Phase 2 后策略 E[PnL] 仍为负，核心问题在**方向判断**——诊断报告显示方向准确率仅 36%。退出机制是防御层，不能把方向性劣势的策略变成正 EV。

**Phase 3 应聚焦：marketScore + intradayScore 的方向判断改进。**

## 七、结论

| 层级 | 发现 | 建议 |
|------|------|------|
| **表层** | TIME_STOP 在 15min 杀死 NVDA CALL，错失 $510 | 放宽 maxHoldMinutes → 25 |
| **代码层** | TIME_STOP 优先级高于 STOP_LOSS，深度亏损时抢先触发 | 增加 `pnlPercent >= -10%` 下界 |
| **系统层** | Phase 1 的多项修复改变了"达到 15min 的亏损交易"的性质 | TIME_STOP 设计前提需更新 |
| **本质** | 退出优化有天花板，方向判断是真正瓶颈 | Phase 3 聚焦 marketScore 改进 |
