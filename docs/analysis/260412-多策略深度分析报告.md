## 一、多策略并发冲突诊断

### 当前隔离矩阵

| 维度 | 隔离级别 | 机制 | 风险等级 |
|------|---------|------|---------|
| **资金** | 完全隔离 | `SELECT FOR UPDATE` 事务锁 | 低 |
| **持仓** | **部分隔离** | DB 追踪 per-strategy，但券商聚合 | **高** |
| **信号** | 完全隔离 | 状态机 + 竞价过滤 | 低 |
| **状态** | 完全隔离 | `UNIQUE(strategy_id, symbol)` | 低 |
| **订单** | **无隔离** | 所有策略共享同一券商通道 | **中** |
| **缓存** | 共享 | 30s 持仓缓存 / 10s 余额缓存 | 中 |

### 四大冲突点

#### 冲突 1: 持仓卖出超卖
- `calculateAvailablePosition()` 返回券商聚合持仓，不区分策略
- `validateSellPosition()` 接收 `strategyId` 参数但**从未使用**
- `strategy_instances.context.quantity` 存储了 per-strategy 数量但在卖出验证中被忽略
- **代码位置**: `basic-execution.service.ts:440-555`

#### 冲突 2: 同标的双重敞口
- 无约束阻止两个策略同时持有同一标的
- `requestAllocation()` 只查本策略资金，无跨策略敞口检查
- `scoringAuction()` 只在单策略内竞价
- `getEffectiveSymbolPool()` 不考虑其他运行中的策略
- **代码位置**: `capital-manager.service.ts:399-590`, `strategy-scheduler.service.ts:2843-2982`

#### 冲突 3: 共享缓存脏读
- 持仓缓存用单一全局 key `'all_positions'`，30s TTL
- 策略 A 买入后，策略 B 在 30s 内看到过时持仓
- **代码位置**: `strategy-scheduler.service.ts:3154-3200`

#### 冲突 4: 相关组隔离
- `crossSymbolState` 按 strategyId 分离，各自独立追踪
- 策略 A 退出 QQQ 后，策略 B 无感知，可能立刻入场
- **代码位置**: `strategy-scheduler.service.ts:202-226`

### 现有缓解机制
- `CROSS_STRATEGY_SYNC`（line 1566-1627）: 反应式清理，当策略 A 卖出后券商持仓归零时，清理其他策略的幽灵 HOLDING。是事后补救，不是事前预防。

## 二、评分逻辑对比：期权 vs 正股

### 正股策略评分（TradingRecommendationService）

```
env_score = SPX趋势强度 × 0.4 + 市场温度(归一化) × 0.4 + VIX(归一化) × 0.2
```

- 3 因子，日线级别
- 决策方式：分类判断（良好/中性利好/中性/中性利空/较差）→ 结合个股趋势
- VIX > 35 一票否决
- 无分钟级信号，无成交量加权，无动态阈值

### 期权策略评分（OptionRecommendationService Phase 3）

```
finalScore = 市场分(20%) + 日内分(60%) + 时间窗口分(20%)
```

**市场分 (20%)**:
- SPX 日线趋势 20%（多窗口：3d/10d/20d 一致性加权）
- SPX 分钟趋势 20%（1min+5min 加权）
- Gap 信号 0-15 点
- USD -10%（日线+分钟）
- BTC 5-10%（共振判断）
- VIX 三级评分（非一票否决）
- 市场温度 10%

**日内分 (60%)**:
- 1min 成交量加权动量 30%（ATR 归一化）
- 5min 趋势确认 15%
- VWAP 偏离 15%
- SPX 日内 25%
- BTC/USD 小时级 15%

**时间窗口分 (20%)**:
- 去偏处理（对称，不偏好方向）
- 开盘 30min +10 → 下午递减 → 最后 30min -50

**动态阈值**:
```
effectiveThreshold = max(absoluteScoreFloor, baseThreshold × vixFactor × timeFactor)
```

### 可迁移信号评估

| 信号 | 期权中的可靠性 | 可迁移到正股 | 理由 |
|------|-------------|------------|------|
| **SPX 多窗口一致性** | 最高 | **是** | 趋势确认不分品种 |
| **VIX 三级实时评分** | 高 | **是** | 替代一票否决，分级更合理 |
| **1min 成交量加权动量** | 高 | **部分** | 日内可用，swing 不需要 |
| **5min 趋势确认** | 高 | **部分** | 验证 1min 方向 |
| **Gap 信号** | 中 | **是** | 隔夜缺口对正股同样关键 |
| **BTC/USD 分钟联动** | 中 | **是** | 风险偏好实时脉搏 |
| **CHOP 震荡检测** | 中（Schwartz） | **是** | 震荡期暂停开仓 |
| **VWAP 偏离** | 低 | **否** | 正股 swing 尺度下意义不大 |
| **时间衰减** | N/A | **否** | 正股无到期日 |

## 三、正股策略蒸馏：Robin 模型 → TREND_FOLLOWING_V1

### Robin 核心特征系统化翻译

| Robin 特征 | 系统参数 |
|-----------|---------|
| 重仓新高股 | 只做 52 周高点附近（>85% 区间）的标的 |
| 截断亏损 | ATR 倍数止损（1.5-2x ATR），无例外 |
| 让利润奔跑 | 追踪止盈（不设固定止盈，只设 trailing stop） |
| 三倍杠杆 ETF | 标的池含 TQQQ/SQQQ/SPXL/SPXS |
| 行业轮动 | 多板块标的池 + 相对强度排名 |
| MA 系统 | MA50/MA200 双均线过滤 |
| 无情感 | 纯评分驱动，不人工干预 |

### 评分模型设计

```
策略名: TREND_FOLLOWING_V1
周期: 60s 扫描
标的池: ~15-20 只（强势股 + 杠杆 ETF + 板块龙头）

┌─────────────────────────────┐
│ 趋势分 (40%)               │
│  - MA50/MA200 位置关系 20%  │ ← Robin 核心
│  - 52周高点距离 10%         │ ← Robin 核心
│  - SPX 多窗口一致性 10%     │ ← 期权验证过
├─────────────────────────────┤
│ 动量分 (30%)               │
│  - 相对强度(RS) vs SPX 15% │ ← 行业轮动
│  - 成交量确认 10%           │ ← 期权验证过
│  - Gap 信号 5%              │ ← 期权验证过
├─────────────────────────────┤
│ 环境分 (30%)               │
│  - VIX 三级评分 10%         │ ← 替代一票否决
│  - BTC 风险偏好 5%          │
│  - USD 逆相关 5%            │
│  - 市场温度 10%             │
└─────────────────────────────┘
```

### 入场条件
1. 综合分 > 动态阈值（VIX 调整）
2. 价格 > MA50 且 MA50 > MA200（金叉或多头排列）
3. 相对强度排名前 5
4. CHOP 检测非震荡期

### 出场条件
1. 价格跌破 MA50 → 减半仓位
2. 价格跌破 MA200 → 清仓
3. ATR trailing stop（2x ATR 从高点回撤）
4. 日内熔断（单日亏损 > 2% 本金 → 停止新开仓）

### 仓位管理
- 每只标的最大 15% 资金（正股杠杆低需分散）
- 杠杆 ETF 标的最大 10%（3x 杠杆自带放大）
- 相关组内最大 30%（防集中暴露）

### Robin 弱点对冲

| Robin 弱点 | 系统化对冲 |
|-----------|-----------|
| 震荡市连续假突破 | CHOP 检测 — 震荡期暂停开仓 |
| 黑天鹅无对冲 | VIX > 25 + 持有多头 → 自动建 SQQQ 仓位 |
| 杠杆 ETF 磨损 | TQQQ/SQQQ 最长持仓 5 天 |
| 追高被套 | 开盘跳空 > 2% 不追，等回踩 MA |

## 四、Schwartz vs Intraday 策略对比

### 架构关系

两者共享完全相同的信号源 `optionRecommendationService.calculateOptionRecommendation()`。
Schwartz 继承 `OptionIntradayStrategyConfig`，本质是 Intraday + 4 个额外过滤器。

### Schwartz 独有过滤器

1. **EMA 硬过滤** — 10 EMA 逆趋势信号一刀切拒绝（`schwartzSignalFilter.checkEMAFilter()`）
2. **CHOP 震荡检测** — MA 缠绕时入场阈值翻倍（`schwartzSignalFilter.checkChopFilter()`）
3. **IV Rank 过滤** — 高隐含波动率时拒绝买方策略（`schwartzSignalFilter.checkIVFilter()`）
4. **仓位缩减** — 连续盈利/大赚后自动缩减合约数（`schwartzSignalFilter.calculatePositionSize()`）

### Intraday 独有能力

1. **智能反向（MarketRegimeDetector）** — regime 判别 + 方向翻转
2. **加速度 bonus（Phase 4）** — FastMo 二阶导提前入场
3. **多策略类型** — 跨式/价差/铁鹰等（Schwartz 只做单边买方）
4. **资金预检缓存** — 超预算标的 5 分钟内不重复评估

### 决策

- **砍掉 Schwartz**，保留 Intraday 作为唯一期权策略
- **4 个 Schwartz 过滤器合入 Intraday** 作为可配置开关（默认关闭）
- 理由：Intraday 有更丰富的基础设施 + 更多已验证数据。Schwartz 的过滤器有价值但不值得独立策略
- 生产环境关闭 → 保护已验证数据；测试环境开启 → 对比后决定是否生产启用

## 五、实施优先级（修订）

> 原方案：5-Phase CrossStrategyCoordinator → 已废弃
> 原因：第一性原理重审，期权+正股天然不冲突，协调层是过度工程

### 新路径

1. **合并 Schwartz 到 Intraday**（4 个过滤器作为配置开关，默认关闭）
2. **停用 OPTION_SCHWARTZ_V1**（DB 标记 STOPPED，代码保留）
3. **实现正股策略 TREND_FOLLOWING_V1**（复用期权验证过的信号）
4. **1 期权 + 1 正股联调**（测试环境验证）
5. **生产灰度上线**

### 关键简化

| 对比 | 原方案 | 新方案 |
|------|--------|--------|
| 新增服务 | CrossStrategyCoordinator ~250 行 | 无 |
| 改动文件数 | 4 个 | 1 个 |
| 改动行数 | ~325 行 | ~40 行 |
| 风险 | 中（插入关键路径） | 低（只加配置开关） |
| 前提条件 | 必须先完成协调层 | 无，可直接开始正股策略 |
