# 行情订阅架构升级分析 — 从 5 秒轮询到毫秒级推送

**日期**: 2026-04-03
**触发**: 策略大修讨论中发现行情获取不及时是入场滞后的底层原因之一

---

## 一、当前架构: 纯轮询（Polling）

```
当前数据流:

setInterval(5000ms)          ← 每 5 秒一次
    ↓
quoteCtx.quote(idleSymbols)  ← LongPort REST API 轮询
    ↓
priceMap: Map<symbol, price> ← 只拿到 lastDone
    ↓
fastMomentumService.feedQuotes(priceMap)  ← 喂入 RingBuffer
    ↓
RingBuffer: 12 samples × 5s = 60s 窗口    ← 最高分辨率 5s
    ↓
checkGate(): 线性回归判断趋势              ← 5 秒颗粒度的回归
```

**关键数据路径** (`strategy-scheduler.service.ts:751-771`):
```typescript
const quoteCtx = await getQuoteContext();
const quotes = await quoteCtx.quote(idleSymbols);  // ← 轮询 API
const priceMap = new Map<string, number>();
for (const q of quotes) {
  const price = Number(q.lastDone?.toString() || '0');
  priceMap.set(q.symbol, price);
}
fastMomentumService.feedQuotes(priceMap);
```

**问题**:
- 每 5 秒才能看到一次价格
- 两次轮询之间发生的价格波动完全不可见
- FastMo 的 60 秒窗口只有 12 个数据点做线性回归
- K 线数据通过 `candlesticks()` API 单独拉取，有 60 秒缓存

## 三、对比: 轮询 vs 订阅

| 维度 | 当前（轮询） | 升级后（订阅） |
|------|-------------|---------------|
| **数据延迟** | 0~5 秒（取决于轮询时机） | **~50-200ms**（WebSocket 推送） |
| **价格分辨率** | 每 5 秒 1 个价格点 | **每笔成交 1 个价格点** |
| **60 秒窗口数据点** | 12 个 | **数百到数千个** |
| **K 线获取** | 每 60 秒轮询一次 `candlesticks()` | **实时推送，收线即到** |
| **API 调用频率** | 每 5 秒一次 REST 调用 | **0 次 REST 调用**（订阅后纯推送） |
| **网络开销** | 高（频繁建立 HTTP 连接） | 低（单条 WebSocket 长连接） |
| **Volume 数据** | 只在 K 线里有 | **每笔报价都有 volume** |
| **盘口数据** | 不获取 | 可实时获取 bid/ask |

## 四、对入场时机问题的影响（核心）

### 当前的入场时序（轮询架构）

```
T-20min:  标的开始形成趋势
T-10min:  1min K 线连续同向收线
T-5min:   momentum 评分开始累积
          但... 轮询每 5 秒一次，K 线每 60 秒缓存一次
          所以 momentum 的实际更新比市场慢 5-60 秒
T-2min:   momentum 终于达到入场阈值
T-1min:   竞价通过，提交订单
T+0:      入场（价格已大幅移动）
T+1min:   均值回归，开始亏损
```

### 升级后的入场时序（订阅架构）

```
T-20min:  标的开始形成趋势
T-10min:  实时报价推送到 FastMo，RingBuffer 数百个数据点
          线性回归精度大幅提升，trend 检测更早
T-8min:   基于实时数据，momentum 变化率（加速度）可精确计算
          可检测"momentum 正在构建"而非等"momentum 已经很强"
T-5min:   高精度 FastMo slope 可直接参与评分（不只是 gate）
          1min K 线收线推送触发 momentum 重算
T-3min:   信号触发，入场
T+0:      趋势继续发展中（还没到峰值）
T+2min:   趋势达到峰值
T+5min:   开始回撤 → 但已有 5 分钟利润缓冲
```

**关键差异**: 订阅架构让系统能"看到"趋势的构建过程，而不是等到 K 线收线后才反应。

## 五、架构升级方案

### 5.1 新增: QuoteSubscriptionService

```typescript
// api/src/services/quote-subscription.service.ts

class QuoteSubscriptionService {
  private quoteCtx: QuoteContext;
  private subscribedSymbols: Set<string> = new Set();
  private priceCallbacks: Map<string, ((price: PushQuote) => void)[]>;
  private klineCallbacks: Map<string, ((kline: PushCandlestick) => void)[]>;

  // 1. 初始化时设置回调
  async initialize() {
    this.quoteCtx = await getQuoteContext();

    // 报价推送 → 直接喂入 FastMomentumService
    this.quoteCtx.setOnQuote((err, event) => {
      if (err) return;
      const { symbol } = event;
      const quote = event.data;
      // 喂入 FastMo（毫秒级）
      fastMomentumService.feedSingleQuote(symbol, Number(quote.lastDone));
      // 通知其他消费者
      this.emit('quote', symbol, quote);
    });

    // K 线推送 → 喂入 momentum 评分
    this.quoteCtx.setOnCandlestick((err, event) => {
      if (err) return;
      if (event.data.isConfirmed) {
        // 收线 K 线 → 触发 momentum 重算
        this.emit('kline', event.symbol, event.data);
      }
    });
  }

  // 2. 策略启动时订阅标的
  async subscribeSymbols(symbols: string[]) {
    const newSymbols = symbols.filter(s => !this.subscribedSymbols.has(s));
    if (newSymbols.length === 0) return;

    // 订阅实时报价
    await this.quoteCtx.subscribe(newSymbols, [SubType.Quote], true);

    // 订阅 1min + 5min K 线
    for (const symbol of newSymbols) {
      await this.quoteCtx.subscribeCandlesticks(symbol, Period.Min_1);
      await this.quoteCtx.subscribeCandlesticks(symbol, Period.Min_5);
      this.subscribedSymbols.add(symbol);
    }
  }

  // 3. 策略停止时退订
  async unsubscribeSymbols(symbols: string[]) {
    await this.quoteCtx.unsubscribe(symbols, [SubType.Quote]);
    for (const symbol of symbols) {
      await this.quoteCtx.unsubscribeCandlesticks(symbol, Period.Min_1);
      await this.quoteCtx.unsubscribeCandlesticks(symbol, Period.Min_5);
      this.subscribedSymbols.delete(symbol);
    }
  }
}
```

### 5.2 改造: FastMomentumService

新增 `feedSingleQuote()` 方法，支持单条实时报价:

```typescript
// 当前: 批量喂入（每 5 秒一次）
feedQuotes(quotes: Map<string, number>): void

// 新增: 单条实时喂入（每笔成交）
feedSingleQuote(symbol: string, price: number): void {
  const buf = this.buffers.get(symbol);
  if (!buf) {
    const newBuf = new RingBuffer<PriceSample>(BUFFER_CAPACITY);
    this.buffers.set(symbol, newBuf);
  }
  buf.push({ price, timestamp: Date.now() });
  // longBuffer 同理
}
```

**Buffer 容量调整**:
```
当前:  BUFFER_CAPACITY = 12  (12 × 5s = 60s)
升级:  BUFFER_CAPACITY = 120 (120 × ~500ms = 60s)

当前:  LONG_BUFFER_CAPACITY = 60  (60 × 5s = 5min)
升级:  LONG_BUFFER_CAPACITY = 600 (600 × ~500ms = 5min)
```

### 5.3 改造: 调度器轮询降级为 Fallback

```typescript
// 轮询仍然保留，但作为订阅的降级方案:
// 1. 正常情况: 订阅推送驱动一切，轮询间隔放宽到 30 秒
// 2. WebSocket 断线: 自动降级到 5 秒轮询
// 3. 重连成功: 恢复订阅模式
```

### 5.4 改造: K 线数据流

```
当前:
  调度周期 → getIntradayVWAP() → candlesticks() API → 240 根 1min K 线 → 缓存 60 秒

升级:
  subscribeCandlesticks(symbol, Period.Min_1)
  subscribeCandlesticks(symbol, Period.Min_5)
      ↓
  setOnCandlestick → 实时更新内存中的 K 线数组
      ↓
  momentum 评分随时可用最新数据（无缓存延迟）
      ↓
  getIntradayVWAP() 仍保留（初始化 + fallback 用）
```

## 六、订阅限额与注意事项

| 限制 | 值 | 说明 |
|------|-----|------|
| 单次请求最多标的 | 500 | 足够（策略池通常 5-15 个标的） |
| 同时订阅最多标的 | 500 | 足够 |
| HK BMP 不支持推送 | N/A | 我们只做美股，不受影响 |
| 频率限制 | 有 | 订阅/退订操作有频率限制，不是推送本身 |

### 需要确认的点

1. **美股实时行情权限**: 需确认 LongPort 账户是否有美股 Level 1 实时行情权限（非延迟）
2. **期权报价订阅**: 确认期权代码（如 `AAPL230317P160000.US`）是否支持 Quote 订阅
3. **WebSocket 连接稳定性**: 生产环境（NAS Docker）的网络稳定性
4. **推送频率**: 活跃标的（SPY/QQQ/TSLA）的推送频率可能非常高，需要考虑性能

## 七、对退出精度的影响 — 阶梯锁利漏损案例

行情订阅升级不仅影响入场，**退出精度同样被轮询延迟严重削弱**。

### 实际案例: 阶梯锁利底线形同虚设

Strategy 10 当前配置:
```
profitLockSteps: [
  { threshold: 6,  floor: 4 },
  { threshold: 10, floor: 7 },   ← 此案例命中的台阶
  { threshold: 15, floor: 11 },
  { threshold: 20, floor: 16 },
  { threshold: 25, floor: 21 },
  { threshold: 30, floor: 27 },
]
```

**案例时间线**:
```
检查 T+0s:  PnL = 12.8% (峰值) → peakPnLPercent 更新为 12.8%
            踩上 10% 台阶 → 底线锁定 7%
            （中间 0-5 秒: 0DTE 高 gamma 加速下跌，价格穿过 7% 底线）
检查 T+5s:  PnL = 2.2% → 远低于 7% 底线 → TRAILING_STOP 触发
            退出日志: "阶梯锁利：峰值12.8%踩上10%台阶，底线7%，回落至2.2%触发"
```

**损失分解**:
| 阶段 | 值 | 说明 |
|------|-----|------|
| 峰值利润 | 12.8% | 最高盈利 |
| 底线应锁定 | 7.0% | 配置的底线 |
| 实际退出 | 2.2% | 下次检查才发现 |
| **漏损** | **4.8%** | 底线到实际退出的差距 |
| 仓位 $500 时 | **$24** | 每笔交易白白多亏 |

### 问题本质

`option-dynamic-exit.service.ts:490-498` 的阶梯锁利检查:
```typescript
if (pnl.grossPnLPercent <= lockedFloor) {
  return { action: 'TRAILING_STOP', ... };
}
```

逻辑完全正确 — 但它只在调度器周期触发时执行（每 5 秒一次）。在 0DTE 高 gamma 环境下:
- 底层标的移动 $0.50 → 期权价格可波动 10%+
- 5 秒内发生的跌穿完全不可见
- 底线只是"下次检查才生效的声明"，不是实时保护

### 升级后的改善

| 架构 | 检测到破底线 | 退出 PnL | 漏损 |
|------|-------------|---------|------|
| 当前(5s轮询) | 下一个 5s 周期 | ~2.2% | ~4.8% |
| WebSocket订阅 | ~50-200ms 内 | ~6.5-7% | ~0-0.5% |

WebSocket 订阅后，`setOnQuote()` 每笔成交都推送:
1. 推送回调中实时计算 PnL
2. PnL 跌破底线的瞬间触发退出
3. 阶梯锁利的底线才真正变成"实时保护"

> **结论**: 行情订阅不仅是入场问题的解药（更早检测 momentum），同时也是退出问题的解药（底线保护即时生效）。这进一步提升了 Phase 2 的优先级。

## 八、对整体方案的影响

订阅架构升级同时解决**入场**和**退出**两个维度的问题，这个发现**改变了入场时机问题的解决思路**:

| 原方案 | 新思路 |
|--------|--------|
| 降低入场阈值 | 不需要降阈值 — 实时数据让 momentum 更早达到阈值 |
| FastMo 从 gate 升级为信号源 | 仍然有价值 — 但 FastMo 的数据精度大幅提升 |
| momentum 加速度检测 | 更可行 — 高频数据让加速度计算有统计意义 |
| 拉 NAS 数据验证 | 仍然需要 — 但验证目标从"提前入场"变为"实时数据效果" |

### 修订后的 Phase 顺序

```
Phase 0: NAS 数据分析脚本（不变）
Phase 1: PnL 修复（不变）
Phase 2: 行情订阅架构升级（新增，优先级提升）
  → QuoteSubscriptionService
  → FastMo 实时喂入
  → K 线推送替代轮询
Phase 3: K 线增强 + 权重恢复（基于订阅数据）
  → volume-weighted momentum（用推送数据）
  → 1min + 5min 双框架（用 K 线推送）
  → 权重恢复 20/60/20
Phase 4: 入场时机微调（如果 Phase 2+3 后仍需要）
```

**核心变化**: 行情订阅升级应该排在 K 线增强和权重恢复**之前**，因为它改变了数据基础设施。先把数据管道从 5 秒轮询升级到毫秒推送，再在新的数据基础上做评分优化。
